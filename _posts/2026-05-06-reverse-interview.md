**Agent Problems**

我了解到公司在积极布局 Agent。在目前的规划中，咱们的 Agent 应用是更偏向于对内赋能（比如研发提效、内部知识库问答、运营自动化），还是已经有明确的对外商业化产品落地场景？ 或者说，大家主要在做一些什么，尝试解决什么问题？我进去以后的主要工作会是什么。

p.s. 对内的容错相对更高且更加容易推进，方便对新技术进行测试和迭代，对外则更加需要关注一些幻觉，稳定性以及用户体验。这个问题主要希望获取核心工作方向和内容。

目前行业里 Agent 的应用百花齐放。咱们团队聚焦的场景，是更偏向于处理明确的结构化任务（比如通过 API 调用执行自动化流程），还是更侧重于对非结构化知识的深度理解与推理（比如基于复杂长文档的结构感知与 Agentic RAG）？

p.s. 前者更加侧重于一些自动化 workflow，需要对业务本身深入了解来实现；后者则侧重于 Agent Dev 的相关技术，更高的自主权将更大程度依赖模型的能力以及开发者对安全边界的认知。

关于咱们关注的 Agent 应用场景，目前的系统设计是更多定位在 Copilot 模式（即需要频繁的人类反馈和确认），还是致力于在特定垂直业务流中实现高自动化率（Autonomous）的闭环工作流？

p.s. human in the loop 目前来看依旧是无法被替代的，如果希望打造高自动化率的系统，那么自动化的高质量校验将是我们关注的核心问题，而如果更多的允许人工介入，那么扩展系统的功能就比打磨细节更值得关注。

考虑到复杂业务场景的需求，咱们目前的重心是在打磨单智能体（Single Agent）的复杂任务拆解能力，还是已经开始探索多智能体协作（Multi-Agent Collaboration）的业务落地？

p.s. 从某种意义上将，CCC 这类项目已经宣告了多智能体协作已经从技术报告变成了 Demo，虽然稳定性堪忧，但是协作依旧是我们值得关注的前沿技术方向，当然它更加侧重于技术研讨而非成熟技术，高自主性的 MAS 很可能会带来和人类协作一样的问题。

团队目前是如何构建 Agent 评估指标的？除了传统的 LLM Benchmark，有没有针对咱们业务建立特定的工具调用准确率，或者执行轨迹（Trajectory）的自动化评测框架？

p.s. Evaluation 永远是 Agent 绕不开的问题

在成本和延迟上的权衡策略是什么？更多的依赖成熟的闭源大模型，还是考虑已经人工/或自动的将不同的请求路由给不同层级的模型，是否考虑了成本/延迟与性能的 trade-off。

p.s. 这是工程化绕不开的一个问题，从技术Demo走向生产需要考虑的关键 trade-off 之一，因为生产环境的安全边界不容trade-off，但慢一点贵一点，好像无所谓。

**Other Problems**

每个开发者有多大的自由来做出决定？团队内部的结构，协作以及日常的工作划分是什么样的？如果遇到不同的意见怎样处理？

AI Coding 已经介入了绝大多数程序员的工作流，尤其是 Agent Dev 领域。公司如何看待 AI Coding，是否允许员工在项目中较多的使用 AI Coding 来加速迭代，是否直接提供 AI。

我可以为开源项目做贡献吗？是否需要审批？
