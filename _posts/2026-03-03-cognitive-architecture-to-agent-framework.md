# 从智能体的认知结构到智能体框架

## 引言：Building A LLM Agent

从2025年开始，语言模型智能体逐渐成为了比基础模型更加火热的方向，无论是学术界和工业界都对此展开了全面的研究与讨论，百家争鸣的局面下，各式各样的LLM Agent被开发，去解决曾经无法解决的问题。

对于为什么需要构建一个语言模型智能体，这不是本文希望详细讨论的问题，其必要性在此以一句简单的总结带过：**LLM Agent的LLM为代理补充了整个世界的Prior**，使其可以解决各种的真实世界中的问题，而不局限了在极度受限的环境中。

LLM Agent 本身是非常简单的东西，**From Next Token Prediction to LLM Agent**，其核心逻辑只是让模型从单纯的对话预测变成一个可以通过对话与工具交互的代理。任何人只要了解调用语言模型的SDK，懂得让模型输出JSON并解析（可解析格式都可以，JSON的Token Efficiency并不高），都是在开发LLM Agent。

但是，构建一个Agent除了对业务流程本身的理解以外，还有非常非常多的问题需要考虑，仅仅出于个人直觉的Agent是一个玩具而不是工具。了解智能体的认知结构，认识框架应该如何被设计，不是为了去使用框架本身，而是去了解如何去构造一个更好的，可以交付于生产环境的Agent。系统化设计,系统化理论,系统化测试,从而实现智能体的robustness。

本文是智能体的基础设施的另一篇文章，希望去讨论智能体的认知结构，一些设计上的实践，以及对Agent框架的讨论。作为Agent的另一个关键组件Memory将会被单独讨论，而不是在此详细介绍其实现。

## CoALA ：智能体的认知结构

CoALA （Cognitive Architectures for Language Agents） 将语言模型本身视为认知架构的核心组成部分，使用附加模块来弥补语言模型本身的无记忆性和无法与外部交换信息的问题。CoALA 是一个很好的，帮助我们理解智能的来源的框架结构。理解 Single Agent，我们才能去理解多智能体之间的协作。

CoALA认为一个智能体的认知结构包含下面的内容

1. 和LLM交互：包括提示与输出的解析，LLM是智能的来源，而这是智能与附加组件之间的桥梁。
2. 内部记忆（检索和学习）：内部记忆可以继续细分为，检索（从长期记忆读取）；推理（更新短期工作记忆）；学习（更新长期记忆）。
3. 外部环境（和外部交互）：包括物理环境，数字环境，与人类的交互，与其他智能体的交互。

从时间顺序上看，智能体检索（内部记忆），推理（内部记忆），评估后续的行动（LLM），在行动后观测（外部环境），学习（内部记忆）然后重新循环这个流程。人类使用类似的框架就可以处理几乎真实世界全部的问题，因此 LLM Agent 用这样的框架来处理信息。


工作记忆（Working memory）： 工作记忆将活跃的信息保留在目前的决策周期里，包括感知到的输入，推理生成的知识，从长期记忆检索的知识以及从之前的决策周期里留下来的核心知识（我们希望进入一轮新的决策周期就进行一次记忆清理，记忆压缩就是针对工作记忆进行的） **工作记忆可以被理解为我们的核心上下文窗口**。工作记忆一边和llm交互，一边和Long memory与Grounding交互。

情景记忆（Episodic Memory）：存储决策的经验，当他进入工作记忆的时候，将会被用于in context learning ；一些决策经验也可以被写入Episodic Memory  留给后续使用。（这是一种长期记忆，当检索认为它有用的时候就会被放入长期记忆）

语义记忆（Semantic Memory）：情景记忆存储经验，语义记忆存储知识，情景记忆和工作记忆都是为了去加入Context帮助模型理解真实问题，但语义记忆更强调知识库本身，LLM积累Episodic Memory，但是一般不积累Semantic Memory。

程序记忆（Procedural memory）：语言模型内建的知识以及在智能体代码中直接编码的知识（比如图流转结构，外接的各种action的信息，甚至是封装的提示词）。程序记忆是需要进行精妙的初始化才能启动智能体的，写入程序记忆来增加新的action是可行的，但也是危险的（我们在这里认为自动化的Workflow 生成是对程序记忆的修改，由于缺少专业知识的介入与核查，生成的Workflow可能不满足我们的需求，但是我们不知道且无法继续纠错）。

三种记忆的读写就是内部动作空间，与之对应的外部交互则是外部动作空间。而外部空间的交互 Action 包括多种形式，物理环境，数字环境，与人类的交互，或者与其他智能体的交互。

Retrival 从长期记忆检索内容到工作记忆里，具体如何理解检索与记忆，如何在工程上实现它值得思考，在记忆增强智能体中，长期记忆和检索是一个问题的两个方面，是终身学习的核心。（在CoALA中，我们没有强调遗忘，因为其工记忆不是累积的结构，但遗忘实际上是非常重要的问题，在Memory管理里我们会很好的讨论他）。

Reasoning 推理读取工作记忆，写入工作记忆，支持学习（写入长期记忆）以及决策（执行action）Learning 用经验更新情景记忆 用知识更新语义记忆 用微调和智能体结构调整更新程序记忆。

**Context将会是Agent Engineer的核心，在Linear Attention实现之前，注意力机制带来的底层约束就是注意力不足和有限上下文，因此我们会单独去讨论Context**

基础模型的进步会改变智能体的设计，如果有效上下文很长，那么长期记忆可能就不那么重要。如果内部推理能力更强，就无需手动的分步规划。基础模型的涌现无法预测，从现在来看CoALA仍旧有用且会在未来的一段时间有用。当基础模型的能力无限强，我们也可以用这样的架构思路去研究模型内部的隐藏回路，记忆，行动，决策，推理，等等等等。事实上，**Agent 的工程发展史，本质上就是不断用外部的系统设计（框架）去弥补当前 LLM 在记忆和推理（认知架构）上的短板的过程。**当短板不再，那Agent工程自然也将消失。

LLM推理规划能力现在可以将模糊的人类需求整理为可执行任务。但目前的LLM受限于基础能力问题，缺少一些必要的知识和推理决策能力。Agent的发展在弥补他的不足，无论是CoALA引入长期记忆和学习机制，引入推理，行动，决策循环，还是工程实践上引入MCP,Skills，Workflow。都是在处理这个问题，去利用强大但不够强大的处理不确定性的能力。

## 智能体工程设计的本质  

从无形之脑到可控之躯，智能体工程设计的本质，在于解决一个核心矛盾：**智能本身的涌现（Emergence）与业务鲁棒性（Robustness）的博弈**。

一个纯粹由 Prompt 驱动的、完全自主的 LLM Agent 充满想象力，但也极易在复杂的生产环境中产生幻觉、陷入死循环或导致流程崩溃。相反，纯粹基于规则的代码虽然极其稳定，却毫无“智能”可言。智能体工程设计的终极目标，就是设计并固化一些决策流程，在性能（Performance）和泛化能力（Generalization）之间取得平衡。复杂的专用流程（Workflow）可以精准解决特定领域的问题，而简单的流转逻辑则能提供更广泛的适应性。简而言之，就是为语言模型这个由于缺乏物理实体而“四处游荡的灵魂”，打造一具有物理规律约束的“肉体代码”。

![Agent Design](https://github.com/The-Pocket/.github/raw/main/assets/design.png)

这张控制流与架构图非常精准地概括了目前工业界构建 Agent 的核心手段（本质上是对状态流转和记忆读写的抽象）。它确实覆盖了绝大多数基础工程模式，但在当前前沿的实践中，我们需要认知神经科学的视角重新理解他们。**这些看似互不相干的工程架构，本质上都是为了弥补基础语言模型在特定认知模块上的短板，而进行的外部系统强化或妥协。**

### 认知落地：工程设计模式如何映射认知架构？

我们可以将图谱中的各种控制流，与 CoALA 框架的认知模块进行一一映射：

#### Workflow (流水线)：程序记忆

图中纯粹的 **Workflow**（如 Summarize Email -> Draft Reply）是单向的流水线。

- **认知映射**：它是对 `Procedural Memory (程序记忆)` 的外部硬编码。
- **工程本质**：这是当下业务落地最稳妥的方案。理论上，完美的 Agent 应该自主规划（Plan）每一步。但在生产环境中，LLM 不稳定的规划能力不可接受。Workflow 通过人类编写的 SOP（标准作业程序），直接剥夺了模型在这部分的自由行动权限，框架的图结构直接且强硬地规定了它的认知路径。

#### CoT 与 Map-Reduce：工作记忆

图中的 **CoT (Chain-of-Thought)** 引入了 `think history store`，而 **Map-Reduce** 引入了 `Batch + Merge` 分治。

- **认知映射**：它们都是针对 `Working Memory (工作记忆)` 容量和处理能力的工程 Hack。
- **工程本质**：工作记忆对应模型的 Context Window。尽管现代模型的窗口越来越长，但注意力衰退（Lost in the middle）仍是痛点。面对复杂逻辑时，CoT 通过在工作记忆中循环写入中间推导步骤，用推理获得更深的智能；而面对超大规模信息，Map-Reduce 将信息拆分（Map）给多个并行的、短生命周期的工作记忆，最后提炼核心知识聚合（Reduce）回主工作记忆。

#### RAG 与 Chat Memory：外挂的长期记忆

图中的 **RAG (Vector DB store)** 和 **Chat Memory** 模式展示了数据的双向读写。

- **认知映射**：它们对应 CoALA 中的 `Retrieval (检索)` 和 `Learning (学习/写入)` 模块，负责打通工作记忆与长期记忆。
- **工程本质**：向量数据库往往充当 Agent 的 `Semantic Memory (语义记忆)`，提供世界观和客观知识；而 Chat History 则构成了 `Episodic Memory (情景记忆)`，维持了 Agent 的**人格连续性**。工程上的记忆，本质上就是智能体在每次决策前，通过 Retrieval 动作将这两种长期记忆唤醒，有选择地加载到当前的 Working Memory 中，赋予其世界知识的广度与时间连续性。

#### Agent (循环与分支)：完整的认知闭环

图中的 **Agent** 模式（Summarize -> Draft -> Review -> Approve/Need review）是一个带有条件分支的自循环。

- **认知映射**：这是一个包含 `External Environment (外部环境交互)` 和内部 `Reasoning (推理/反思)` 的完整认知闭环。
- **工程本质**：这对应了经典的 ReAct 与 Reflexion 机制。执行动作并获取外部反馈后，Agent 不急于进行下一步，而是进入内部 Critique 阶段：读取工作记忆中的目标与当前反馈进行比对，若发现偏差，则更新工作记忆并重新流转。这是模型第一次从开环指令机器走向具有自我纠错能力的闭环控制系统。ReAct 与 Reflexion 也是在所有的Agent组成模式里我们认为最重要的一个，也是目前主流Agent产品的核心架构（如OpenClaw和Claude Code）

#### Agent Team (Handoff 转移)：动态认知路由

当前工业界正在流行 **Agent Team/Swarm (Handoff 模式)**，几个高度专精的 Agent 像接力赛一样工作。

- **认知映射**：当一个单体大脑无法处理庞杂任务时，转移 Context（工作记忆）是一种极度贴近人类企业层级运转的高级组织形态，可视为动态的 `程序记忆` 路由，这个路由完全的交给了预先提升的Agent实现，而不是固化的图流转。
- **工程本质**：相比于静态 Workflow，Agent 遇到无法解决的问题时，会主动将控制权和 Context 移交（Handoff）给更专业的节点。高度的自由度意味着模型能够处理更加宽泛的问题，而不是一个偏向专用的智能体。
- **特别的**：正如本节的开头，这种模式是**智能本身的涌现（Emergence）与业务鲁棒性（Robustness）的博弈**，智能在此略胜一筹。

### Single Agent 与 MAS

当工程图谱延伸到 **Multi-Agent (pub/sub)** 和 **Supervisor (Nesting)** 时，一个极具争议的问题出现了：究竟什么是单体智能体（Single Agent），什么又是多智能体系统（MAS）？其实，**角色（与结点）的数量从未定义边界，真正的界线在于工作记忆的归属权与控制流的自主性。**

**单一智能体（Single Agent）：共享全局状态的多角色**

无论是复杂的 Workflow、包含 Review 循环的 Agent，还是带有嵌套结构的 Supervisor，只要它们在工程实现上**共享同一个全局的 State（状态字典或上下文对象）**，并且由一套中心化的代码逻辑（如 LangGraph 的 StateGraph）来决定节点流转，它本质上就是一个 Single Agent。

即使在 Supervisor 模式下，存在多个不同的 Persona（系统提示词），它们也更像是一个大脑在不同阶段戴上了不同的面具。它的 `工作记忆` 是全局透明（有时是多Agent之间隔离的）的，其 `程序记忆` 被外部图结构锁死，内部节点无法拒绝被图谱调用。

**多智能体系统（MAS）：基于 Actor 模型的消息社会**

图中真正的 Multi-Agent 模式（带有 pub/sub 机制）揭示了 MAS 的本质——它是一个**分布式去中心化架构**。

- **独立的认知黑盒**：每个 Agent 拥有绝对私有的 `Working Memory` 和长期记忆。Agent A 看不到 Agent B 的内部状态，除非 B 主动共享。
- **交互环境化**：在 MAS 中，其他 Agent 属于外部环境。它们之间的交互不是全局状态变量，而是消息传递（Message Passing），完美契合计算机科学中的 Actor Model。
- **高度自主性**：接收到消息的 Agent，会经历完整的内部认知循环（感知 -> 检索 -> 推理），然后**自主决定**是回复、执行操作、转发，还是直接忽略。

**现实中工程上的悖论：**

现实是，真正的 MAS（如多个不受控的思想家开会）效率极低、极易发散。为了保证生产的成功率，开发者常常在 MAS 的外层套上强约束的 Workflow。这就形成了一个尴尬的悖论：**我们用多智能体的架构（独立记忆与通信），跑着单智能体的工作流（剥夺自主规划）。** 

这也是为什么如今的 Agent 框架让人眼花缭乱的阵痛期现象。用户缺乏对Agent本身的理解，却被各种不同的框架的抽象范式干扰，导致无法形成对Agent设计核心理解。

### 自主执行还是专家决策：Agent or Workflow？

纵观现有的 Agent 工程设计，其发展脉络始终在尝试平衡**专家经验（人类的固定规则）**与 **Agent 的自主性（模型的涌现智能）**。

将自主性推向极端，我们得到的是一个极致通用的 Agent——它通常表现为一个原生支持内部推理循环的 Chatbot，通过 MCP (Model Context Protocol) 等标准协议接口动态发现并调用一切可用工具。这样的单一 Agent 理论上能够自主分析并处理几乎所有领域的问题。

相反，将专家经验固化到极致，我们得到的则是一个严格的自动化执行工具。内部完全通过人工编写的代码判断与分支来决定走向，一旦规则被彻底锁死，它就退化成了一个传统的脚本，我们唯一能做的就是触发它并等待必然的结果。

在当下的语言模型开发语境中，这两种路线被直观地收束为两大阵营：**Agent** 与 **Workflow**。
- **Agent**：赋予大模型极高的自由规划权（自主性），容忍其在发散试错中寻找答案。
- **Workflow**：通过强制模型输出特定的标志词或受限语法（如结构化 JSON 输出），将不可控的智能强行约束在一个预设的流转图（通常是严格的 DAG）上，以确保业务的底线。

无论向哪一方走极端，都难以完美地替代人类复杂的脑力劳动。原因在于：多轮开放式的 Agent 调用**不可避免地会遭遇上下文膨胀（Context Bloat）从而致使推理失控**；而缺乏分支探索空间的 Workflow 又**无法处理动态世界中未定义的异常变动**。因此，面向真正复杂的生产级问题，下面是两种相对可行的工程解法：

1. **Subagent（子智能体）：阻断 Context 污染**
   * 这是一个非常契合直觉且有效的设计。当一个大型的通用主 Agent 需要处理极大颗粒度或轮次跨度极长的任务时，它不是简单地调用一个单点工具，而是将这个独立的复杂子任务**直接抛给另一个具备完整自主规划与工具决策能力的 Subagent**。
   * 在这个局部沙盒里，Subagent 就是一个绝对完整的大脑，它运行着独立的内部循环（感知->检索->推理）。最核心的在于：**当它完成探索后，只会将极其精简的关键结论与必要上下文交还给主 Agent。** 这完美模拟了人类的神经系统。

2. **Workflow + Agent 混合嵌套架构：图与节点的融合**
   为了拆解极其庞大且必须要求稳定性的工作流，框架最终必然走向不同决策形态的嵌套与连接。在这样一个可能带有环图的混合系统中，我们会根据不同业务环节的容错率，编排三种完全不同自由度的节点：
   - **纯粹规则化的 MCP Tool**：被确定性调用，快速给出无推理结果。
   - **由 LLM 决策的短程 Workflow**：任务在多个预设好的约束节点中流转，这些 Node 虽然是流程图上的一环，但其内部可能封装了一个独立运作的 Agent。
   - **完全自主决策的 Agent**：仅接收高层抛来的抽象任务指令与 Context，拥有极高的试错和工具调配权限。

在这种混合架构中，虽然 Node（节点）仍是最小的流转单元，但它们往往依托一个系统级的**全局共享记忆池**。我们通过外围 Workflow 编排的严密规则（专家经验的沉淀），精准控制着底层 Node 产生的大量探索历史中，什么内容能被沉淀到上层的状态中，什么过程噪音必须被及时丢弃。这是目前构建一个复杂的Agent的必须手段。

### 框架的意义

从认知结构到工程模式，跨越了 Agent 与 Workflow 的两难之争，我们终于可以把“框架”这个流行词重新说清楚：**它们本身并不提供智能，而是在冻结一套管理 LLM 不确定性的工程约定。**

框架一边告诉开发者，一个 Agent 应该如何组织状态、控制流、工具和记忆；一边把这些重复劳动封装成可复用的基础设施。也正因如此，它们既是认知脚手架，也是工程脚手架。但脚手架不是建筑本身，任何一种抽象都不该被误认成 Agent 的本体。

## 智能体框架——从认知脚手架到下一代基础设施

### 常用智能体框架与它们沉淀出的开发范式

如果用一句话定调：框架不是在提供智能，而是在冻结如何管理 LLM 不确定性的工程约定。不同框架的真正区别，不是名字或界面，而是它们选择把什么当成核心抽象单元，把什么交给模型自由发挥，又把什么提前锁死在人类写好的控制流里。

#### LangChain / LangGraph：从 Chain 到 StateGraph

LangChain 代表了 Agent 工程最早一波非常典型的思路：先把 Prompt 调用、工具调用、RAG、记忆这些常见动作封装成可拼装组件，再通过 Chain 把它们串起来。它提供的不是某一个固定智能体，而是一套“把LLM 当成应用组件来组合的接口。对很多开发者来说，LangChain 最重要的历史意义，是它第一次把写 Prompt推进到组织一段 LLM 驱动的软件流程。

但当 Agent 开始从一次性调用走向长任务与循环闭环时，单纯的 Chain 很快不够用了，LangGraph 于是把核心抽象升级为显式的 `StateGraph`：共享状态、节点、边、条件路由、循环、检查点恢复。这种范式极其适合需要多轮推理、工具执行、错误重试和中途恢复的场景，尤其像 coding agent 这类天然带有“读代码 -> 改代码 -> 跑测试 -> 继续修复”的长回路任务。它的强项在于状态机思维：你知道当前系统处在哪个节点，为什么跳到下一个节点，以及哪里可以被持久化和回放。

代价也同样明显。它的抽象层很厚，开发者往往需要先把业务翻译成图、节点和状态更新规则，才能开始写真正的业务逻辑。对已经理解 Agent 本体的人来说，这是一种精准控制；对尚未形成心智模型的人来说，这也可能变成“先学会框架语法，后理解问题本身”的倒置过程。

#### Dify：面向应用交付的 Agent 平台

如果说 LangGraph 更像是给工程师的代码级状态机，那么 Dify 代表的是另一种更偏应用交付的范式：把知识库、工具、工作流、观测、部署和运营接口尽可能放进同一个平台里。它的核心吸引力不在于某一项抽象特别前沿，而在于把构建一个业务 Agent 所需的大量琐碎配套统一了起来，让团队可以更快把一个能用的应用推到线上。

这种一体化平台的优势非常直接。对问答、助手、检索增强、表单处理、内容生产这类业务型 Agent，开发团队往往并不想自己从零搭建记忆层、日志面板、发布链路和后台配置系统。Dify 通过更低门槛的编排方式，把“先跑起来”这件事做得非常顺手。它更像一套 Agent 应用工厂，而不是一组孤立的库。

但它的边界也正来自这种产品化封装。越是复杂的控制流、越是细粒度的状态设计、越是需要底层透明调试的场景，平台式抽象就越容易让人感到“不够伸手进去”。当任务开始需要自定义恢复逻辑、精细化上下文裁剪或非标准工具路由时，平台的便利性也会变成工程自由度的上限。

#### n8n：自动化工作流吸收 AI 节点

n8n 的出发点并不是 Agent，而是自动化。它原本解决的是“如何把不同系统和 SaaS 服务可靠连起来”的问题，因此它的真正强项是连接器生态、触发器机制和可视化流程编排。当 AI 节点被接进来之后，n8n 形成的是一种非常有代表性的开发范式：以确定性流程为主体，在某些节点上局部引入模型的模糊判断能力。

这类范式特别适合“流程自动化 + 一点智能”的任务。例如表单收集、消息路由、摘要生成、规则触发后的检索补充、跨系统同步时的语义分类。对很多企业场景来说，这其实已经足够有价值，因为大部分问题本来就不需要一个高度自主的 Agent，而只需要在已有自动化链路上加入几个智能节点。

它的限制也因此很清楚：n8n 擅长的是把 AI 当成工作流节点，而不是把工作流本身升级为完整认知闭环。它可以承载简单循环，却不天然擅长表达复杂记忆管理、反思、长期状态恢复和多 Agent 协商。换句话说，它非常适合把智能嵌进流程，但不天然适合把流程演化成真正的 Agent runtime。

#### Coze：面向产品化入口的 Agent 工厂

Coze 代表的不是“怎样精确编码一个 Agent”，而是“怎样把 Agent 变成一个可交付、可运营、可分发的产品入口”。它把 Persona、知识、工具、插件、工作流、聊天界面和发布渠道放进一个更偏产品层的整体体验里。对很多并不以底层工程控制为第一诉求的团队来说，这种抽象非常有效，因为他们真正想做的不是一个后端框架，而是一个能上线、能被用户直接接触的 Agent 产品。

因此 Coze 的特色不在于它把单一技术点做得最深，而在于它更早把“对话入口、能力拼装、应用分发”视作同一个问题。它更像是 Agent 时代的应用工厂：开发者在里面定义角色、知识和流程，用户在外面接触的是一个已经被包装好的数字产品，而不只是某段后端编排逻辑。

它的代价也很典型。因为它的抽象重心偏产品层，很多工程细节被平台替你做了选择。对于想快速做出面向终端用户的产品，这是好事；但对于想深入控制状态结构、协议层、日志系统或跨系统运行时的人来说，平台提供的便利也意味着更强的约束。

#### AutoGen：多智能体对话式编排

AutoGen 代表的是另一条路线：不先把世界画成图，而是先把世界拆成角色，再让这些角色通过消息传递和对话来协作。它冻结的是“多智能体消息社会”这一开发范式：不同 Agent 拥有不同职责、不同提示与不同工具，然后通过会话、Handoff 和任务分解来完成整体目标。

这种范式之所以有吸引力，是因为它天然贴近人类对团队协作的直觉。你可以很快搭出一个 planner、一个 coder、一个 reviewer、一个 tool agent，让它们像开会一样交换信息。这对展示多 Agent 的组织形态、探索开放式任务分工，以及研究自主协作的上限都很有价值。

但也正因为它把更多控制权交给了 Agent 间对话，AutoGen 式系统会更早撞上收敛性、可观测性和生产稳定性的问题。对话为什么发散？任务为什么在不同角色间反复横跳？哪个 Agent 真正拥有状态主权？这些问题一旦进入真实业务，调试成本会迅速上升。因此它非常适合帮助我们理解多 Agent 的潜力，却不天然等同于最容易落地的生产方案。

#### 小结：这些框架到底冻结了哪些开发约定？

| 框架 | 抽象单元 | 控制流 | 状态模型 | 自主性 | 可观测性 | 适用场景 | 主要代价 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| LangChain / LangGraph | Chain / Node / StateGraph | 显式图路由与循环 | 共享 State + 检查点 | 中等，可被图精确约束 | 较强，但依赖开发者设计 | coding agent、多步闭环、长任务 | 抽象，学习成本高 |
| Dify | 应用、知识库、工具、工作流 | 平台式编排 | 平台托管状态 | 中低，以平台能力为主 | 中等，够用但不完全透明 | 问答、业务助手、快速交付 | 灵活度有限，底层细节难伸入 |
| n8n | 自动化节点 | 确定性流程 + AI 节点 | 流程上下文 | 低到中，AI 只在局部决策 | 强于系统集成，弱于认知追踪 | 流程自动化、系统串联 | 不是原生 Agent runtime |
| Coze | Persona、插件、工作流、产品入口 | 产品化工作流与对话入口 | 平台内聚状态 | 中等，围绕产品体验组织 | 中等，工程透明度受平台影响 | 面向终端的 Agent 产品 | 平台约束强，工程自由度有限 |
| AutoGen | Agent / Message | 对话驱动协作与 Handoff | 分散状态或隐式会话状态 | 高 | 较弱，收敛路径难追踪 | 多 Agent 实验、开放任务协作 | 易发散，难稳定落地 |

收束来看，这些框架不是在回答谁更先进，而是在分别回答三个更具体的问题：如何约束模型，如何组织状态，以及如何处理不确定性。所谓框架差异，归根结底就是这三道题的不同解法。最后解法影响到的，就是你给用户最后暴露了一些什么。

### 当框架先于规范：智能体框架为何迅速变成技术债务

问题从来不是框架存在，而是智能体设计规范尚未稳定时，框架过早把一套暂时性的做法固化成高抽象。于是很多框架并不是在沉淀已经被证明的工程规律，而是在把某个阶段“看起来够好用”的经验提前神圣化。

#### 规范未定，抽象先行

今天我们仍然没有真正稳定下来的 Agent 设计规范。Context 应该如何分层？Memory 应该暴露为数据库、缓存还是显式写回操作？Tool Use 的权限边界如何定义？失败恢复是按节点、按步骤还是按任务？评测到底评过程、评结果还是评轨迹？这些核心原语都还在快速变化。于是很多框架实际上只是在把阶段性工程经验包装成通用真理，而不是在表达已经成熟的学科边界。

#### 过早抽象抬高了使用者心智负担

更糟的是，过早抽象会直接改变学习顺序。很多开发者在尚未理解 Agent 本体之前，就先被迫学习 `StateGraph`、`Router`、`Node`、`Agent Team`、`Memory Store` 这些框架语汇。结果不是先理解 Context、工具和状态，再选择合适抽象；而是先学习框架如何命名问题，再倒推 Agent 到底是什么。认知顺序一旦颠倒，框架就不再是脚手架，而开始变成新的障碍。

#### 黑盒封装削弱了可观测性

框架的另一个代价是把关键工程细节藏起来。Prompt 怎么拼出来的，状态在什么时候 merge，记忆何时被读取或写回，失败后到底是谁决定重试，很多时候都被封进框架的默认逻辑里。系统顺畅时，这些封装很省心；系统失控时，它们就变成最难拆开的黑盒。对于 Agent 这种本就充满不确定性的系统来说，可观测性不是锦上添花，而是生产可用的前提。

#### 框架很容易跟着模型一起过时

而模型能力的演进速度，又会进一步放大这种问题。原生 Tool Calling、更长的 Context、并行推理、供应商 SDK 自带的 structured output 与 tracing，都会迅速改变哪些抽象是必须的。很多曾经很聪明的封装，几个月后就可能变成多余的一层胶水。于是开发者开始把大量时间花在“怎么绕开框架限制”上，而不是花在真正的任务建模和业务逻辑上。

LangGraph 有一个经典笑话，”当你在阅读这个Blog的时候，LangGraph的某个接口又经历了几次修改，并且文档丝毫不懂“ 这实际上对于开发者是巨量的负担，他们不仅需要理解Agent应该怎么设计，还要熟悉框架本身是什么样的，这种接口悄无声息的变动对于人类或者是Agent来说都是巨大的心负担。

收束来说，框架依然有用，但它更像脚手架，而不该被误当成智能体设计本身。它应该帮助我们暴露问题，而不是替我们隐藏问题；应该降低认知负担，而不是制造新的认知债务。

### Claude Code 与 OpenClaw：为什么前沿系统回到原生开发

如果我们把视角从框架名称转回任务本身，就会发现一个更有意思的现象：很多真正有效的前沿系统，并不是先选一个通用框架，再把业务塞进去；相反，它们往往是先围绕具体环境搭出原生 runtime，再决定哪些部分值得抽象。

#### Claude Code：围绕 coding workflow 的原生闭环

Claude Code 的关键意义，不在于它证明了某个通用 Agent Framework 足够强，而在于它把 coding 任务空间组织成了一个高反馈密度的原生闭环。代码库是可读的，终端是可操作的，测试和构建能提供快速反馈，Git diff 与 code review 构成了天然的人类审查位点。换句话说，它真正依赖的是代码、命令、验证、回滚、审查这整套已经高度数字化的工程系统。

因此 Claude Code 的成功更像是任务结构先跑通了，而不是框架抽象先统一了。它的核心不是把一切包进某个通用框架，而是让模型直接工作在真实开发环境里：读代码、改代码、运行命令、观察结果、再继续迭代。这是一种非常原生的 runtime 思路，框架在这里充其量只是辅助手段，而不是中心。而现在随着他自身的成熟，Claude Code SDK现在变成了Agent开发工具的一部分，我们可以把这个成熟的工具封装进去，从而避免重复去开发大量的接口与稳定性工作。

#### OpenClaw：围绕个人环境的原生系统拼装

OpenClaw 则把另一类问题暴露得更彻底。它面向的不是高度标准化的开发流程，而是聊天、语音、设备、自动化、个人知识、长期记忆、权限控制这些异构而敏感的现实表面。在这种任务空间里，真正困难的部分不是先声明几个 Agent 角色，而是怎样把碎片化环境组织成一个可被感知、可被操作、又不至于失控的系统。

这意味着 OpenClaw 更需要的是原生系统整合：设备接入怎么做，权限怎么隔离，自动化如何触发，长期记忆怎样管理，技能系统怎样渐进暴露，用户如何随时接管。这里的挑战首先是 runtime 和系统工程挑战，其次才是框架接口挑战。如果过早套入一个通用 Agent 抽象，反而可能掩盖真正关键的问题。

#### 轻封装能力单元：Skills 的位置

这也是为什么像 Skills 这样的轻量封装会在这一阶段显得有吸引力。Skills 并不试图把能力压缩成一套厚重框架，而是把说明书、脚本、模板和参考资料打成一个渐进披露的能力目录。它适合承载规则、流程和组织经验，却不强行假装自己已经是一个完整 runtime。关于 Skills 更详细的讨论，可以参考我的另一篇博客[《从 MCP 到 Agent Skills》](/blog/2026/from-mcp-to-agent-skills/)。

但 Skills 也再次提醒我们：轻封装并不等于没有代价。模型面对的是半结构化文档与脚本，它仍然需要自主判断何时启用、读取什么、是否执行。这说明真正重要的并不是有没有框架，而是系统是否把不确定性放在了正确的位置：该交给模型的交给模型，该用工程约束锁死的部分则必须明确锁死。

#### 这两者共同说明了什么

把 Claude Code 和 OpenClaw 放在一起看，更重要的不是谁更先进，而是它们共同说明了一个先后顺序：**先定义环境、反馈、权限、恢复与任务结构，再决定是否需要框架。** 它们不是以通用 Agent Framework 为中心构建，而是以原生 runtime / 原生工程系统为中心构建。框架不是起点，任务结构才是起点。**在Agent开发中，关注业务比关注技术重要的多得多**

### 我理解中的智能体框架：它真正应该提供什么

如果承认重框架很容易制造心智负担，那么下一步就不是简单地拒绝框架，而是重新收窄框架的职责边界。一个好的框架，不该替你定义智能体，只该提供最小但关键的基础设施。

**一个好的框架，不该替你定义智能体，只该提供最小但关键的基础设施**

首先，它应该提供**透明的状态管理**。上下文里有哪些内容，状态如何更新，检查点在哪里，任务如何恢复，都应该是显式可见、可被审计、可被人为理解的。状态不透明，Agent 就只能停留在看起来会动的阶段。

其次，它应该提供**原生工具与协议接入**。MCP、函数调用、外部 API、文件系统、消息系统、乃至像 Skills 这样的轻封装能力单元，都应该能被直接接入，而不是被迫再套一层框架自己的专有协议。好框架应该减少适配成本，而不是制造新的兼容层。

第三，它必须有**强可观测性**。Prompt 如何被组装，工具何时被调用，状态发生了什么变化，哪个节点触发了重试，整条轨迹是否可回放，这些都不该依赖框架的黑盒 UI 猜测。Agent 不是传统后端服务，观测能力越弱，调试成本就会指数上升。

第四，**记忆应该成为基础设施**。无论是 chat history、episodic memory 还是 semantic memory，框架都不应假装自动帮你管好了。它应该把读写入口、检索策略、压缩与淘汰机制暴露出来，让开发者知道记忆究竟如何影响当前决策。

第五，框架必须原生支持**Human in the loop 与权限边界**。审批点、人工接管、回滚点、危险操作确认，不应该靠业务方事后补洞。智能体一旦走进真实工作流，人类不只是兜底者，也是系统的一部分。OpenClaw 和 Claude Code 已经告诉我们了，Human in the loop 是可靠的思路，不要总想着去完整的取代人。 现有的 Agent 产品也发现想要完全取代人只有一种可能，就是把人的经验以Workflow的形式封装，Use in the loop 对于目前的自主决策Agent无法被替换，纯粹的Agent结构难以在自由性和Robustness中找到一个合适的平衡。

最后，它还应该对**Agentic Coding** 友好。随着模型编码能力继续增强，越来越多系统会让 Agent 直接阅读、修改甚至扩展自己的运行逻辑。此时框架越贴近原生语言、原生工程对象和原生工具链，就越容易被人和 Agent 一起理解；越是高层魔法封装，越容易成为未来自动化自扩展的阻力。 或者说除了Agent Framework 以外，任何开发框架都需要向Agentic Coding进行适配，Training解决内建的基础知识，Agentic Doc处理AI利用框架的能力，从而进一步释放Coding Agent的逻辑。

归根结底，好框架的目标不是制造更高抽象，而是**减少认知债务，同时保留工程控制权**。它不是替开发者思考Agent 应该是什么，而是在开发者已经理解任务结构之后，提供那些确实值得被复用的基础能力。
