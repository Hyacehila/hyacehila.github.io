# Context is All You Need：智能体的上下文工程

## 引言：Context is All You Need

在前文[《从智能体的认知结构到智能体框架》](/blog/2026/03/03/cognitive-architecture-to-agent-framework/)中，我们以 CoALA 为框架讨论了智能体的认知结构，并得出了一个核心论断：**Agent 的工程发展史，本质上就是不断用外部的系统设计去弥补当前 LLM 在记忆和推理上的短板的过程**。同时我们也留下了一个悬念——Memory 将会被单独讨论。本文就是对这个承诺的兑现。

Context Engineering 和 Memory 应该被理解为一个高度交叉的系统。根据 CoALA 的设计理解，可以将 Context Engineering 的上下文管理视为对**工作记忆（Working Memory）**的管理——前者侧重资源管理范式，后者则侧重整体的认知记忆建模。Memory 可以被看做一个更广泛的范畴，而 Context Engineering 则是在强调工程上的实现，我们在之前的Memory只从认知神经科学的角度讨论了Memory应该是怎么样的，而现在我们则讨论在工程上如何将其融入一个智能体或者一个智能体框架中。

在 Linear Attention 真正实现之前，注意力机制带来的底层约束就是注意力不足和有限上下文。因此，**如何在有限的窗口中放入最有价值的信息，如何将不再需要的信息移出，如何在需要时将它们召回**——这三个问题，构成了 Agent Context Engineering 的全部。

持续学习与在线学习是 ML 的重要方向。基于 Context 的技术很容易做到这一点（因为 non-parametric），甚至还支持定向的遗忘。当我们不需要一边学习一边更新模型参数，只需要将与环境互动中学到的规则和经验整理成合适的数据存储为记忆，在下一次遇到同样的事情时能够快速检索记忆数据并作为上下文的一部分作为提示——在基础模型足够好用的情况下，这种自主学习就可以让智能体表现得更聪明，

这也将意味着我们将从The Second Half利用更好的评估系统和RL泛化去提升解决真实问题的智能，到Agent自我进化和泛化的阶段。从定义需要解决的问题到定义合适的智能结构（融入人工经验）直接解决问题，在这个阶段，非参数化记忆与Context的控制将会是解决问题的核心。

Context Engineering 充满了各式各样的工程问题和工程实践，因此这篇Blog目前且在未来的很长时间里可能仅仅是一篇草稿，而难以形成完整的实现指南。不过这也不重要，先开始写吧。哪怕比较零碎，会有形成体系的时候的。

## Agent Memory 的分层体系

### 严格区分 LLM Memory 与 Agent Memory

在讨论记忆之前，一个必须澄清的概念混淆是：**KV Cache、RoPE、Attention 变种等方法，其目的是扩展 LLM 的上下文窗口和能力，而不是为决策智能体提供记忆库**。它们属于模型架构层面的升级改进，而不是 Memory 系统的范畴。

LLM Memory 关注的是模型如何在推理过程中高效利用上下文窗口（如 Flash Attention、Ring Attention、KV Cache 压缩等），这是底层基础设施；而 Agent Memory 关注的是智能体如何在多轮决策中积累、检索和遗忘知识，这是上层应用逻辑。将两者混为一谈，会导致工程设计上的根本性错误。

### CoALA 视角下的记忆层次

回顾 CoALA 的认知架构，智能体的记忆被划分为四个层次：

- **工作记忆（Working Memory）**：保留活跃信息的决策周期，包括感知到的输入、推理生成的知识、从长期记忆检索的知识，以及从之前决策周期留下来的核心知识。**工作记忆可以被理解为我们的核心上下文窗口**——它一边与 LLM 交互，一边与长期记忆和外部环境交互。我们希望每进入一轮新的决策周期就进行一次记忆清理，记忆压缩就是针对工作记忆进行的。

- **情景记忆（Episodic Memory）**：存储决策的经验。当它进入工作记忆时，将被用于 in-context learning；反过来，一些决策经验也可以被写入情景记忆留给后续使用。这是一种长期记忆——检索认为它有用时才会被放入工作记忆。

- **语义记忆（Semantic Memory）**：如果说情景记忆存储的是经验，语义记忆存储的就是知识。两者都是为了加入 Context 帮助模型理解真实问题，但语义记忆更强调知识库本身。LLM 在使用过程中会积累 Episodic Memory，但一般不积累 Semantic Memory——后者需要外部的知识构建过程。

- **程序记忆（Procedural Memory）**：语言模型内建的知识，以及在智能体代码中直接编码的知识（比如图流转结构、外接的各种 Action 信息，封装的提示词以及Persona）。程序记忆是需要精妙的初始化才能启动智能体的，写入程序记忆来增加新的 action 是可行的，但也是危险的。

三种长期记忆的读写构成了智能体的**内部动作空间**，与之对应的外部交互则是**外部动作空间**。从时间顺序上观察一个智能体和智能体系统：智能体检索（长期记忆）→ 推理（工作记忆）→ 评估后续行动（LLM）→ 行动后观测（外部环境）→ 学习（更新长期记忆）→ 循环。

### 事实记忆、经验记忆与工程上的记忆系统

在通用 Agent Memory 的工程实现中，Token Level（语义层面）的记忆应该是研究的重点。而在记忆的语义层面，有一个常被忽略但极其重要的分类：

- **事实记忆（Factual Memory）**：保证 Agent 输出符合自我内建记忆设定以及宏观环境记忆设定的核心。作为一种语义记忆的子类，事实记忆需要被分离出来单独考虑——它直接决定了智能体的"世界观"是否一致。

- **经验记忆（Experiential Memory）**：其核心意义在于 Agent 的**非参数自我进化**。经验是案例库、案例总结，还是抽象为可执行代码片段的行动——这根本不重要。它们都是经验，应该一起分层存储，并且在需要的时候互补使用，依据情况进行更新。

这种分类比 CoALA 原始的情景/语义二分法更贴近工程实践：事实记忆关乎**稳定性**（你的 Agent 是否始终如一），经验记忆关乎**成长性**（你的 Agent 是否越用越聪明），这种思路可以作为CoALA的一个侧面。

针对特定场景来定制Memory结构，在各种工程与论文中都非常的常见，并且都能取得还不错的效果。但是如何将他们变成一个框架管理的问题，让开发者只需要利用框架就可以实现Memory管理，仍旧是未知数。很明显，对CoALA的工程实践或许还算容易解决，但是框架封装的难度是巨大的。最基本的，我们需要
* 工作记忆的压缩：保护有限的Context
* 分层的索引：压缩不等于消失，我们要维护索引保证可以随时再次唤醒相关记忆

同时，对于具备固定能力与风格的专用任务，通过微调形成参数化记忆是有价值的；对于高度集成的专用 Agent 系统，手动维护一定程度的记忆是有价值的，对业务的深度理解能够带来很多自动化Memory系统难以带来的优势（CLAUDE.md 以及 `/init` 命令）。但对于大部分简单的Agent开发来说，**自动化的记忆管理是方便开发工作的核心系统**，让你可以专注于开发逻辑。

### Agent as Memory

社区已经出现了大量试图将记忆管理通用化的框架和服务，它们明确区分短期存储和长期存储，并提供基于图、基于档案或模块化的记忆空间：

- **MemGPT**（Packer 等人，2023）：将操作系统的虚拟内存概念引入 LLM，实现上下文的自动换入换出
- **Mem0**（Chihikara 等人，2025）：面向开发者的记忆层服务，自动从对话中提取和管理记忆
- **Zep**（Rasmussen 等人，2025）：提供长期记忆与知识推理的企业级服务
- **LangMem**（LangChain，2025）：LangChain 生态的记忆管理组件
- **Cognee**、**Memary**、**MemEngine** 等：各具特色的开源记忆框架

这些框架的共性是尝试将记忆管理变成一个**基础设施**：用户只需要准备好数据就可以构建知识库并查询，而无需关注 RAG 的具体实现细节。如 Grok Collection 以及 Google 等各大公司提供的 RAG 服务，都在朝这个方向演进。这种基建服务需要简单到用户可以随便上传内容构建知识库，并在获得新查询时从知识库获取数据。

但是目前对记忆管理的设施还不够通用化，各大RAG服务只处理了记忆的一个小部分（检索），各种对记忆管理的尝试也停留在实验阶段而不是讨论其工程上实现的可用程度，指导智能体的开发者管理记忆只是第一步，而下一步应该是则应该将记忆功能本身封装到框架内部，仅暴露局部的接口来控制记忆的写入与读取。

目前来看用智能体本身作为记忆管理器（Agent as Memory Manager）是可能正确的选择。将记忆的组织、检索和遗忘交给一个专门的 Agent 完成，而不是依赖预设的规则。这对于多智能体的共享记忆尤其有意义：简单的全局共享并不正确，记忆的共享应该由具备语义理解能力的 Agent 来调控。

Agent as Memory 可能可以带来
* 允许对记忆重要性的声明：允许开发者用简单的语言声明记忆可能可以的设计方向，因为实现记忆的是一个语言智能体，具备处理不确定性的能力。
* 压缩但不损失信息：对话内部的上下文自动压缩是缓解单一智能体记忆有限的唯一解法，Agent as memory则可以让压缩本身更加智能。让压缩损失的信息有可能再次被唤醒。压缩和其他组件的结合才是解决Context问题的可能思路。
* 分层的记忆暴露：Agent Skills对记忆的暴露思路值得考虑，而它本身可以理解为一个Agent as Memory的简单实现，允许开发者通过一个很简单的机制实现长期记忆的分层逐步加载。
* 非常自然的长期记忆形成：当记忆本身被具备智能的语言模型控制，非参数的记忆更新将成为非常自然的事情。而如何反思性的整合更新则是此时要考虑的问题。
* 更加混合的记忆检索：检索是记忆的目标，甚至可能比记忆本身更重要。Agent as Memory则可以混合语义检索，关键词匹配甚至更加复杂的检索与Rerank机制，主Agent只需要用几句话告诉Memory自己可能需要什么，剩下的交给预先开发的智能体。他将检索，重排序，整合并进行信息压缩。

值得特别关注的是多项前沿研究，它们从不同角度探索了 Agent as Memory 的可能性：**CAM** 从皮亚杰的建构主义理论出发设计结构化记忆图式（附录 A）；**Memory-R1** 通过强化学习训练专门的 Memory Manager Agent 自主执行记忆的增删改操作（附录 B）；**ACE** 则提出将上下文视为不断演化的行动手册、通过反思与增量更新实现自适应记忆的方案（附录 C）。它们分别代表了理论驱动、数据驱动和工程驱动三条路径。此外，ACE 的设计思路建立在两项重要的前置工作之上：**GEPA** 代表了追求精炼简洁的进化式提示词优化方向（附录 D），而 **Dynamic Cheatsheet** 则开创了测试时自适应记忆的范式，直接启发了 ACE 的"不断完善而非压缩"的核心哲学（附录 E）。

## 检索：Memory 的另一面

检索是记忆的目标，甚至可能比记忆本身更重要。在 CoALA 框架中，Retrieval 从长期记忆检索内容到暴露到工作记忆里。处理 Context 的暴露和消除，就是解决一切 Context 问题的唯一思路。

### RAG 向量检索 vs Grep 关键词搜索

目前智能体的上下文检索存在两种主流方法：

- **基于 RAG 的向量数据库检索**：这是 Cursor 等编程 Agent 的选择。通过将文档嵌入为向量，利用语义相似度进行模糊匹配。其优势在于能够捕获语义层面的关联，但模糊且嵌入不一定可靠，并且系统更加复杂。RAG 的技术栈远比 Grep 丰富——涉及知识库构建、查询增强、检索器选型、向量索引和结果重排等多个环节（附录 F）。

- **使用 Grep 进行关键词搜索（字符串匹配）**：这是 Claude Code 和 Gemini CLI 的选择。直接通过关键词在代码库中进行精确搜索。其优势在于简单、确定性强，但会带来更大的 token 消耗，产生大量的无关匹配。

- **一种针对代码库的特殊情况**：LSP（Language Server Protocol）在编码任务中可以很好的进行记忆追溯，因为内建数据库（代码）本身存在这套规则，他对我们的启发时可能在构建真实Agent系统的时候考虑构造类似的可查询数据库，而不是在大量资料里检索。

两者各有优劣，但混合语义检索和关键词匹配得到一个可用的检索系统非常关键。在实际的 Agent 系统中，将两者结合使用——先用 Grep 快速定位候选，再用 RAG 进行语义精排——往往是更务实的选择。

### Agentic RAG：让检索进入决策循环

**Agentic RAG 是最接近 Agent Memory 的 RAG**，它可以被看做Agent as Memory的一个简单实现，让检索进入自主决策循环，并且兼容记忆的修改，维护内部、持久、自我进化的跨任务知识库。

传统的 RAG 是一次性的"给问题→查文档→拼答案"的流水线，而 Agentic RAG 的核心区别在于：

1. **自主性**：Agent 自己决定何时需要检索、检索什么、以及检索结果是否足够
2. **记忆修改**：检索不仅是读取，还包括对知识库的更新、整合和遗忘
3. **跨任务持久化**：检索到的知识可以沉淀为长期记忆，在后续任务中复用

同时需要注意，**RAG 从不局限于内建数据库，外部检索仍旧是检索**。Search Tool 侧重于外部的封装检索工具，而 RAG 应该侧重于从内建数据库召回。将内部检索和外部检索整合起来，才是完整的检索系统。

### 检索增强：查询重写与 Rerank

原始输入不是合适的查询。用户的自然语言提问和知识库中的数据之间往往存在巨大的语义鸿沟。如 AgentKB（Agent KB: Leveraging Cross-Domain Experience for Agentic Problem Solving）等研究指出，需要将原始的查询重写去弥补查询和数据库之间的语义差别。

AgentKB 通过其 Reason-Retrieve-Refine 流水线，在规划阶段使用跨领域工作流做种子引导，在反馈阶段应用定向修复，并通过"分歧门（Disagreement Gate）"确保检索到的知识增强而非干扰推理。

检索后的 Rerank 和信息压缩同样是构成系统的关键——检索到一百条结果和精准挑出三条最相关的结果，对 Context 的质量影响天壤之别。检索技术不仅仅是找出相关的上下文，还要去提供最有用的上下文，当我们做到了完美的检索，再搭配合适的遗忘机制就是一个优秀的auto context系统了。


### 分层级的上下文管理

自动上下文管理应该是分层级执行的：

- **编排层预加载**：Agent 在编排的时候提前完成一些上下文的脱离 context。它们应该放在共享存储中并在合适的时候交给 context。比如编程 Agent 中的项目初始化——将整个项目结构和内容摘要交给了共享存储，适时加载使用。

- **对话层自动压缩**：作为兜底的手段，避免用户和终端 Agent 过长的交流导致 context 爆满。这个自动压缩功能应该知道"谁更重要"——比如用户的提示就比自己索取的代码重要。它们最好 free 于场景，但针对 Tool Call 等高频操作需要考虑压缩。在任何场景下都可以使用**预设规则 + LLM 自主分析决策**，实现更好的上下文压缩。

- **记忆重要性接口**：这一点可能可以作为提供给开发者的接口——他们可以声明记忆应该如何设计，长期记忆包含知识和经验，如何把长短期记忆混合是最值得思考的问题。

实现对这个自动化管理是有难度的，将它和其他组件有机地结合起来，则是更重要的。

### 上下文隔离策略

如果说AutoContext实现了压缩和补充，那**隔离**就是利用Mutiple Agent 的思路来处理Context的问题，它也可以属于上下文处理的范畴：

**Subagent 上下文隔离**：子智能体只保留总结后的结果返回给主智能体，其探索过程中的大量中间信息不会污染主上下文。

### 遗忘：比加载更重要的操作

如果载入阶段的优化空间有限（如渐进式披露在全量载入后即失去意义），那么什么是真正重要的？答案是**遗忘**。

系统需要正确且及时地将非关键的信息从上下文中移出，腾出宝贵的注意力空间给核心任务：
- 对于某次错误的尝试，可以只保留最终结论，而遗忘试错的过程，或者将正确的做法加入长期记忆
- 对于某些使用过的工具，可以暂时遗忘，当需要使用之时重新加载回来

这如同计算机操作系统中的内存管理与 Cache 机制，及时地进行换入换出（Swap in/out）。当然，遗忘也是有成本的：一方面遗忘会导致 KV Cache Miss，增加模型调用成本；另一方面回忆起遗忘的内容，也要引入额外的 Token。这些都需要基于具体的业务场景，在重复评估、迭代的循环中进行权衡。

**更长的上下文不等于更高成本**——完全重写带来的缓存无法命中可能进一步拉高处理成本，而适当的增量更新可以更好地利用 KV Cache 技术。

### 走向类人记忆系统

分层次的记忆压缩是有必要的：维护一个到细节内容的索引，在有需要的时候引入——这或许就是类似 RAG 的记忆系统。如 **H-MEM**（Hierarchical Memory）将记忆组织为 Domain Layer → Category Layer → Memory Trace Layer → Episode Layer 四层结构，通过索引路由实现高效的分层检索，而不依赖穷举式的相似度计算。最好能让记忆本身存在一些逻辑上的层次关系，从而可以使用数据库进行结构化查询。

如果需要类人的记忆系统，我们需要在**完整记忆 + 压缩摘要记忆 + 向量与 Grep 检索机制**以外的一个关键组件——**遗忘更新机制**。记忆和检索是一体两面，加载和遗忘也是一体两面，而它们都在回应 Agent 最核心的开发概念——Context。

*一点题外话，类似的记忆系统不等于人，模型的主动交互不应该由人作为唯一的输入。人的大脑总是会产生某种随机想法或者因为环境和感知或者记忆而产生某种相关联的想法从而产生某个目的。那机器也一样，我们应该有一个主动意识产生器，他能根据环境感知和相关联的记忆产生某种目的，而这种目的可以引导承载AI模型的实体去主动与环境交互。这样我们就可以完成一个自主学习的闭环了。主动目的产生器根据记忆和环境与感知产生目的，AI Agent 与环境互动，并收集长期记忆，而长期记忆具备CoALA的三层架构，以及使用我们刚刚介绍的工程上的方法实现。关于规则和流程与认知放入永久存储，而对某些对话或者环境的印象根据长短期记忆的规则去淘汰最久不用的数据，对反复使用的知识增加留存权重。这样基础模型记忆主动意识产生交互学习可以形成闭环了。*



## 结论：Context 管理的核心矛盾与未来

回顾全文，Agent 的 Context 管理始终在处理一组核心矛盾：

- **丰富性 vs 高效性**：更多的上下文提供更好的决策依据，但也消耗更多的注意力资源
- **稳定性 vs 适应性**：事实记忆需要恒定不变，经验记忆需要持续进化
- **自动化 vs 可控性**：开发者需要足够的自动化来降低开发门槛，但又不能失去对记忆行为的掌控

针对场景来定制 Memory 结构和读取方式在各种 Memory 论文中都非常常见，但**如何将它们变成一个框架管理的问题**——让开发者只需要利用框架就可以实现 Memory 管理——这是真正的挑战。工作记忆的压缩肯定是重要的部分，但一定不是全部。开发者在手动优化记忆中一般是需要长期记忆一些东西：它们临时离开上下文，但会随时被检索系统介入。

**MCP 与 Agent Skills 解决了加载，但不能处理遗忘；AutoContext 等新探索则开始考虑遗忘。** 作为长期记忆的载体（如 MCP Server），应当被适当地检索后加入 Context；而当我们不需要它的时候，遗忘则应该是自然的。

Memory 应当成为基础设施——可能不是框架的附属品，但完全手动控制记忆不可接受。未来的 Agent 基础设施中，记忆管理应该像数据库之于 Web 应用一样，成为开发者可以信赖的、开箱即用的底层能力。在那之前，理解 Context Engineering 的本质和权衡，或许是每一个 Agent 开发者的必修课。

## 附录：Agent as Memory 的多条路径

本附录详细介绍五项代表性研究，它们分别从认知科学、强化学习、工程实践和提示词优化等方向探索了如何让智能体自主管理记忆和上下文。前三者形成一条递进的技术线索：CAM 回答"记忆应该被组织成什么结构"，Memory-R1 回答"记忆的增删改操作能否被学习"，ACE 回答"上下文如何在任务执行中自适应演化"。后两者则补充了 ACE 的设计背景：GEPA 代表了 ACE 所挑战的"追求简洁"的提示词优化范式，Dynamic Cheatsheet 则是 ACE "不断完善而非压缩"设计哲学的直接思想来源。

### 附录 A：CAM —— 建构主义智能体记忆

**核心问题**：如何为 LLM 阅读智能体设计一个有原则的记忆结构，而不依赖临时启发式？

**CAM（Constructivist Agentic Memory）**（Li 等人，arXiv:2510.05520，NeurIPS 2025）从 **Jean Piaget 的建构主义理论** 中汲取设计原则。Piaget 认为记忆不是被动的信息堆积，而是一个通过**同化（Assimilation）** 和**顺应（Accommodation）** 不断演化的认知系统——前者将新信息纳入现有图式，后者在新信息无法适配时调整图式结构。两者共同维持认知的**平衡态（Equilibration）**。

CAM 将这一理论映射为记忆系统的三个设计目标：

1. **结构化的图式（Structured Schemata）**：记忆被主动组织为**层次化的语义网络**。给定输入文本块 V = {v1, v2, ..., vn}，CAM 构建底层语义网络 G0 = (V, E)，然后逐层聚合为多层记忆图式 M = {G0, G1, ..., Gl}，天然整合抽象概念和细节信息。

2. **灵活的同化（Flexible Assimilation）**：每个底层信息单元可以同时归属于多个高层抽象节点（多对多映射），通过**重叠聚类（Overlapping Clustering）** 捕获复杂信息的多维关联。

3. **动态的顺应（Dynamic Accommodation）**：当图式局部过载或失衡时，只调整受影响的局部结构，而不需要全局重建——**增量式局部调整**确保了可扩展性。

#### 技术实现

CAM 的核心是一个统一了同化与顺应的**增量重叠聚类算法**，包含三步：

1. **基础网络扩展**：使用**语义相似度 + 位置邻近度（高斯衰减）** 的复合得分将新文本块整合进语义网络
2. **自我中心解纠缠（Ego-Centric Disentanglement）**：分析受影响节点的局部结构，通过节点复制显式解纠缠重叠聚类
3. **在线聚类更新**：增量标签传播算法更新聚类分配，LLM 摘要更新抽象节点，变化逐层向上传播

#### 记忆检索

CAM 采用模拟人类联想思维的两阶段检索：**快速定位（Fast Localization）** 通过余弦相似度选出 top-s 候选节点；**联想探索（Associative Exploration）** 由 LLM 沿图式结构迭代扩展激活集——收集激活节点的同层邻居和下层子节点，直到不再增长。这种**剪枝 + 生长（Prune-and-Grow）** 工作流结合了全局语义匹配和局部结构探索。

#### 评价与启示

CAM 在长文本阅读理解任务（问答、查询式摘要、事实验证）上展现了性能和效率的双重优势。其核心贡献在于从认知科学中提取了清晰的设计蓝图。但也存在局限：LLM 摘要过程中的**幻觉传播风险**、对**矛盾信息的处理能力不足**，以及**固定提示词而非可训练策略**。作者指出，未来引入可训练的记忆控制器是重要方向——而这恰好是 Memory-R1 所探索的。

### 附录 B：Memory-R1 —— 通过强化学习训练记忆管理智能体

**核心问题**：能否让模型自己学会什么该记、什么该改、什么该忘，而不依赖人工设计的规则？

**Memory-R1**（Yan、Yang 等人，arXiv，2025 年 8 月）通过**强化学习（RL）** 训练 LLM 学习如何主动管理外部记忆。传统方法大多依赖**静态的 RAG 检索**或**启发式规则**来管理记忆，缺乏一种**学习到的（learned）** 记忆管理机制。Memory-R1 的回答是：将记忆管理建模为 RL 问题，让模型从任务反馈中自主习得最优策略。

#### 双 Agent 架构

Memory-R1 设计了两个专门化的、通过 RL 微调的 Agent：

**Memory Manager（记忆管理器）** 负责动态维护外部记忆库，学习执行四种结构化操作：

- **ADD**：将全新的、值得记住的信息加入记忆库
- **UPDATE**：修正或丰富已有记忆（如"领养了一只狗 Buddy" -> "领养了两只狗，Buddy 和 Scout"）
- **DELETE**：移除被否定或过时的信息
- **NOOP**：信息已存在或无关时跳过

Memory Manager 被建模为策略函数，以提取的信息和检索到的相关记忆作为输入，输出操作类型和关联内容。关键在于，这个策略**通过 RL 从结果反馈中学习，而非由人工规则定义**。

**Answer Agent（回答智能体）** 利用记忆库回答查询，其关键创新是**记忆蒸馏（Memory Distillation）**：先通过 RAG 检索候选记忆（最多 60 条），然后智能过滤筛选出核心条目，最后基于精选记忆推理生成回答。检索到 60 条和精选出 3 条最相关的记忆，对回答质量的影响天壤之别——**减少噪声、避免被无关上下文干扰**。

#### 强化学习训练

两个 Agent 都通过**结果驱动的 RL** 微调，采用 **PPO** 和 **GRPO** 两种算法。训练的奖励信号来自最终任务表现（如 F1 得分），系统**端到端地学习**什么样的记忆管理策略最终能带来更好的回答质量。这避免了人工标注记忆操作标签的需求——仅需 **152 个问答对** 即可完成训练。

#### 评价与启示

Memory-R1 在 LoCoMo、MSC 和 LongMemEval 等基准上取得显著性能提升，且在 3B 到 14B 参数规模间展现良好泛化能力。其工程启示包括：

1. **记忆操作可以被学习**：ADD/UPDATE/DELETE/NOOP 构成完整动作空间，RL 证明了模型可自主学会何时执行哪种操作
2. **管理与使用分离**：Memory Manager 和 Answer Agent 的职责分离是优秀的架构选择
3. **端到端优化**：通过最终任务奖励反向优化记忆管理策略，避免了人工定义"好的记忆管理"
4. **记忆蒸馏的必要性**：在检索和使用之间加入智能过滤层是提升记忆利用效率的关键

与 CAM 的理论驱动路径相比，Memory-R1 走的是**数据驱动路径**——不预设记忆应该长什么样，而是让模型通过试错学习最优策略。

### 附录 C：ACE —— 自适应上下文演化

**核心问题**：如何让上下文在任务执行过程中自主演化，而不是被压缩到失去价值？

**ACE（Agentic Context Engineering）**（Stanford、SambaNova、UC Berkeley，arXiv，2025 年 10 月）提出了一种自适应的上下文管理模式。如果说 CAM 关注记忆的静态组织，Memory-R1 关注记忆的动态管理，那么 ACE 关注的是**上下文在使用过程中如何变得越来越好**。

#### 核心洞察

现有的提示词优化框架如 **GEPA**（Genetic-Pareto）将简洁视为优势，通过进化算法追求精炼指令。但 ACE 发现了一个反直觉的事实：**人类能从简洁中获益，但 LLM 相反，会在冗长的上下文中表现更好**。上下文压缩系统依赖 LLM 重写，随时间推移很可能导致领域知识和技术细节被压缩丢失。

基于这一洞察，ACE 借鉴 **Dynamic Cheatsheet**（动态备忘录，arXiv，2025 年 4 月）的设计哲学，将上下文视为不断演变的行动手册（Playbook），随时间不断完善而不是被摘要和压缩。

#### 方法：增量 Delta 更新

ACE 的上下文是**结构化的要点集合而非单一整体提示词**，方便增量更新而非完全重写。基于这个设计，ACE 引入三个核心模块：

1. **Generator（生成器）**：执行任务，产生结果和执行轨迹
2. **Reflector（反思器）**：从错误和成功中提取反思和 insights
3. **Curator（策展器）**：将相关内容整合到结构化上下文更新中

增量增长以外也需要 Refine——新的要点被增添的同时，需要 Refine 机制删除冗余。两者联合实现**自适应可解释上下文**。ACE 同时适用于**离线场景**（系统提示词优化，在训练集上迭代）和**在线场景**（测试时记忆适应，预测后更新上下文），在 AppWorld 和 FINER 等基准上持续优于包括 Dynamic Cheatsheet 在内的强基线。

#### 评价与启示

ACE 的关键点在于**能否反思得到有价值的 insight——如果反思不可靠，就是在进行上下文污染**。并非所有问题都需要丰富的 context 进行 learning——对于模型内在知识就能很好处理的问题，引入 ACE 只是浪费，GEPA 等追求简洁的框架就完全足够。ACE 的价值在于为知识密集型、需要持续适应的任务场景提供了系统化的上下文演化方案，但也揭示了一个更深层的矛盾：**上下文管理的收益高度依赖于任务特性和模型能力的匹配**。

### 附录 D：GEPA —— 进化式提示词优化

**核心问题**：能否通过自然语言反思而非强化学习的标量奖励，更高效地优化 LLM 的提示词？

**GEPA（Genetic-Pareto）**（Agrawal、Tan 等人，arXiv:2507.19457，ICLR 2026）是一个基于**进化算法 + 自然语言反思**的提示词优化器。在 ACE 的讨论中，GEPA 被定位为"追求简洁精炼"的代表——它通过多目标 Pareto 选择来进化出高性能的指令。理解 GEPA 的设计逻辑，有助于理解 ACE 为何选择了相反的方向。

#### 核心设计

GEPA 的名字来源于其两个核心机制：**Genetic（遗传进化）** 和 **Pareto（帕累托选择）**。传统的 RL 方法（如 GRPO）依赖稀疏的标量奖励信号来优化模型策略，而 GEPA 认为**自然语言本身就是比标量奖励更丰富的学习媒介**——LLM 可以通过分析完整的执行轨迹（推理过程、工具调用、错误信息）进行反思，诊断问题根因并提出针对性的修复方案。

GEPA 的工作流程由三个紧密耦合的组件驱动：

1. **反思式变异（Reflective Mutation）**：不同于传统遗传算法的随机变异，GEPA 使用 LLM 分析系统的执行轨迹和性能反馈。反思器诊断提示词失败的原因，然后生成有针对性的改进方案——这是一种**语义层面的定向变异**，而非盲目的文本扰动。

2. **Pareto 前沿维护（Pareto Frontier Maintenance）**：GEPA 不追求单一的"最佳提示词"，而是维护一个 Pareto 前沿——其中的每个候选方案至少在一个维度上是最优的。从 Pareto 前沿中随机采样父代进行变异，既保持了种群多样性，又避免了过早收敛到局部最优。

3. **交叉与整合（Crossover and Integration）**：GEPA 从 Pareto 前沿中选取互补的候选方案进行交叉操作，将来自不同"优势维度"的经验整合为更强的新候选。

#### 与 RL 的对比优势

GEPA 在六个任务上平均超越 GRPO 6%（最高 20%），同时仅需**不到 1/35 的 rollouts**。它还以超过 10% 的优势超越了领先的提示词优化器 MIPROv2（例如在 AIME-2025 上提升 12%）。这种效率优势的本质在于：RL 从标量奖励中学习（"这次得了 0.7 分"），而 GEPA 从完整执行轨迹中学习（"为什么这次失败了，哪条规则需要修改"）。后者的信息密度远高于前者。

#### 评价与启示

GEPA 代表了一种**追求精炼简洁**的上下文优化思路：通过进化算法不断精炼提示词，使有限的 token 承载最高密度的信息。这在模型内在知识即可处理的任务上非常高效。但 ACE 的研究揭示了一个反直觉的事实——**LLM 与人类不同，往往在更冗长的上下文中表现更好**。领域知识和技术细节在推敲精炼的过程中可能被误删。因此，GEPA 和 ACE 实际上代表了上下文管理的两个端点：**GEPA 追求信息的极致压缩，ACE 追求信息的持续积累**。最优策略取决于任务特性——对于推理密集型任务，精炼的 GEPA 提示词可能足够；对于知识密集型、需要持续适应的任务，ACE 的增量演化更具优势。

GEPA 已被广泛集成至 DSPy（`dspy.GEPA`）、MLflow、Pydantic AI、OpenAI Cookbook、HuggingFace Cookbook 以及 Google ADK 等主流框架，其通用性使其不仅可以优化提示词，还可以优化代码、Agent 架构甚至调度策略。

### 附录 E：Dynamic Cheatsheet —— 测试时自适应记忆

**核心问题**：能否让黑盒 LLM 在推理过程中自主积累经验，像人类一样从过去的问题中持续学习？

**Dynamic Cheatsheet（DC）**（Suzgun、Yüksekgönül 等人，arXiv:2504.07952，2025 年 4 月）为黑盒语言模型提供了一种**持久的、在推理时不断演化的外部记忆**。ACE 明确将 Dynamic Cheatsheet 作为其核心设计灵感——ACE 的"将上下文视为不断演变的行动手册"这一理念，正是对 DC "累积式学习"哲学的继承与发展。

#### 核心洞察

当前的 LLM 在部署后是静态的：每个输入查询被独立处理，模型不保留任何关于过去成功、失败或策略的显式记忆。这与人类认知形成鲜明对比——人类通过**增量学习**，不断将新经验内化到持久的心智模型中。DC 的核心洞察是：**不需要修改模型参数，仅通过动态管理输入上下文中的记忆内容，就可以实现测试时的持续学习**。

#### 两个核心模块

DC 的框架由两个模块组成，可以运行在同一个 LLM 之上（通过不同的 prompt），也可以分别运行在不同的模型上：

1. **Solution Generator（解题生成器）**：在第 i 步，生成器接收当前查询 xᵢ 和当前记忆状态 Mᵢ，产生候选解 ỹᵢ。记忆 Mᵢ 帮助模型复用或改编之前存储的策略、洞察和启发式规则。

2. **Memory Curator（记忆策展器）**：在生成器产生回答后，策展器对记忆进行更新。它主要评估三个方面：(i) 新产生回答的**有用性和可泛化性**——如果回答正确或包含有价值的洞察，将被蒸馏为适合后续引用的形式；(ii) **现有条目的修正或移除**——如果已有记忆条目被证明有误或被更优策略取代，策展器会更新或删除它；(iii) 整体记忆的**清晰度和紧凑性**——记忆条目被整合以保留简洁、高影响力的引用和启发式规则。

关键注意：策展器**没有访问真实标签的权限**——它必须自行评估解的正确性和效率。这使得 DC 完全兼容黑盒 API（如 GPT-4o、Claude），不需要任何梯度更新或人工反馈。

#### DC-RS：引入检索与综合

基础版本 DC-Cu（累积式）在处理完查询后更新记忆，但存在两个不足：模型在当前推理过程中无法利用本次查询的新洞察，且无法直接检索和利用历史响应。为此，DC-RS（检索与综合版本）引入了一个**检索机制**：在生成回答之前，先检索 top-k 个最相似的历史输入-输出对，连同当前记忆一起交给策展器更新，然后再基于更新后的记忆生成回答。这将记忆更新提前到生成之前，使模型能够在推理时即刻受益于最新的策略演化。

#### 惊人的实验效果

DC 展现了令人印象深刻的性能提升：

- **AIME 2024 数学竞赛**：Claude 3.5 Sonnet 的准确率从 23% 跃升至 50%——在积累了代数和组合洞察后**翻倍**
- **Game of 24**：GPT-4o 的成功率从 10% 飙升至 99%——模型发现了一个 Python 暴力求解方案并将其存入记忆，后续查询直接复用
- **方程平衡**：GPT-4o 和 Claude 从约 50% 跃升至近 100%——通过"回忆"代码化方法取代手动计算
- **知识密集型任务**：GPQA-Diamond 提升 9%，MMLU-Pro 提升 8%

但 DC 并非万能。**较小的模型（如 GPT-4o-mini）受益有限**——它们产生正确解的能力本身就不足，导致记忆中充斥着有缺陷的策略，且缺乏精炼存储内容的能力。DC 可以**放大强模型的优势**，但无法弥补弱模型在推理能力上的根本缺陷。

#### 与全量历史追加的区别

DC 与简单的"把所有历史对话拼接到上下文中"截然不同。后者会导致上下文长度膨胀，检索困难。DC 的记忆是**精心策展的**——只保留简洁、有用、可迁移的知识，而丢弃原始的冗长对话记录。这种**选择性保留**机制确保了记忆的质量和检索的可行性。实验也证实，DC 显著优于多数投票（Majority Voting）等被动聚合方法——主动的记忆精炼远比统计投票更有效。

#### 评价与启示

Dynamic Cheatsheet 的价值在于证明了一个重要命题：**非参数化的测试时学习是可行的，且效果显著**。它不修改模型权重，不需要训练数据标签，仅通过管理输入上下文中的记忆内容，就实现了类似人类累积学习的效果。

DC 对 ACE 的影响是直接的。ACE 继承了 DC 的两个核心哲学：(1) **上下文应该不断完善而非被压缩**——DC 的策展器持续丰富记忆内容而非摘要压缩；(2) **记忆应该是结构化、可策展的**——DC 存储的是精炼的策略和代码片段，而非原始对话。ACE 在此基础上引入了更系统的反思机制（Reflector）和增量 Delta 更新，将 DC 的思路从"测试时自适应"扩展为"上下文的持续演化"。

同时，DC 也暴露了这一范式的局限：**错误的启发式规则一旦进入记忆，会被持续放大**；**长上下文生成时模型倾向于截断或省略已有记忆**；以及**检索质量直接决定记忆利用效率**。这些问题在 ACE 中同样存在，是整个"自适应上下文"范式需要持续解决的核心挑战。

### 五项研究的交汇

CAM、Memory-R1、ACE、GEPA 和 Dynamic Cheatsheet 从不同维度回应了 Agent as Memory 的愿景：

| 维度 | CAM | Memory-R1 | ACE | GEPA | Dynamic Cheatsheet |
|------|-----|-----------|-----|------|--------------------|
| **核心问题** | 记忆怎么组织 | 记忆操作怎么学习 | 上下文怎么演化 | 提示词怎么优化 | 推理时怎么学习 |
| **方法路径** | 认知科学蓝图 | 强化学习训练 | 工程反思循环 | 进化算法 + 反思 | 测试时自适应 |
| **记忆结构** | 层次化语义图 | 外部记忆库 + 四种操作 | 结构化要点集合 | Pareto 候选池 | 策展式知识片段 |
| **更新机制** | 增量重叠聚类 | RL 策略决策 | 反思 + Delta 更新 | 变异 + Pareto 选择 | 策展器增量更新 |
| **上下文策略** | 联想式检索 | 记忆蒸馏 | 持续丰富 | 追求精炼 | 累积不压缩 |

五者形成了一个从基础到应用的完整图景：**CAM 提供记忆的结构蓝图，Memory-R1 证明记忆操作可以被学习，GEPA 展示了追求精炼的上下文优化路径，Dynamic Cheatsheet 开创了测试时累积学习的范式，ACE 则在 DC 的基础上实现了系统化的上下文持续演化**。其中，GEPA 和 ACE 代表了上下文管理哲学的两极——精炼压缩 vs 持续积累——而最优策略始终取决于任务特性与模型能力的匹配。

### 附录 F：RAG —— 检索增强生成技术

在正文中，我们讨论了 RAG 向量检索与 Grep 关键词搜索的对比。RAG 的技术栈远比 Grep 复杂，本附录对 RAG 的核心技术进行补充介绍，帮助读者理解为何 RAG 在 Agent Memory 体系中扮演如此重要的角色。

#### 为什么需要 RAG

在海量训练数据和模型参数的作用下，LLM 展现了令人惊叹的生成能力。但受限于训练数据的正确性、时效性和完备性，LLM 无法完全覆盖用户需求。由于现有训练范式的不完美，LLM 可能无法给出正确的答案，并产生**幻觉（Hallucination）**——看似合理但违背事实的输出。具体而言：

- **知识过时**：模型的知识停留在训练数据的截止时间，无法感知实时更新
- **知识边界与偏差**：大规模预训练的数据质量可能存在偏差，对某些垂直领域的信息覆盖不足
- **学习不充分**：即使训练数据中包含正确知识，模型也可能因学习程度不足、对齐偏差或解码随机性而给出错误回答

然而，如果我们在 Prompt 中直接通过**上下文学习（In-Context Learning）**的方式补充问题相关的知识，LLM 更大概率会给出正确的回答。这正是 RAG 的核心思想——**利用 LLM 的上下文学习能力与外部检索技术进行知识增强**，无需更新模型内在知识，避免了计算成本与对旧知识的遗忘。

一个基本的 RAG 系统包含三个核心组件：**外部知识库**（通常是向量数据库）、**信息检索器（Retriever）**、**生成器（即 LLM）**。给定自然语言问题后，检索器将问题编码并从知识库中检索相关文档，然后将补充知识与原始问题一起交给 LLM 生成输出。

> RAG 一般使用向量数据库，可以处理非结构化数据（如知识库文档）。如果考虑使用 MCP 等工具进行检索，则往往要求具备结构化数据库。

#### 知识库构建

构建高质量的知识库是 RAG 系统的基础，包含三个关键步骤：

1. **数据采集与预处理**：除了原始文本信息，还需要携带文档相关的**元数据（Metadata）**——这些元信息对后续的过滤与检索极有帮助。数据预处理主要包括**数据清洗**（去除特殊字符、HTML 标签、高度重复文档）和**文本分块**（将长文本切分为适配模型上下文窗口的段落，同时减少单个段落的信息量以增强检索精度）。

2. **知识库增强**：在数据处理完成后，需要改进并丰富知识库的内容与结构。**查询生成**，利用 LLM 生成与文档内容密切相关的伪查询，使用查询的视角来表示文档含义（作为文档的 key），方便后续查询的定位；**标题生成**，从概括总结的角度来索引信息，而非查询的角度。

3. **向量化与索引**：将处理后的文本通过嵌入模型转化为稠密向量表示，并存入向量数据库以支持高效的相似度检索。

#### 检索器：稀疏与稠密

检索器负责找到用户查询与知识库中的相关文档，主要分为两大类：

- **稀疏检索器（基于词频）**：使用 TF-IDF、BM25 等经典统计方法对文档进行编码，进行的是**文本匹配**检索，考虑的是词汇层面的重复与共现。简单、确定性强，但无法捕获语义关联。

- **稠密检索器（基于向量）**：利用神经网络对文本生成低维、密集的向量表示，通过向量之间的相似程度进行**语义检索**，考虑的是文本的深层含义。**稠密检索是目前热度最高、效果较好的 RAG 方式**。

此外还存在**生成式检索器**，通过生成模型直接对查询进行处理得到相关文档的标识符，但需要设计复杂的 DocID，在使用普及程度上远低于稠密检索器。

#### 查询增强

知识库中的知识形式有限，而用户的提问方式千变万化。通过对用户查询进行改写与增强，可以更好地匹配知识库中的文本：

- **同含义改写**：利用 LLM 将原始查询改写为多个含义相似但表述不同的版本，扩大召回范围
- **多视角分解**：通过分治方法，将复杂查询分解为不同视角的子查询，获取不同角度的信息
- **查询内容增强**：通过 LLM 生成原始查询的背景文档，提供更多的查询信息支撑

这些技术与正文中讨论的 AgentKB 的查询重写本质相同——弥补用户查询和知识库数据之间的语义鸿沟。

#### 向量索引与检索效率

知识库通常包含海量文本，逐一计算相似度极为低效。因此需要引入**向量数据库**来实现高效的向量存储与查询，其本质是一种近似最近邻（ANN）索引算法。常用的索引技术主要包括三类：

- **基于空间划分**：如 KD-Tree 等树结构索引，通过递归划分向量空间实现快速查询
- **基于图**：如 HNSW（Hierarchical Navigable Small World），构建图结构进行高效的近似搜索
- **基于乘积量化**：将向量分解为子空间的量化编码，压缩存储的同时保持检索精度

常用的开源向量数据库如 Chroma、Milvus、Pinecone 等都封装了这些索引算法，开发者通常只需了解大致原理并选择合适的工具即可。

#### 检索结果重排（Rerank）

如同推荐系统中的粗排与精排，RAG 也需要对初始检索结果进行重排序：

- **Cross-Encoder 重排**：使用效率更低但性能更高的交叉编码器对候选文档进行精细打分
- **LLM 重排**：如 RankGPT，利用 LLM 的理解能力通过滑动窗口方式处理大量文档进行精排

重排是 RAG 系统中常被忽视但极为关键的环节——它直接决定最终传入 LLM 上下文的信息质量。正如正文所言，检索到一百条结果和精准挑出三条最相关的结果，对 Context 的质量影响天壤之别。

#### 生成增强

生成增强研究如何将检索到的信息有效地交给 LLM。最主流的做法是将检索结果添加到 Prompt 中，通过上下文学习实现增强。同时，RAG 也需要考虑**降本增效**——并非所有检索到的文本都需要传入 LLM，筛选掉低相关性的内容可以减少 token 消耗，同时可能提升生成质量（减少噪声干扰）。

#### RAG 在 Agent Memory 中的定位

从 Agent Memory 的视角看，RAG 本质上解决的是**从长期记忆（语义记忆）中检索内容到工作记忆**的问题。一个完整的 RAG 流水线——知识库构建、查询增强、多路检索、结果重排、生成增强——正是 CoALA 框架中 Retrieval Action 的工程化实现。但传统 RAG 仅处理了记忆的**读取**；如正文所讨论的，未来的 Agentic RAG 还需要兼容记忆的**写入、更新和遗忘**，才能真正成为 Agent Memory 系统的核心组件。
