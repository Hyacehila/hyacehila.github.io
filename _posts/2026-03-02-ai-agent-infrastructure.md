# Agent 时代的基础设施建设

## 引言：新的抽象层

计算机系统经常沿着同一条路演进：常见问题被抽成更稳定的层。高级语言隐藏了一部分机器细节，云服务隐藏了一部分机房和运维细节。Agent 开发也会经历类似过程。

今天的 LLM 应用正在从“调用一个模型”变成“组织一套 Agent 系统”。开发者仍然要处理 Prompt、检索、记忆、上下文截断、工具调用和状态恢复，但这些工作不会永远停留在手工拼装阶段。检索、记忆和基础推理路由会逐渐变成可复用的 Agent 基础设施。

## 为什么 Agent 能力会基础设施化？

目前开发一个稳定的 AI Agent，认知负荷很高。开发者既要理解业务，又要理解模型行为，还要搭建脚手架，处理提示词、上下文管理、外部检索和工具执行。Agent 能力会被基础设施化，主要有几个现实原因：

- **减少重复造轮子：** RAG、工具注册、上下文压缩和任务状态管理已经有很多相似实现。把这些流程标准化，能减少每个项目从零搭一遍。
- **降低运行成本：** 向量检索、缓存、模型路由和推理服务一旦成为共享组件，就更容易做批处理、缓存命中和软硬件协同。
- **把注意力放回系统设计：** 单体 Agent 的基础组件足够稳定后，开发者才有空间处理多智能体协作、业务流程和产品边界。
- **帮助 Coding Agent 复用工程经验：** Coding Agent 越强，越需要稳定的项目接口、脚手架和公共组件，而不是每次都临场拼一套实现。

## Agent 基础设施的支柱

如果把 Agent 开发拆开看，下面几个方向会逐渐变成基础模块：

### Agent 框架（Framework）：搭建应用的基础

在 Agent Coding 的时代，框架的作用不只是降低人类的认知负载。相比多层高度抽象的框架，一个对人和智能体都更友好的 Agent Framework 更值得关注。由于现有 Agent 基础设施还没有形成确定模式，具备高扩展性的 Agent 框架比 All-in-one 框架更适合长期演进。之后会用一篇单独的 blog 讨论我理解中的 Agent Framework。

### 检索（Search / Retrieval）：AI 的感知与外部知识库

检索基建不会只停在“把 Query 丢给向量数据库，然后返回 Top-K”。更常见的形态会是混合检索（Hybrid Search）和主动检索：处理多模态数据，决定什么时候搜索互联网、什么时候查本地图数据库（GraphDB），并把噪声整理成模型能消费的上下文。检索讨论的是外部知识如何被暴露给 Agent。

### 记忆（Memory）：从无状态到持久化的状态机

LLM 可以看作无状态函数。记忆基础设施的任务，是提供低延迟、高召回的短期/长期状态管理。它不仅要记录发生了什么（对话历史），还需要具备自我反思（Reflection）能力，通过持续总结和压缩，形成用户画像与任务经验沉淀。

### 推理与规划（Reasoning & Planning）：AI 的大脑与路由引擎

作为基础设施的推理模块，需要在不同模型、不同成本和不同延迟之间做选择。路由技术和底层推理框架会共同回答这个问题：这一步任务应该交给哪个模型，用多少预算，失败后怎么升级。

### 其他：没有真理的时代

Agent 还没有稳定到能用一套教科书完全覆盖。这个系列会记录我认为有用的基础能力：扩展库、API 服务、协议、后端组件和开发方法。判断标准很简单：它是否能降低 Agent 开发和运行中的重复成本。

> 后面介绍的许多框架和 Package 功能高度重叠。有人从数据库切入做 Agent，有人从工作流切入，有人从多模型编排切入。由于行业标准（Protocol）尚未确立，后续讨论不仅会关注它们能做什么，也会分析底层架构取向，帮助大家在面对业务时做出更合适的选型。

## 漏洞抽象与结语

将 Agent 能力基础设施化，会遇到经典的**漏洞抽象（Leaky Abstractions）**风险。数据库、网络协议这类基础设施通常有更清楚的确定性边界；基于大模型的检索和推理组件则带有概率性。

当底层检索召回了错误内容，或者推理引擎在某一步判断失误时，上层业务如果只把这些组件当黑盒，就很难定位问题。Agent 基础设施必须暴露可观测性（Observability）和干预接口：能看到输入、检索结果、工具调用、状态变化和失败原因。

Agent 基础设施真正要解决的，不只是“把模型接进系统”，而是让模型相关的检索、记忆、推理和工具调用变得可维护、可调试、可替换。
