# 求职准备、项目串联与面试问答手册

## 面试通用策略

**经历的描述是重要的,不要按照时间流水线展开,研究目标,难点,做法,结果,价值,在做什么,承担了什么角色,要提前准备好.**

### 回答组织原则

- 面试时要做充分的信息沟通，不要只给出表面化、过短的回复，而要在合理范围内把背景、做法和判断讲完整，体现表达和协作能力。
- 推荐统一使用 STAR 法则提前细化自己的经历，把每段项目都整理成情境、任务、行动、结果四部分，避免临场想到哪说到哪。
- 讲经历时尽量不要简单按时间顺序流水账展开，而是优先按“目标 -> 难点 -> 做法 -> 结果 -> 价值”去组织，更容易让面试官抓住重点，也能避免在细节里越讲越散。


### 面试重点模块

- 常见主问题通常集中在科研经历、工程经历、实习经历和项目经历，本质上都在看你是否真正做过事，以及你在其中承担了什么角色。
- 代码测试既考基础算法知识，也考实际应用场景中的 Coding 能力；面试官看的不只是结果对不对，也会看你的实现习惯、调试思路和工程经验。
- 反向提问同样重要，它不仅是礼貌性环节，更是你获取团队、方向、评价标准和工作方式信息的重要机会；准备时也可以参考 `reverse-interview` 这类问题清单。

### 面试官重点观察的能力

- 面试官通常不只看技术本身，还会看你能不能把技术真正转化成产品或工作流价值。
- 除了技术实现，也要体现你能否识别新的业务需求，是否具备产品思维，而不是只盯着手头的模型或代码。
- 也需要体现一定的 Socialize 和信息感知能力，说明你会关注学术前沿、行业变化和外部协作，而不是只埋头做局部任务。
- 沟通表达能力非常重要，遇到问题时不要只做一句话式回答，而要展示你如何理解问题、拆解问题和推进问题。

### 遇到不熟悉问题时怎么答

- 如果遇到不熟悉的问题，先坦然承认自己不太会，不要硬编。
- 在承认边界之后，可以自然引导到自己熟悉的相关方向，说明你对相邻问题的理解、你会如何分析，以及如果进入工作会如何补齐。
- 这类回答的重点不是“装作都会”，而是体现诚实、学习能力和问题拆解能力。

## 绿盟科技实习问答

### 项目整体介绍

这段实习的核心目标，是为大模型在安全领域的 Post-training 提供高质量、可验证的数据。围绕这个目标，我把工作拆成两条并行推进的主线：一条负责“造数据”，另一条负责“用数据”。

第一条主线是构建生成数据的 CodeQL 代码审查 Agent。最开始的问题是，单纯依赖 LLM 生成安全分析数据，幻觉和不可验证结论都比较重。因此我做了一个面向 CodeQL 的 Single Agent，把信息检索、源码浏览、污点分析和 CodeQL 引擎验证串成闭环，让模型在真实环境反馈里不断反思和纠错，尽量保证沉淀下来的轨迹不是“看起来合理”，而是经过执行验证的可靠过程。

第二条主线是把这些过程变成可训练的数据。我搭建了底层漏洞数据管线，围绕语言类型和漏洞类型做样本分布控制；在 Agent 运行过程中，又进一步沉淀出 2,500 条结构化 SFT 轨迹和 300 条经验证的污点流数据。这样，前面的 Agent 不只是完成一次代码审查任务，也变成了后续冷启动和 RL 基线的数据来源。

### 高频问答

#### Q1：数据管线具体做了什么？

- 结果：我基于 GitHub 开源数据和内部数据库，提取并校验了 8,000+ CVE 实体，最终清洗沉淀约 4,000 条高质量样本，并完成了 CWE/OWASP 多维类型标注。
- 机制：核心流程是自动化预处理、正则与源码 Diff 抽取、实体校验和语义标注；在这之后我还做了一次样本混合，去控制漏洞类型和语言类型的分布。
- 价值/边界：这项工作的价值不只是“收集更多数据”，而是让数据分布更贴近训练需求；我这里强调的是数据工程和标注体系，不把它讲成完整的漏洞知识图谱系统。

#### Q2：为什么样本混合很重要？

- 结果：样本混合让最终训练集的漏洞类型和语言类型分布更接近目标配比。
- 机制：原始开源漏洞数据天然是长尾且偏斜的，如果直接拿来训练，模型会更偏向头部语言和高频漏洞类型，所以需要做受控混合。
- 价值/边界：这说明我不只是做数据抓取，还会考虑训练目标与数据分布的一致性；这里不需要展开具体采样参数，重点强调“让数据真正可用于训练”。

#### Q3：为什么选择 Single Agent，而不是 Multi-Agent？

- 结果：我最终落地的是 Single Agent，而不是多 Agent 协作框架。
- 机制：因为这条链路从信息检索到 CodeQL 验证是强依赖、强状态连续的，尤其真实环境执行后的报错和恢复决策需要统一的上下文和策略循环。
- 价值/边界：Single Agent 更适合沉淀稳定的 tool-use 序列和失败恢复逻辑；如果后续目标是并行化大规模扫描，再考虑 Multi-Agent 会更自然。

#### Q4：CodeQL 引擎验证到底验证什么？

- 结果：我会把验证概括成三层质量信号：命令是否正确、建库是否成功、最终查询是否编译/执行并返回可用信号。
- 机制：前面的信息检索、仓库浏览和污点分析是为了提出假设，CodeQL 验证则是把这些假设放到真实环境中检验，并把成功或失败的工具调用沉淀成可复用监督。
- 价值/边界：这一段不要只说成“跑了 CodeQL”，而要强调它是把多个工具串成闭环质量控制；但也不要夸大成完全自动化的工业级审计平台。

#### Q5：2,500 条 SFT 轨迹和 300 条污点流数据是怎么来的？

- 结果：我基于 500 个原始 CVE，沉淀出 2,500 条结构化 SFT 轨迹，以及 300 条经验证的可靠污点流数据。
- 机制：一个原始 CVE 会展开成多阶段过程，包括信息搜集、源码浏览、工具执行、分析收敛和验证通过；其中高质量的过程轨迹可以沉淀为 SFT 数据，而完整闭环且通过最终验证的部分可以沉淀为更强监督的污点流数据。
- 价值/边界：这里要强调“原始案例数”和“结构化轨迹数”不是同一层级，所以 500 对应 2,500 是自然展开结果，而不是数字矛盾。

#### Q6：冷启动数据不只是最终成功样本吗？

- 结果：不是，冷启动数据来源比“最终完全成功样本”更宽。
- 机制：信息搜集阶段里由 LLM 整理出的漏洞信息、正确 stop 的轨迹、自我验证通过的中间结果，以及最终查询通过的高质量样本，都可以作为冷启动来源，只是监督强度不同。
- 价值/边界：这样做能缓解冷启动阶段的幻觉和无效探索；但我会明确说明不同来源的数据可靠性有差异，不会把它们混成同一质量等级。

#### Q7：静态分析部分做到什么程度？

- 结果：我做的是偏启发式的静态分析辅助，而不是一个完备的静态分析器。
- 机制：我补充了一些预制的读取和搜索代码工具，让模型结合已有信息去生成语言级污点理解，并在过程中沉淀 tool-use 轨迹和带代码证据的分析报告。
- 价值/边界：这个回答要让对方理解我解决的是 Agent 分析链条中的“辅助推理与证据组织”问题，而不是从零自研了一个替代 CodeQL 的静态分析引擎。

#### Q8：SFT 和 RL 的分工是什么？

- 结果：我会把两者分成“先学会正确做事”和“再在真实约束里做得更稳更好”两层。
- 机制：SFT 主要负责工具调用格式、核心意图理解和推理模式；RL 则是在真实执行和验证约束下，激发并优化 Agent 行为，尤其关注执行成功率、验证通过率和长程决策稳定性。
- 价值/边界：这个回答可以体现你对训练范式的理解，但不要把 RL 讲成已经完成了大规模效果验证。

#### Q9：RL 部分哪些完成了，哪些还在规划？

- 结果：已完成的是规则导向奖励设计和基础 RL 训练流程打通；进阶部分还在规划和验证阶段。
- 机制：后续规划主要包括三块：一是用 Faiss+LSH 做向量抽样，提高高价值样本的覆盖效率；二是用 Curriculum Learning 做动态难度调节；三是用 LLM as Judge 提供更细粒度的奖励信号。
- 价值/边界：你要主动把“基线完成”和“进阶规划”分开讲，这样既能体现你理解 RL，也不会让面试官觉得你在夸大实验结果。

#### Q10：你在这段实习里最核心的贡献是什么？

- 结果：我最核心的贡献是把安全分析任务从分散的工具调用，组织成了一个可闭环、可验证、可沉淀训练数据的 Agent 工作流。
- 机制：我同时处理了数据、工具链和训练衔接三层问题：前面有数据清洗和分布控制，中间有 Single Agent 闭环和真实环境反思，后面有 SFT/RL 数据沉淀和 RL 基线搭建。
- 价值/边界：这个回答适合在开放式提问里用来收束全场，把你的定位落在“能把 Agent 设计、数据沉淀和训练衔接起来的人”。

### 补充高频追问

#### Q11：为什么必须用 CodeQL 做最终验证？

- 结果：我不会把前面的漏洞分析直接当成结论，最终一定要落到 CodeQL 的真实执行验证上。
- 机制：前面的信息检索、源码浏览和污点分析，本质上都是在形成漏洞假设；只有当建库成功、查询可编译、执行结果返回了可用信号，这个闭环才算真正落地。
- 价值/边界：这样做的价值是把“模型看起来说得通”变成“工程上可验证”；但我也不会把它讲成已经替代人工审计的完整工业平台。

#### Q12：CVE、CWE、OWASP 这些标注怎样服务下游训练？

- 结果：这些标注最直接的作用，是让漏洞样本从“能看”变成“能组织、能抽样、能对齐训练目标”的数据。
- 机制：CVE 提供原始案例入口，CWE/OWASP 补齐漏洞类型、攻击面和安全语义；这样后面无论做样本混合、类型覆盖检查，还是按任务目标组织训练集，都会有统一坐标系。
- 价值/边界：这说明我做的不只是原始数据抓取，而是在搭一套能支撑后续训练与分析的标签体系；但我不会把它讲成完整的安全知识图谱。

#### Q13：你说的“真实环境反馈”具体是什么？

- 结果：我说的真实环境反馈，不是模型自己判断“像不像对”，而是工具链执行后返回的客观信号。
- 机制：这些信号包括命令是否正确、CodeQL 数据库是否建成、查询能不能编译执行、返回结果是否可用，以及报错信息本身能不能帮助回退和修正。
- 价值/边界：它的价值是让 Agent 的反思建立在环境约束上，而不是建立在自我一致性上；但我不会把一次执行反馈讲成完整评测体系。

#### Q14：失败恢复怎么理解？

- 结果：我理解的失败恢复，不只是“这次没跑通再重试”，而是 Agent 能不能根据报错和中间证据调整后续路径。
- 机制：比如查询失败时，不是机械重复，而是回到证据链里重新确认污点路径、缩小代码范围、调整工具调用顺序，再进入下一轮验证。
- 价值/边界：这体现的是长程决策稳定性和闭环鲁棒性；但我会把它描述为已经打通的工作流能力，而不是已经全面优化到最优的恢复策略。

#### Q15：为什么 500 个 CVE 能展开成 2,500 条轨迹？

- 结果：因为一个 CVE 对应的不是一个静态答案，而是一整段可拆分、可复用的分析过程。
- 机制：单个案例会自然展开成信息搜集、仓库定位、代码浏览、工具调用、验证收敛等多个阶段；这些阶段里只要是高质量、结构清晰、可复用的片段，都可以沉淀成结构化轨迹。
- 价值/边界：这个回答的重点是区分“案例数”和“轨迹数”两个层级，说明数据沉淀是过程展开，不是简单重复计数。

#### Q16：冷启动数据和强监督数据怎么分层？

- 结果：我会把它们按可靠性和监督强度做分层，而不是混成一种数据。
- 机制：前面的漏洞信息整理、正确 stop 的轨迹、自我验证通过的中间结果，更适合作为冷启动或弱监督来源；而完整闭环、通过最终验证的污点流样本，才是更强的监督信号。
- 价值/边界：这样做能在早期缓解冷启动问题，同时保证高质量样本不被稀释；但我会明确说明不同层级的数据不应该被等价使用。

#### Q17：为什么不直接做 end-to-end 模型？

- 结果：因为这个任务天然需要工具、环境和验证约束，单纯 end-to-end 很难把中间错误暴露出来。
- 机制：漏洞分析不是只做一个分类或生成答案，它需要检索信息、读代码、做污点理解，再把假设放到 CodeQL 里验证；如果把这些都合并成一个黑盒输出，很多关键失败点是不可见的。
- 价值/边界：我选择 Agent 路线，是因为当前任务更需要可控性、可验证性和可沉淀的数据，而不是否定未来更强的端到端模型。

#### Q18：这个 Agent 的价值怎么说明？

- 结果：我会把它的价值概括成一句话：它把分散的安全分析步骤组织成了可验证、可复用、可沉淀训练数据的工作流。
- 机制：一方面它能把信息检索、源码理解和 CodeQL 验证串成闭环；另一方面它会把成功和失败过程都沉淀成后续可用的 SFT/RL 数据。
- 价值/边界：所以它的价值不只是“找到一个漏洞”，而是把分析流程产品化、数据化；但我不会把它讲成已经大规模替代人工分析员。

#### Q19：你做的是系统，还是训练？

- 结果：我会说我做的是“系统驱动的训练闭环”，而不是单纯只做其中一边。
- 机制：前面的 Agent 系统负责产生真实执行轨迹和验证信号，后面的 SFT/RL 则把这些轨迹变成可学习的监督；两边其实是互相喂数据、互相增强的关系。
- 价值/边界：这个回答能体现我不是只会搭流程，也不是只会讲训练，而是能把两者衔接起来；但我不会把自己包装成完整训练体系的唯一负责人。

#### Q20：你在这段实习里的个人 ownership 是什么？

- 结果：我的核心 ownership 更偏“把中间关键链路打通”，也就是把数据、Agent 和训练衔接成一个闭环。
- 机制：具体来说，我做了数据预处理与标注组织、Single Agent 分析闭环、SFT/RL 数据沉淀，以及基础 RL 基线这几层的衔接工作。
- 价值/边界：这个说法能把我的贡献落到真正有耦合度的关键链路上；同时我也不会把整个项目所有模块都笼统说成自己一个人完成。

#### Q21：你在这个项目里的安全理解体现在哪里？

- 结果：我会把自己的安全理解概括成一点：安全分析不是“说服别人你觉得像漏洞”，而是组织证据链并完成验证。
- 机制：从 CVE 到 CWE/OWASP 的结构化标注，到源码级污点分析，再到 CodeQL 的真实环境验证，这条链路本身就在逼着我用更偏安全分析的方式去理解任务。
- 价值/边界：这个回答能体现我不是只会调用模型，而是理解安全任务为什么强调证据和验证；但我也不会把自己讲成传统安全研究员出身的深度审计专家。

### 回答边界

- 不要说“绝对可靠”，统一改成“经验证的可靠”或“高质量”。
- 不要把启发式静态分析讲成完备静态分析器，也不要把 Agent 讲成已经完全替代人工审计。
- 不要把 RL 进阶方案讲成已经完成实验验证，统一说是后续规划或正在验证。
- 不要过度展开命令细节、建库细节和查询允许信号；除非面试官继续深挖，否则把重点放在闭环、质量信号和数据沉淀价值上。

## 可信舆情分析多 Agent 系统问答

### 项目整体介绍

这个项目不是一个简单调用大模型生成报告的脚本，而是一个面向舆情研究任务的多阶段 Agent 系统。它的主线是：先把原始社交媒体数据变成可分析的结构化样本，再让本地数据分析 Agent 和外部搜索 Agent 并行补证，最后把统计图表、搜索证据、论坛讨论和工具轨迹融合成带证据链的研究报告。

Stage1 的核心是补数据底座。原始微博或社交媒体帖子本身是非结构化文本，不能直接支持稳定统计和报告生成，所以系统会做六维增强，包括情感极性、情感属性、主题、发布者类型、信念信号和事件身份，同时补充关键词、实体、词典情感和文本相似聚类等本地 NLP 字段。这里的重点不是“用了 LLM”，而是把 LLM 输出限制在可验证的分类空间内，让后面的工具分析不会被开放式生成文本污染。

Stage2 是项目的核心亮点：双信源 Agent 加 ForumHost 的受控自主分析。DataAgent 基于增强后的本地数据调用 MCP 分析工具，产出图表、表格和统计事实；SearchAgent 基于外部搜索材料补齐官方回应、时间线和关联事件；ForumHost 负责综合两边结果，判断是否继续补数据、补搜索或补视觉分析。这样做的原因是，舆情分析既不能只看样本内数据，也不能只看外部新闻，两个信源必须互相校验。

Stage3 解决报告可信和过程可排错的问题。系统不是让模型一次性写完报告，而是先规划 outline，再并行生成章节 JSON IR，最后通过 ReviewChapters 做硬规则检查，包括占位符、缺少 `[E#]` 证据角标、非法方括号引用、重复标题、未授权 SWOT/PEST 组件等。配合 trace、status 和 Dashboard，分析员不仅能看到最终报告，也能看到每一步为什么发生。

如果用一句话总结：这个项目最有价值的地方，是把“单 Agent 黑箱生成报告”改造成“确定性主链 + 局部 Agent 循环 + 证据链审计”的可信分析工作流；如果放到 2026 年重新设计，我会继续保留三阶段主线，但把它升级为可组合、可记忆、可评测、可安全约束的 Agent runtime。

### 高频问答

#### Q1：这个项目的三阶段架构具体怎么理解？

- 结果：我会把它概括成“结构化数据底座、双信源 Agent 分析、证据链报告生成”三段。
- 机制：Stage1 把原始帖子增强为可统计、可检索、可图表化的结构化样本；Stage2 让 DataAgent 和 SearchAgent 分别处理本地数据和外部事实，由 ForumHost 控制补证和收敛；Stage3 把 outline、章节 JSON IR、证据角标和章节审查串起来，生成可追溯报告。
- 价值/边界：这样设计的价值是生命周期稳定、每层职责清楚，也方便测试和 Dashboard 观测；但我不会把它讲成完全自动化解决舆情分析，而是一个面向可信交付的系统框架。

#### Q2：为什么采用“确定性主链 + 局部 Agent 循环”？

- 结果：因为舆情分析是交付型任务，必须保证输入、处理、输出的顺序稳定；但分析阶段又确实需要 Agent 自主判断补什么证据、调什么工具。
- 机制：Stage1 的数据增强和 Stage3 的报告落盘更适合工作流，因为它们有明确输入输出和质量约束；Stage2 需要根据数据缺口、外部背景和图表异常做动态决策，所以把自主性集中放在 ForumHost 控制的局部循环里。
- 价值/边界：这个取舍能避免单 Agent 从头到尾自由发散，同时保留工具选择、补搜索、补视觉分析这类判断空间；我会强调这是工程上的可控自主，不是追求完全开放的 Agent 漫游。

#### Q3：为什么这里用 Multi-Agent，而不是 Single Agent？

- 结果：因为舆情分析天然是多信源任务，本地样本事实和外部事件背景需要分开处理、互相校验。
- 机制：DataAgent 只负责增强数据、统计工具和图表覆盖，SearchAgent 只负责公开信源、时间线和官方回应，ForumHost 再把两边的差异转化成下一步补证动作。这样分工也避免把“样本内统计结论”和“网页检索事实”混成一团。
- 价值/边界：Multi-Agent 的价值在于角色分离和证据制衡，不是因为多 Agent 天然更强；如果任务是强状态连续、强执行验证的链路，我反而会优先考虑 Single Agent。

#### Q4：ForumHost 具体解决了什么问题？

- 结果：ForumHost 解决的核心问题，是让多个 Agent 不只是并行产出材料，而是围绕同一个分析目标形成受控协作和收敛。
- 机制：它会综合 DataAgent 的统计事实和 SearchAgent 的外部背景，判断是否需要补充数据分析、继续搜索、查看图表或进入报告阶段；如果两边结论冲突，它不直接把冲突交给最终报告，而是把冲突转化为下一轮补证任务。
- 价值/边界：这个设计让系统更接近 orchestrator-worker 模式，但我不会把它说成已经是最优调度算法；更准确的说法是，当前已经把多信源协作和收敛机制工程化落地。

#### Q5：MCP 在这个系统里扮演什么角色？

- 结果：MCP 是分析工具的统一接入层，也是未来工具治理的基础。
- 机制：当前项目把 30 多个分析工具暴露为 MCP 工具，让 DataAgent 可以在运行时发现、选择、执行工具，并把执行结果写入 trace。下一步我不会简单增加工具数量，而是加入三层治理：任务相关的工具检索、每轮工具预算、文件写入或浏览器操作等高权限工具的权限控制。
- 价值/边界：MCP 的价值在于协议化、可替换和可审计，但 MCP 本身不等于可信；真正的可信还要靠工具选择质量、证据交叉验证、权限边界和人工监督。

#### Q6：Stage1 的数据增强为什么重要？

- 结果：Stage1 是整个系统的可信数据底座，它把海量无序原始博文转成后续 Agent 能稳定分析的结构化样本。
- 机制：系统会做清洗、语义标注和六维增强，把情感极性、情感属性、主题、发布者类型、信念信号、事件身份等字段落到候选集合里，同时补充关键词、实体、词典情感和相似聚类等本地字段。
- 价值/边界：这说明我不是只关注后端推理链，也关注数据入口是否可统计、可校验、可复用；但我不会把高并发清洗或标注吞吐量直接等同于分析质量，真正的质量还要看后续证据组织和审查。

#### Q7：Stage3 为什么强调证据角标和章节审查？

- 结果：因为 Agent 报告系统最容易失败的地方不是“写不出来”，而是“写得像真的但不可追溯”。
- 机制：系统先规划 outline，再生成章节 JSON IR，随后用 ReviewChapters 做硬规则检查：关键结论必须绑定 `[E#]` 证据角标，非法方括号引用、占位符、重复标题、未授权 SWOT/PEST 组件等都会触发修订。原有的内联引用、可靠性评分和人工复查入口，也都是为了让报告回到证据链上。
- 价值/边界：这能降低无依据扩写、引用漂移和自我强化幻觉的风险；但我只会说它提高可追溯性和可监督性，不会说它能彻底消除幻觉。

#### Q8：trace、status 和 Dashboard 为什么也是可信分析的一部分？

- 结果：可信交付不只是最终报告看起来合理，还要能解释每一步为什么发生、哪里可能出错、人工如何介入。
- 机制：系统通过 `trace.decisions`、`trace.executions`、`trace.reflections`、`trace.loop_status` 记录决策、工具执行、反思和循环状态；`status.json` 和 Dashboard 负责把 DataAgent、SearchAgent 等分支的 enter/exit、进度和异常暴露出来。
- 价值/边界：这让原本黑盒的 Agent 调度变成白盒化过程，方便复盘、调试和人工监督；前端在这里不是装饰性的 Demo，而是分析员检查系统运行状态的入口。

#### Q9：Context engineering 在这个项目里怎么落地？

- 结果：我会把它理解成动态组织上下文，而不是简单把内容越塞越多，或者在上下文太长时粗暴 compact。
- 机制：任务约束始终保留；工具说明按任务检索，只注入候选工具 schema 和必要示例；搜索证据保留 URL、时间、主体和摘要，正文按需展开；论坛历史保留每轮 conclusion 和 unresolved gaps；报告证据用 evidence cards 管理，正文只引用 `[E#]`。
- 价值/边界：这样能降低 token 浪费和模型跑偏概率，也能避免 compact 时丢掉关键证据；但它不是一个单独算法点，而是贯穿工具、搜索、论坛和报告的上下文治理策略。

#### Q10：trace 怎么升级成 trajectory evaluation 和长期记忆？

- 结果：当前 trace 更像一次运行内的审计日志，下一步可以把其中可复用的部分沉淀成评测和记忆。
- 机制：trajectory evaluation 可以基于 trace 判断每轮工具选择是否减少盲区、是否重复调用低价值工具、是否及时补搜索或收敛；长期记忆则可以把历史任务、配置、失败原因、工具表现、证据卡片和用户偏好分别沉淀成 run memory、tool memory、evidence memory 和 preference memory。
- 价值/边界：真正有价值的记忆不是把上下文无限变长，而是在新任务开始时检索少量高相关经验影响决策；我会明确说这是 P1 方向，不把它讲成已经完全成熟的生产级 memory layer。

#### Q11：如果 2026 年重新设计，你会怎么升级这个项目？

- 结果：我不会推翻 Stage1/Stage2/Stage3 的主线，因为这条主链保证了交付稳定性。我真正想升级的是 Stage2 的 Agent runtime：从“图里固定流转的 DataAgent/SearchAgent/ForumHost”升级成“ForumHost 作为 orchestrator，动态调度多个 specialist worker”的真多智能体系统。
- 研究依据：2026 年这几个月 Agent 方向的几个信号都指向同一件事：复杂任务开始从单 Agent 工具调用转向 orchestrator-worker 式并行研究；Skills 强调用 progressive disclosure 管理长提示词和领域流程；MCP 生态增长后重点从“能调用工具”变成“工具治理和权限边界”；Agent memory 研究开始从简单检索走向 trajectory、因果关系和长期经验复用；OpenClaw 类系统则说明 browser/computer-use 能力有价值，但安全边界必须前置。
- 开发路线：第一步保留 PocketFlow 主链，只替换 Stage2 内核。原来是 DataAgent、SearchAgent、ForumHost 三个节点按图流转；升级后 ForumHost 变成 Lead Orchestrator，先根据任务拆出 research plan，再动态派发 Data Analyst、Search Analyst、Visual Analyst、Skeptic/Verifier、Report Planner 等子 Agent。每个 worker 有独立 brief、工具预算、上下文窗口和输出 schema，最后把 finding cards 回交给 ForumHost 汇总，而不是直接进入报告。
- Stage2 的新协作协议：Data Analyst 负责本地数据、统计函数、图表覆盖和异常点；Search Analyst 负责外部信源、官方回应、时间线和争议点；Visual Analyst 负责图表视觉解读、异常趋势和报告可读性；Skeptic/Verifier 负责反驳结论、寻找证据缺口、标记幻觉风险；Report Planner 只负责把已经收敛的证据组织成章节计划。ForumHost 不再只是状态机节点，而是负责分配任务、读取 worker 结果、判断是否需要追加 worker、控制收敛条件。
- P0：先做“真多智能体 Stage2”。落地工具包括 worker registry、role prompt/Skill 装载、任务队列、独立 local shared、finding card schema、证据去重、工具预算和 trace-based trajectory evaluation。成功标准不是 Agent 数变多，而是每轮补证能减少 blind spots，报告证据缺口更少。
- P1：做 Skills 和 memory。把每个阶段的稳定经验沉淀成 Skills；把历史 trace 转成 run memory、tool memory、evidence memory、preference memory；让新任务开始时能召回相似事件、有效工具组合、常见失败模式和已核验证据。
- P2：引入受限 browser/computer-use verifier。它只负责网页核验、截图、发布日期和来源确认，不参与自由写作，也不拥有完整文件系统或终端权限。
- 取舍：真多智能体会带来更强的探索能力和证据覆盖，但 token 成本、延迟、并发错误、结果合并和评测都会更难。我的判断是只在 Stage2 引入，因为 Stage2 是信息量最大、最适合并行探索的部分；Stage1 和 Stage3 仍保持确定性流程，避免整个系统变成不可控的 Agent 实验场。

#### Q12：Skills 对这个项目有什么价值？

- 结果：Skills 的价值是补齐“MCP + prompt”组合的短板。MCP 解决的是工具怎么被调用，prompt 解决的是当前轮怎么指挥模型；但复杂项目里还需要长期稳定的操作手册、案例、schema、失败处理和阶段经验，这部分更适合用 Skills 做渐进加载。
- 和 MCP+prompt 的区别：MCP 暴露的是函数和数据源，例如统计工具、图表工具、搜索工具；Skill 暴露的是“怎么完成一类任务”，可以包含阶段目标、输入输出约定、工具选择策略、常见错误、示例、参考表和模板。这样模型启动时只看到 Skill 的 name/description，真正需要时再加载 `SKILL.md`，更长的参考资料、脚本和模板再按需加载，不用把所有提示词一次性塞进上下文。
- Stage1 Skills：`social-data-enhancement` 负责清洗、六维增强、候选标签校验、断点续传和异常样本处理；`schema-validation` 负责字段完整性、标签候选集、重复样本和结构化输出校验。它们让数据增强阶段不只靠 prompt，而是有稳定的分类标准、反例和错误恢复步骤。
- Stage2 Skills：`opinion-analysis-toolkit` 服务 Data Analyst，描述 MCP 工具怎么选、统计函数怎么组合、图表覆盖如何判断、什么时候需要补跑函数；`evidence-search-and-verification` 服务 Search Analyst，描述搜索查询改写、信源分级、时间线抽取和网页可信度判断；`visual-signal-analysis` 服务 Visual Analyst，描述图表异常、拐点、比例误读和视觉证据如何转成 finding card；`forum-host-orchestration` 服务 ForumHost，描述如何分派 worker、如何合并冲突、如何定义收敛；`skeptic-verifier` 服务 Skeptic，描述如何找证据缺口、反驳过强结论、标记不确定性。
- Stage3 Skills：`evidence-bound-reporting` 负责 `[E#]` 引用、章节 JSON IR、证据卡片、禁止无依据扩写和 ReviewChapters 修订策略；`report-style-control` 负责面向面试或业务报告的表达粒度、标题结构和可读性；`agent-run-audit` 负责 trace 解读、失败定位和 trajectory evaluation。
- 取舍：Skills 会让提示词准备更充分，也更利于复用和协作，但会带来版本维护、Skill 污染、过度碎片化和触发错误的问题。我的策略是只给稳定、高频、边界清楚的能力做 Skill；每个 Skill 要有 owner、版本、适用条件和反例，外部 Skill 需要 review，不能让 Skill 变成另一个不可控 prompt 仓库。

#### Q13：OpenClaw 或 browser/computer-use 能力应该怎么接入？

- 结果：我会把 browser/computer-use 接成“Evidence Verifier Agent”，只服务证据核验，不让它接管整个分析流程。OpenClaw 类系统说明自主执行能力在增强，但 2026 年的安全研究也说明，拥有本地文件、浏览器、账号和命令行权限的 Agent 会显著扩大攻击面。
- 适合做的事：打开搜索结果网页，确认标题、作者、发布时间、来源域名和正文是否支持 SearchAgent 的摘要；保存网页截图、PDF、公告页和关键表格；检查同一事实是否被多个信源互相印证；识别网页里的 prompt injection 文本，把网页内容作为 untrusted evidence，而不是 instruction。
- 不应该做的事：不让它自由浏览无关网页，不让它登录私人账号，不让它改项目代码，不让它写报告正文，不让它执行终端命令，不让它访问全盘文件系统。它的写入范围只允许是 `report/evidence/` 或等价证据目录，输出只能是 evidence card、screenshot path、source metadata 和 risk flag。
- 接入流程：Search Analyst 先产出待核验 URL 和核验问题；ForumHost 判断是否值得启动 verifier；Evidence Verifier 在沙箱浏览器里打开网页并截图；Verifier 把标题、日期、来源、摘要、截图路径和可信度写成 evidence card；Skeptic/Verifier 再检查这张 card 是否能支撑报告里的具体结论。
- 安全设计：浏览器默认无登录态、无账号密钥、无自动下载执行；所有外部内容进入系统时都标为 untrusted；网页正文只进证据解析器，不进入系统指令；高风险动作需要 Dashboard approval；每次浏览、截图、下载和写入都进入 trace。
- 取舍：浏览器核验会提高证据可信度，尤其能避免 SearchAgent 摘要错误、网页过期和来源伪造；但它会增加延迟、成本和运行不稳定性，也会引入 prompt injection、下载文件安全、动态网页反爬和截图存储成本。所以我会把它作为 P2 能力，只对高价值事实、争议事实和最终报告关键证据启用。

#### Q14：Agent 安全在这个项目里怎么体现？

- 结果：Agent 安全不是附属问题，而是未来升级的前置条件。当前系统已经有三类基础安全机制：ForumHost 的多轮辩论和交叉质疑、报告生成阶段的证据引用和自我一致性校验、trace/status/Dashboard 的白盒监控。下一步要补强的是：当外部检索、MCP 工具、Skills、长期记忆和浏览器 Agent 都进入系统后，如何把不可信输入隔离在证据层，而不是让它污染指令层和记忆层。
- 威胁模型：第一类是不可信网页和搜索结果，它们可能包含 prompt injection、SEO 垃圾页、过期页面、来源伪造、断章取义摘要或与事实冲突的材料；第二类是检索引擎本身返回低质量结果，包括 query 改写失败、召回不足、排序偏差、snippet 与原文不一致、发布日期缺失或把转载源排在原始源前面。Tavily 这类检索引擎本身会做质量和安全过滤，但系统不能因此默认“检索返回就可信”，SearchAgent 只能把它当作候选 evidence。
- 检索故障处理：SearchAgent 拿到结果后先做结构化校验，检查 URL、域名、标题、发布时间、作者/机构、摘要和来源类型是否齐全；再做信源分级，优先官方公告、权威媒体、原始平台和可追溯数据源，降低营销页、二次转载、匿名帖和时间不明页面的权重；再做交叉验证，同一关键事实至少需要本地数据、外部信源或浏览器核验中的两类证据互相支持。若只有单一检索结果支撑，就只能标成 low-confidence evidence，不能支撑强结论。
- 检索失败恢复：如果结果不一致、来源弱、日期缺失或和本地数据冲突，ForumHost 不直接进入报告，而是触发 query rewrite、补充搜索、换关键词、限定站点/时间范围，必要时启动 Evidence Verifier Agent 打开网页核验标题、发布时间和正文。若仍无法确认，就在 evidence card 中标记 unresolved/uncertain，让报告写成“目前检索材料不足以证明”，而不是让模型补全结论。
- 防护设计：外部网页永远只作为 evidence，不作为 instruction；MCP 工具按 read/analyze/write/browser/admin 分级，默认只开放 read/analyze；写文件、访问网络、浏览器、下载和认证类工具需要审批；工具返回值做 schema 校验、大小限制和来源标记；Skill 需要版本锁定、来源 review、allowed-tools 限制和回滚；memory 写入要有 provenance、timestamp、confidence、superseded-by/contradicted-by 标记，未经核验的检索结果不能直接写入长期记忆。
- 运行时治理：ForumHost 负责把安全策略纳入调度，例如低置信检索结果触发补证，高风险工具调用先走 approval；Skeptic/Verifier 负责检查证据是否被网页指令污染、是否存在单源强结论、是否有检索结果和本地数据冲突；Dashboard 展示所有高权限动作和低置信证据；trace 记录 query、返回结果、筛选原因、证据等级、核验动作和最终是否进入报告。
- 取舍：这些控制会增加搜索轮数、延迟和实现复杂度，也可能让报告在证据不足时更保守；但这是必要的，因为舆情分析系统的可信性不只取决于模型会不会写，也取决于检索结果、工具权限、报告证据和长期记忆是否可审计、可降级、可撤销。

#### Q15：开发中最大的 Coding 难题是什么？

- 结果：当前最大的 Coding 难题是异步双 Agent 运行时；如果按未来规划继续升级，最大的难题会变成“动态多智能体 runtime 的状态隔离、调度、合并和评测”。
- 当前难点：主流程已经运行在 PocketFlow `AsyncFlow` 里，DataAgent/SearchAgent 分支内部也要跑异步 flow，直接嵌套 `asyncio.run()` 会有事件循环冲突；所以分支运行要放进 worker thread，线程内再运行子 flow。MCP 工具调用又涉及 stdio 子进程和线程池；两个分支不能共享同一个 `shared`，否则会有并发写入、trace 覆盖和状态污染，所以要用 local shared 隔离，完成后按协议合并 charts、tables、execution_log、trace 和 search result。status 事件还要用 branch_id 区分不同分支，让 Dashboard 能看见每个分支的 enter/exit 和异常。
- 未来难点：真多智能体以后，worker 数量不是固定的，ForumHost 可能根据 blind spot 动态追加 Search Analyst、Visual Analyst 或 Skeptic。这里需要实现 agent registry、任务队列、并发预算、取消机制、超时处理、worker 输出 schema 校验、finding card 去重、冲突合并、证据引用合并和失败恢复。否则 Agent 多了以后，系统会从“更聪明”变成“更难调试”。
- Skill 和 memory 的工程难点：Skill 触发要准确，不能每次都加载一堆无关说明；Skill 版本要和运行 trace 绑定，方便复盘；memory 写入不能简单 append，要区分 run memory、tool memory、evidence memory 和 preference memory，还要处理更新、删除、冲突和过期。结合 AMA-Bench 这类 2026 年 agent memory 评测思路，以及 Memory-R1/agent memory manager 这类把 ADD/UPDATE/DELETE/NOOP 显式化的方向，我会避免只做相似度向量检索，而是把“哪个工具导致了哪个发现、哪个证据支撑了哪个结论、哪条旧经验被更新或废弃”一起写入记忆。
- 评测难点：多 Agent 路径不是固定的，不能只测“有没有走某条流程”。更合理的是做 checkpoint 和 end-state evaluation：证据是否覆盖关键主体、工具调用是否减少盲区、关键结论是否有 `[E#]` 支撑、Verifier 是否发现了不确定性、最终报告是否少了无依据断言。trace-based trajectory evaluation 会成为开发闭环的一部分，而不是上线后再补的评测。
- 取舍：这个升级会明显增加工程复杂度，包括并发调度、成本控制、日志体积、测试难度和失败恢复成本。因此我会分阶段做：先把现有双 Agent 分支的 trace/schema/合并协议规范化，再加 Skeptic/Verifier 作为第三个真实 worker，最后才开放动态 worker spawning。每一步都要有回退路径，保证失败时系统仍能退回确定性主链。

#### Q16：这个项目的核心贡献和边界怎么讲？

- 结果：我最核心的贡献，是把舆情分析从“会生成结论”推进到“能组织证据、能交叉验证、能白盒交付、能继续演进”的 Agent 工作流。
- 机制：具体来说，我把结构化数据增强、双信源 Agent、ForumHost 收敛、MCP 工具调用、证据链报告、trace/status 可观测性和后续 runtime 升级路线串成了一条完整链路。
- 价值/边界：这个回答适合收束全场，突出我做的是系统级可信 Agent 设计；但我会明确说它仍是面向可信分析的工程探索，不是已经通过完整 benchmark 证明的成熟平台，也不是对人工分析员的完全替代。

### 回答边界

- 不要把“可信”讲成“绝对正确”或“完全没有幻觉”，统一改成“通过多重机制降低风险、提高可监督性”。
- 不要把 ForumHost 讲成已经验证最优的调度算法，要强调它是把多 Agent 协作机制工程化落地。
- 不要把 P0/P1/P2 升级路线讲成已经全部完成，已完成和规划中的能力要分开。
- 不要把 browser/computer-use Agent 讲成无限权限自主执行，统一强调受限证据核验、沙箱、审批和 trace。
- 不要补充简历或材料里没有写明的量化指标、benchmark 成绩或部署规模，重点放在架构设计、证据组织和可信交付上。

## 跨项目对照问答

### 高频问答

#### Q1：为什么绿盟用 Single Agent，而舆情项目用 Multi-Agent？

- 结果：因为两个任务的证据结构和执行结构完全不同，所以最合适的 Agent 组织方式也不同。
- 机制：绿盟那条链路从信息检索、源码浏览、污点分析到 CodeQL 验证，是强依赖、强状态连续、强执行反馈的闭环，更适合用 Single Agent 维护统一上下文和失败恢复策略。舆情项目是开放世界、多信源、多视角任务，本地样本、外部搜索、图表解读和报告表达需要互相制衡，所以更适合 Multi-Agent 加 ForumHost。
- 价值/边界：这个回答的重点是说明我选 Agent 形态是任务驱动，不是追求 Single Agent 或 Multi-Agent 标签。

#### Q2：这两个项目共同的方法论是什么？

- 结果：我会把共同方法论概括成一句话：让 Agent 不是只会回答，而是围绕证据、工具、验证、记忆和评测去工作。
- 机制：两个项目都不是裸 LLM 生成答案。绿盟项目强调信息检索、CodeQL 工具验证、失败恢复和 SFT/RL 数据沉淀；舆情项目强调结构化数据、MCP 工具、双信源交叉验证、证据角标、trace 审计和后续 trajectory evaluation / memory。
- 价值/边界：这能体现我关注的是 Agent 的工作流化、可验证化和可沉淀化，而不是单点模型能力展示；但不同项目的验证方式不同，不能把安全任务里的 CodeQL 验证直接等同到舆情任务里。

#### Q3：这两段经历如何体现你兼顾研究和工程？

- 结果：我会说这两段经历一个让我把 Agent 可信性落到系统里，一个让我把系统运行轨迹继续转回训练和评测问题。
- 机制：舆情项目里，我更偏架构设计、协同机制、证据链报告、可观测性和 Agent runtime 演进；绿盟项目里，我更偏可验证安全任务下的数据管线、Tool-use 闭环、环境反馈和 SFT/RL 衔接。
- 价值/边界：这样回答可以同时体现工程落地和训练理解；但我不会把自己包装成纯研究或纯工程的单一角色。

#### Q4：如果被问你更偏产品、算法还是工程，怎么答？

- 结果：我通常会回答自己更偏“工程落地驱动下的算法与系统结合”。
- 机制：因为我做项目时，习惯先把任务链路和证据结构拆清楚，再决定哪里需要 Single Agent、哪里需要 Multi-Agent、哪里需要工具治理、哪里要沉淀记忆或训练数据，而不是先选一个流行方法再硬套场景。
- 价值/边界：这个回答既能体现我有算法理解，也能说明我更关心系统最后能不能真的跑起来、交付出去。

#### Q5：这两段经历最终体现了你的什么核心能力？

- 结果：我觉得它们共同体现的是，我能把 Agent 的架构设计、工具接入、数据沉淀、验证机制、记忆和评测意识串成一条完整链路。
- 机制：舆情项目锻炼的是我在开放场景下组织多源证据、MCP 工具、trace 审计和可信交付的能力；绿盟项目锻炼的是我在可验证任务里把数据、工具、环境反馈和训练闭环衔接起来的能力。
- 价值/边界：这个回答适合在最后收束时使用，把自己的定位落到“能把 Agent 真正做进工作流，并且知道如何让过程可检查、可复用、可迭代”的人。

#### Q6：如果面试官追问你为什么一直围绕 Agent 来做项目，怎么答？

- 结果：因为我更感兴趣的不是模型单点能力，而是模型怎么在真实任务里持续工作、接受约束并产生可复用价值。
- 机制：无论是舆情分析还是安全分析，我都会自然关注数据入口、工具调用、证据组织、验证反馈、记忆沉淀、评测和人工监督，因为这些才决定了系统能不能真正被使用。
- 价值/边界：这个回答能把项目选择讲成一致的方法论和问题意识，而不是简单追热点；同时也能说明我关心的是可控 Agent runtime，而不是无限自主的通用 Agent。

#### Q7：如果被问两个项目和 Agent 现代化趋势有什么关系，怎么答？

- 结果：我会说它们都在回答同一个问题：怎么把 Agent 从“会调用模型”推进到“可组合、可验证、可记忆、可治理”的系统。
- 机制：绿盟项目更靠近工具闭环和训练数据沉淀，强调真实环境反馈如何变成 SFT/RL 数据；舆情项目更靠近多智能体编排、MCP 工具治理、Context engineering、trace-based evaluation 和长期记忆。两者合起来，能覆盖 Agent 落地里从执行、验证到经验复用的关键环节。
- 价值/边界：这个回答适合在高阶面试里使用，但要注意把已完成能力和后续规划分开，不要把趋势判断包装成已经充分验证的系统成果。

### 统一回答边界

- 不要说“完全可信”“完全自动化”“已经显著验证优于人工”，统一改成“面向可信”“工程上可验证”“通过多重机制降低风险”。
- 不要把架构规划、后续对齐方案和未完成评测讲成既定结论，统一说“已经完成的是哪些，仍在迭代的是哪些”。
- 不要把系统里的辅助分析、交叉验证和白盒化能力讲成对人工的完全替代，统一强调“提升效率、提高可监督性、支持分析员决策”。
- 如果面试官深挖细节，优先回到证据链、工具链、质量信号和边界条件上，不要为了显得厉害去补没有根据的数字或实验结论。
