# 从进程、线程到协程与异步：后端并发与竞争问题入门

**后端并发的第一性问题，不是“怎么让代码看起来同时运行”，而是“当很多请求同时到来、又大量时间都花在等待时，系统如何不浪费资源，并且不把共享状态写乱”。**

上一篇我们从 `AI Agent Demo` 讲到了大规模服务：一个本地能跑的 Agent，一旦要服务真实用户，就会面对请求入口、数据库、队列、worker、状态恢复、权限和观测这些后端问题。那里面有一个词反复出现：并发。

并发听起来很底层，也很容易让人联想到操作系统、线程、锁、`epoll`、`CPU cache` 之类的名词。但如果从零开始理解，它其实很贴近生活。我们可以继续沿用一个简单比喻：**后端服务器像一家餐厅，并发请求就像同时进店的顾客。**

如果餐厅一次只接待一个顾客，那么第一个顾客点完菜、等后厨做完、吃完、结账以后，第二个顾客才能进来。这样当然最简单，也不容易出错，但效率会非常低。真实餐厅不会这么做，因为大部分时间并不是服务员真的在忙，而是在等后厨、等支付、等客人决定。真实后端也一样：很多请求并不是一直在消耗 CPU，而是在等数据库、等磁盘、等网络、等外部 API，或者等一次 LLM 调用返回。

所以后端并发最重要的直觉是：

```text
请求 A 到来
  -> 需要查数据库，开始等待
  -> 服务器先去处理请求 B
  -> 请求 B 又需要等外部 LLM
  -> 服务器继续处理请求 C
  -> 数据库返回 A 的结果，再回来收尾
```

这不是在“凭空变快”，而是在把原本浪费在等待上的时间利用起来。

但是，只要很多请求真的开始交错执行，另一个问题也会出现：大家可能会同时读写同一份东西。比如同时修改同一个任务状态、同时增加同一个 token 计数器、同时扣同一个账户余额、同时更新同一个 Agent 会话。于是，并发不仅是性能问题，也会变成正确性问题。

这篇文章就从最基础的概念开始，回答四个问题：

- `进程`、`线程`、`协程`、`异步`到底是什么？
- 为什么后端系统最怕的常常不是计算，而是等待？
- `Go`、`Java / Spring`、`Python / FastAPI`、`Node.js` 分别怎么处理并发？
- 为什么并发会产生竞争，锁、原子操作、事务和幂等又分别在解决什么？

## 先分清三组基础概念

在进入进程、线程、协程之前，先要拆开三组经常被混在一起的概念。很多初学者之所以觉得并发难，是因为这些词看起来都在描述“同时做很多事”，但它们其实不是同一个层面的东西。

### 并发不等于并行

`并发` 关注的是**同时处理多个任务的能力**。

`并行` 关注的是**多个任务在同一时刻真的一起运行**。

如果一个服务员在 1 号桌点完菜后，趁后厨做菜时去服务 2 号桌，再去服务 3 号桌，这叫并发。因为他在一段时间内处理了多个顾客，但某一个瞬间可能仍然只在跟一个顾客说话。

如果餐厅里有多个服务员，同时分别服务不同桌，这更接近并行。因为在同一时刻，多个动作真的同时发生。

放到计算机里：

- 单核 CPU 也可以并发，因为它可以快速切换任务。
- 多核 CPU 才能更自然地并行，因为多个核心可以真的同时运行不同任务。
- 后端系统通常同时需要并发和并行：既要在等待 I/O 时切走，也要在多核机器上利用多个核心。

所以，并发不是“CPU 一定同时在跑很多代码”，而是系统调度任务的能力。

### 阻塞不等于慢，非阻塞也不等于快

`阻塞` 的意思是：当前任务遇到某个操作后，必须停在这里等结果。

比如：

- 查数据库，等数据库返回。
- 读文件，等磁盘返回。
- 调 LLM API，等远端模型返回。

如果一个线程在等待期间什么也不能做，这个线程就被阻塞住了。

`非阻塞` 的意思是：当前任务发起操作后，不傻等，可以先去做别的事，等结果准备好以后再回来处理。

但要注意：非阻塞不等于结果更快回来。数据库该查多久还是查多久，LLM 该生成多久还是生成多久。非阻塞真正节省的是**等待期间被闲置的执行资源**。

这就是为什么后端常常强调异步、事件驱动、协程和连接池。它们的核心价值并不是让数据库本身更快，而是让服务器不要在等待数据库时浪费线程、内存和调度成本。

### CPU 密集和 I/O 密集是两种不同问题

`CPU 密集型任务` 是真的需要 CPU 大量计算。

比如：

- 视频编码
- 大规模压缩
- 复杂数学计算
- 大批量数据转换
- 本地模型推理

这种任务的瓶颈通常是 CPU 算力。你让它异步，也不会神奇变快，因为 CPU 还是要算那么多步。

`I/O 密集型任务` 是大部分时间都在等待外部输入输出。

比如：

- 查询数据库
- 请求另一个 HTTP 服务
- 读写文件
- 调用 Redis
- 调用 LLM API

Web 后端最常见的是 I/O 密集型任务。一个普通接口也许只做了很少的计算，但它可能要等数据库 20ms、等缓存 2ms、等另一个服务 100ms、等 LLM 几秒甚至几十秒。并发模型的价值，主要就在这些等待里。

这也是为什么 Agent 服务天然会放大并发问题。因为 Agent 很可能一边维护任务状态，一边调用模型，一边调工具，一边等待外部系统返回。它不是一个短促的函数调用，而是一串会等待、会重试、会恢复、会读写状态的过程。

## 四种“同时接客”的方式：进程、线程、协程与异步

理解了等待之后，就可以看后端到底用什么方式处理很多请求。最常见的四个词是：`进程`、`线程`、`协程`、`异步事件循环`。

它们的区别可以先看这张表：

| 模型 | 谁在调度 | 和操作系统的关系 | 开销 | 适合场景 | 主要问题 |
| --- | --- | --- | --- | --- | --- |
| 多进程 | 操作系统 | 每个进程有相对独立的内存空间 | 最大 | 隔离性强、崩溃互不影响、CPU 并行 | 内存占用高，进程间通信成本高 |
| 多线程 | 操作系统 | 一个进程内多个线程共享内存 | 中等 | 传统 Web 服务、多核并行、阻塞 I/O | 线程多了上下文切换重，共享内存容易竞争 |
| 协程 / 轻量线程 | 语言运行时为主 | 运行时把大量轻量任务映射到少量系统线程 | 较小 | 高并发 I/O、很多等待型任务 | 需要理解运行时调度、取消、泄漏和共享状态 |
| 异步事件循环 | 语言运行时 / 事件循环 | 通常由少量线程监听 I/O 就绪事件 | 很小 | 网络 I/O、多连接、高等待任务 | 不能阻塞事件循环，代码会被 `await` 切分 |

这张表先给出整体图景，下面再逐个拆开。

### 多进程：多开几家店

`进程` 可以粗略理解成操作系统眼里的一个独立程序实例。每个进程有自己的地址空间、文件描述符、资源表和运行状态。一个进程崩了，通常不会直接把另一个进程的内存写坏。

多进程的优势是隔离强。比如 Web 服务可以启动多个 worker 进程，一个进程挂掉，主进程可以重新拉起它。很多服务器和运行时都会使用多进程来利用多核 CPU 或隔离故障。

但多进程的缺点也明显：

- 每个进程都有自己的内存空间，开销更大。
- 进程之间不能像普通变量一样直接共享数据。
- 要通信通常需要管道、socket、共享内存、消息队列等机制。

如果用餐厅类比，多进程像是多开几家分店。每家店都有自己的厨房、桌椅、账本和员工。它很稳，但不轻。

### 多线程：一家店里多个服务员

`线程` 是进程里的执行单元。一个进程可以有多个线程，它们共享同一份进程内存，但各自有自己的调用栈和执行位置。

多线程比多进程轻，因为线程之间共享内存，创建和通信成本通常更低。传统 Web 后端很长时间都采用“一个请求交给一个线程处理”的模型。请求进来，线程从线程池里取出来，处理数据库、业务逻辑、模板渲染或 API 调用，处理完再把线程还回池子。

这种模型的好处是非常直观：

```text
请求 1 -> 线程 1
请求 2 -> 线程 2
请求 3 -> 线程 3
```

代码也通常可以写成同步风格：

```text
读取请求
查询数据库
调用服务
生成响应
```

但线程不是免费的。每个线程需要栈空间，线程之间切换也需要操作系统保存和恢复上下文。当并发请求非常多时，如果每个请求都占住一个线程，那么大量线程可能都在等待 I/O，真正做计算的很少，但内存和调度成本已经上来了。

更重要的是，线程共享同一份内存，这就带来了竞争问题。两个线程同时改同一个变量、同一个列表、同一个缓存对象，如果没有保护，就可能把状态写坏。

### 协程：更轻的执行单元

`协程` 可以先理解成由语言或运行时管理的轻量执行单元。它比操作系统线程更轻，创建和切换成本更低。程序员可以写出接近同步风格的代码，但运行时可以在等待 I/O、通道通信、定时器或其他挂起点把任务切走。

Go 的 `goroutine` 是最典型的例子。Go 官方文档把 goroutine 描述成运行在同一地址空间里的轻量执行线程，并鼓励用 channel 进行通信。实际运行时会把很多 goroutine 调度到较少的操作系统线程上运行。

Java 21 之后的虚拟线程也在解决类似问题：传统平台线程数量太多会变贵，而虚拟线程是 JVM 管理的轻量线程。它仍然使用 Java 的 `Thread` 编程模型，但目标是让“一个请求一个线程”的写法在大量阻塞 I/O 场景下更便宜。

协程的直觉是：不要真的雇十万个重量级服务员，而是让运行时管理很多更轻的“小服务员”。它们遇到等待就挂起，等结果回来再继续。

但协程不是银弹。它仍然会遇到：

- 共享状态竞争
- goroutine / virtual thread 泄漏
- 取消信号没有传递
- 下游资源被打爆
- 阻塞调用破坏调度效果

所以协程解决的是“执行单元太重”的问题，不自动解决“状态怎么正确共享”的问题。

### 异步事件循环：一个人盯很多等待

`异步事件循环` 的核心是：不要为每个等待中的请求都占住一个线程，而是用事件循环统一管理大量 I/O 事件。

它大概像这样工作：

```text
事件循环：
  1. 看看哪些网络连接有新数据
  2. 执行对应回调或恢复对应协程
  3. 遇到 await / I/O 等待就把任务挂起
  4. 继续处理下一个就绪事件
```

Node.js 是这个模型最典型的代表。JavaScript 代码主要运行在事件循环上，网络 I/O 通过底层机制和 `libuv` 协作。Python 的 `asyncio` 也提供了事件循环、协程、任务和 Future，用 `async / await` 描述“这里会等待，等待期间可以让出控制权”。

异步事件循环非常适合网络 I/O，因为一个线程可以管理很多连接。但它有一个很重要的限制：**不要阻塞事件循环。**

如果你在事件循环里做了很重的 CPU 计算，或者调用了真正阻塞的同步函数，那么事件循环就没法继续处理其他请求。餐厅里那个超级服务员被你拉去搬半小时货，所有顾客都会卡住。

所以需要记住两句话：

- 协程不是异步的同义词。协程是一种执行单元，异步是一种等待和调度方式。
- 异步也不是多线程的同义词。很多异步程序主要靠少量线程处理大量 I/O，但也可以配合线程池或 worker 处理阻塞任务。

## 主流后端系统是怎么用这些模型的

知道模型以后，再看主流后端框架就会清楚很多。它们不是谁更高级、谁更落后，而是在不同语言传统、运行时能力和应用场景里做了不同取舍。

### Go：goroutine、channel 与 context

Go 的并发模型围绕 `goroutine` 展开。你可以用 `go f()` 启动一个 goroutine，它比操作系统线程轻很多，Go 运行时负责调度它们。Go 的 HTTP 服务通常会为请求创建 goroutine，所以写起来非常直接。

Go 还提供了两个很重要的配套工具：

- `channel`：用通信传递数据，让某个 goroutine 成为状态的唯一拥有者。
- `context`：传递取消、超时和请求范围内的元数据。

Go 的优势是高并发 I/O 写起来比较自然。比如一个 Agent worker 需要同时调用多个工具，可以启动多个 goroutine 并等待结果；如果用户取消任务，可以通过 `context` 把取消信号传下去。

但 Go 的坑也很典型：

- goroutine 太容易创建，忘记退出就会泄漏。
- channel 用不好会死锁。
- 多个 goroutine 共享 map、slice 或业务对象时，仍然需要锁、channel ownership 或其他同步机制。
- `context` 不传递，取消和超时就只停留在入口层。

所以 Go 的关键不是“随便开 goroutine”，而是要让 goroutine 的生命周期、取消路径和共享状态边界都清楚。

### Java / Spring：传统线程池仍是主流，虚拟线程正在进入生产选项

Java 后端长期采用线程池模型。以传统 Spring MVC / Servlet 应用为例，一个请求通常会被分配给容器线程池里的某个线程。这个线程同步执行控制器、服务层、数据库访问等逻辑，最后返回响应。

这个模型的优点是成熟、稳定、生态强大。企业系统里大量数据库驱动、事务管理、连接池、监控工具和调试工具都围绕它建立起来。

它的问题也很明显：如果每个请求都占住一个平台线程，而请求又大量等待数据库或远程服务，那么线程数上去以后，内存和上下文切换成本会变高。线程池可以限制资源，但线程池满了以后，请求就会排队。

Java 21 引入的虚拟线程让这个模型有了新选项。虚拟线程仍然表现为 `Thread`，但由 JVM 管理，创建和阻塞成本远低于传统平台线程。Spring Boot 官方文档也提供了启用虚拟线程的配置选项。

因此更准确的说法是：

- 传统线程池仍然是 Java / Spring 生态里的重要主流。
- Java 21+ 和 Spring Boot 已经提供虚拟线程选项，适合大量阻塞 I/O 的场景。
- 虚拟线程降低了“一个请求一个线程”的成本，但不消灭数据库连接池、锁竞争、事务冲突和下游限流这些问题。

如果你把一个同步 Java 服务直接切到虚拟线程，却不限制数据库连接数、不控制外部 API 并发、不处理共享状态，那么只是把入口层变轻了，下游仍然可能被打爆。

### Python / FastAPI：asyncio 与线程池边界

Python 的异步生态主要围绕 `asyncio` 和 `async / await`。FastAPI 则建立在 Starlette / AnyIO 之上，鼓励在 I/O 密集接口里使用 `async def`，这样请求在等待数据库、HTTP 调用或其他异步 I/O 时可以让出事件循环。

FastAPI 还有一个对初学者很重要的设计：如果你写的是普通 `def` 路由函数，而不是 `async def`，FastAPI 会把它放到外部线程池里运行并等待结果。这样可以避免一个阻塞函数直接卡住事件循环。

这背后的直觉是：

- `async def` 适合你能 `await` 的异步 I/O。
- 普通阻塞函数应该被隔离，不要直接堵住事件循环。
- 真正重 CPU 的任务应该交给进程池、专门 worker，或者单独服务，而不是塞进 Web 事件循环。

Python 的坑也很常见：

- 在 `async def` 里调用阻塞库，会卡住事件循环。
- 以为有 GIL 就没有竞争问题，但业务层面的并发冲突仍然存在。
- 多请求同时更新数据库状态时，问题不在 Python 变量，而在数据库记录和事务边界。
- LLM 调用耗时很长，如果直接让 HTTP 请求一直挂着，入口层会被长任务拖住。

所以 FastAPI 很适合做 AI / Agent 服务的 API 层，但长任务、重计算、批处理和可靠重试通常仍然需要队列、worker 和持久化状态配合。

### Node.js：事件循环擅长 I/O，但不要做重 CPU

Node.js 的核心优势是事件循环和非阻塞 I/O。JavaScript 代码主要在单线程事件循环上执行，请求发起网络 I/O 后不需要一直占住一个线程等待结果。对大量网络请求、代理转发、实时连接和 API 聚合场景来说，这个模型非常自然。

这也是为什么 Node.js 很适合很多 Web API、BFF、网关和实时应用。前端工程师转全栈时，也很容易用同一门语言写前后端。

但 Node.js 的限制同样清楚：

- 如果在主事件循环里做重 CPU 计算，所有请求都会被拖慢。
- 单线程 JavaScript 不代表没有竞争问题，因为多个异步请求可能交错读写同一个外部状态。
- 对数据库、Redis、文件、远程 API 的并发访问仍然需要事务、条件更新、幂等和限流。
- 如果确实要做 CPU 密集任务，应考虑 worker threads、独立 worker 服务或把任务交给更适合的运行环境。

换句话说，Node.js 避免了很多传统线程共享内存的复杂性，但没有让后端正确性问题消失。只要多个请求会读写同一份业务状态，竞争就还在。

### 小结：框架选择背后是同一个取舍

可以把四类主流后端放到同一张图里：

| 技术栈 | 常见并发方式 | 擅长什么 | 主要提醒 |
| --- | --- | --- | --- |
| Go | goroutine + channel + context | 高并发 I/O、微服务、worker | 管住 goroutine 生命周期和共享状态 |
| Java / Spring | 线程池；Java 21+ 可用虚拟线程 | 企业系统、事务、生态、稳定性 | 线程变轻不等于下游无限，仍要控连接池和事务 |
| Python / FastAPI | asyncio + async/await；阻塞函数进线程池 | AI API、快速开发、I/O 接口 | 不要在事件循环里调用阻塞库 |
| Node.js | 事件循环 + libuv；必要时 worker | 网络 I/O、BFF、实时连接 | 不要阻塞事件循环，业务状态仍要并发控制 |

它们解决的都是同一个问题：**当请求很多、等待很多时，怎样用尽量少的资源保持吞吐。**

但接下来要讲的，是另一个同样重要的问题：**当这些请求真的同时进行时，怎样保证结果不乱。**

## 并发为什么会产生竞争问题

先看一个最经典的例子：统计所有 Agent 调用 LLM 消耗的 token 总数。

你可能会写出这样的伪代码：

```text
total_tokens = total_tokens + tokens_used
```

这看起来只有一行，但在计算机眼里，它至少包含三步：

```text
1. Read：从内存或数据库里读出 total_tokens
2. Modify：把它加上 tokens_used
3. Write：把新值写回去
```

如果只有一个请求，这没有问题。

如果两个请求同时执行，就可能变成这样：

```text
当前 total_tokens = 100

请求 A 读取 total_tokens，得到 100
请求 B 读取 total_tokens，得到 100
请求 B 计算 100 + 20，并写回 120
请求 A 计算 100 + 10，并写回 110

最终结果：110
正确结果：130
```

请求 A 和请求 B 都没有“报错”，但结果已经错了。这个问题叫 `竞态条件`：程序结果取决于多个并发执行之间不可控的先后顺序。

注意，这里真正危险的是“读-改-写”之间被别人插队。它不一定只发生在线程里，也可能发生在数据库、缓存、文件或远程服务里。

比如：

- 两个 worker 同时把同一个任务从 `running` 改成 `finished`。
- 用户连续点了两次“购买”，余额被重复扣除。
- 两个请求同时更新同一个会话摘要，后写入的覆盖了先写入的内容。
- 队列重投了一条消息，worker 又执行了一遍已经成功的工具调用。
- 两个异步函数都读取旧状态，`await` 之后再写回，导致一个结果覆盖另一个结果。

所以，“单线程 async”也不等于绝对安全。单线程意味着同一时刻只有一段 JavaScript 或 Python 代码在执行，但多个请求仍然会在 I/O 和 `await` 之间交错。更重要的是，后端真正共享的往往不是进程内变量，而是数据库记录、Redis key、对象存储文件、用户余额、任务状态和外部副作用。

并发正确性要回答的不是“有没有两个 CPU 核同时写内存”，而是：

- 同一份状态谁能改？
- 两次修改谁先谁后？
- 中间失败了怎么办？
- 重试一次会不会重复生效？
- 扩容到多进程、多机器后，规则还成立吗？

## 解决竞争：不要只想到 Lock

一说并发安全，很多人第一反应是“加锁”。锁当然重要，但它不是唯一方案，也不是永远最好的方案。更好的学习顺序是：先减少共享，再保护共享，最后把跨请求、跨进程、跨机器的正确性交给数据库和外部系统。

### 第一招：尽量不要共享可变状态

最好的并发 bug，是从设计上让它没有机会发生。

如果一个请求处理过程中用到的数据只属于这个请求，那么它就不需要和别人抢。比如：

- 请求参数放在请求自己的上下文里。
- 中间计算结果用局部变量保存。
- Web 服务尽量无状态，真实状态放到数据库或队列。
- 大对象写入对象存储后只传引用，不到处传可变对象。

Go 里常说“不要通过共享内存来通信，而要通过通信来共享内存”，本质也是这个思路。你可以让一个 goroutine 成为某份状态的唯一拥有者，其他 goroutine 只通过 channel 发消息给它。这样状态修改就被集中到一个地方，竞争会少很多。

这类似于餐厅里不要让所有服务员都去改同一本账，而是让他们把小票交给收银员，由收银员按顺序入账。

### 第二招：需要共享时，用锁保护临界区

如果确实有一份内存状态需要多个线程或 goroutine 共同访问，就要用同步机制保护它。最常见的是互斥锁。

互斥锁的意思是：同一时间只能有一个执行单元进入这段关键代码。

```text
lock()
  total_tokens = total_tokens + tokens_used
unlock()
```

这样请求 A 进来时，别人不能同时修改 `total_tokens`。等 A 改完释放锁，请求 B 才能进来。

锁解决的是正确性问题，但也会带来代价：

- 被锁保护的代码会变成串行执行。
- 锁范围太大，会让吞吐下降。
- 多把锁顺序不一致，可能死锁。
- 高竞争场景下，线程等待和唤醒也有调度成本。

所以锁的使用原则不是“能锁就锁”，而是：

- 锁的范围越小越好。
- 锁里不要做慢 I/O。
- 锁保护的状态边界要清楚。
- 能用更高层的数据库事务或原子命令时，不要手写复杂锁。

除了互斥锁，还有读写锁和信号量。读写锁适合“读很多、写较少”的共享数据；信号量适合限制并发数量，比如最多同时调用 10 个外部工具，避免把下游 API 打爆。

### 第三招：简单计数器可以用原子操作

如果共享状态只是一个简单计数器、标志位或引用替换，原子操作通常比锁更轻。

原子操作的核心是 CPU 或运行时提供一种“不可被打断”的更新方式。常见实现是 `CAS`：Compare-And-Swap。

它的逻辑大概是：

```text
我读到旧值是 100
我想把它改成 101
写回前检查：现在还是 100 吗？
  是：写入 101
  否：说明别人改过了，重新读，再试一次
```

这被称为乐观并发控制。它假设冲突不一定发生，所以先尝试；如果真的冲突，再重试。

原子操作适合：

- 计数器
- 状态标志
- 简单统计
- 无锁数据结构里的底层步骤

但它不适合复杂业务不变量。比如“检查余额是否足够、扣款、写订单、发道具”这不是一个简单整数加一，不能靠一个 `AtomicInteger` 解决。它需要事务、唯一约束、幂等和业务状态机。

### 第四招：用事务、条件更新和唯一约束保护业务状态

后端真正重要的状态通常不在进程内存里，而在数据库里。

比如：

- 用户余额
- 订单状态
- 任务执行状态
- token 账单
- 工具调用记录
- 租户配额

这些状态必须在多个请求、多进程、多 worker 下都正确。此时单机内存锁已经不够，真正可靠的边界通常是数据库事务和约束。

一个简单计数可以写成：

```sql
UPDATE usage_stats
SET total_tokens = total_tokens + 10
WHERE user_id = 1;
```

这比“先 SELECT 出来，在应用内加 10，再 UPDATE 回去”更安全，因为数据库可以把这次更新作为一个原子操作处理。

更复杂的情况可以使用：

- 事务：保证一组修改要么一起成功，要么一起失败。
- 条件更新：只有状态仍然符合预期时才更新，比如 `WHERE status = 'running'`。
- 版本号：更新时检查 `version`，发现被别人改过就重试或报错。
- 唯一约束：防止同一个幂等键插入两次。
- 幂等键：让重复请求或重复消息不会重复生效。

比如一个 Agent 任务从 `running` 变成 `finished`，可以要求只有当前状态确实是 `running` 时才能更新：

```sql
UPDATE tasks
SET status = 'finished', version = version + 1
WHERE id = ? AND status = 'running' AND version = ?;
```

如果影响行数是 0，就说明状态已经被别人改过了，当前 worker 不能再假装自己成功。

这类机制是后端正确性的核心。锁保护的是“这段代码现在只有我能进”，事务和条件更新保护的是“这份业务状态只能按规则变化”。

### 第五招：用幂等面对重试和重复执行

并发系统里还有一个很现实的问题：同一件事可能被执行多次。

原因可能是：

- 用户重复点击。
- HTTP 请求超时后客户端重试。
- 队列采用至少一次投递，同一消息可能再次出现。
- worker 执行成功了，但回写状态时失败。
- 调用外部服务时，网络断了，你不知道对方到底做没做。

这时只靠锁不够，因为问题已经不是“同时进来”，而是“同一意图重复进来”。解决方式是幂等。

幂等的意思是：同一个操作执行一次和执行多次，最终效果应该一样。

常见做法是：

- 每个外部请求带一个 `idempotency_key`。
- 数据库用唯一约束保存这个 key。
- 第一次执行成功后记录结果。
- 后续相同 key 的请求直接返回已有结果，不再重复扣费或重复执行副作用。

对 Agent 服务来说，幂等尤其重要。因为工具调用、计费、消息发送、任务状态推进都可能遇到重试。如果没有幂等，系统会把“提高可靠性”的重试机制变成“重复制造副作用”的事故机制。

## 多实例以后：单机锁管不到整个系统

到目前为止，我们主要讨论的是单机内的并发。但真实后端服务通常会部署多个实例：

```text
用户请求
  -> 负载均衡
  -> Web 实例 A
  -> Web 实例 B
  -> Web 实例 C
```

每个实例都有自己的内存、自己的锁、自己的线程或协程。实例 A 里的 `mutex` 管不到实例 B，实例 B 里的原子变量也管不到实例 C。

这就是为什么分布式系统里，很多正确性问题要下沉到共享基础设施：

- 简单全局计数可以用 Redis 的 `INCR` 这类原子命令。
- 核心业务状态通常放到数据库，用事务、条件更新和唯一约束保护。
- 复杂跨实例互斥有时会用分布式锁，但必须处理租约、超时、误释放和 fencing token 等问题，不能把它当成本地锁的简单放大版。
- 队列和 worker 要假设消息可能重复，所以必须配合幂等消费和去重记录。

这也是为什么本系列后面会继续讲数据库、事务、幂等、缓存、队列和工作流。并发不是只在语言运行时里发生，它最终会落到系统边界上。

如果把单机并发和分布式并发放在一起看，可以得到一个很实用的判断：

| 问题 | 常见手段 | 适用边界 |
| --- | --- | --- |
| 进程内共享变量 | mutex、rwlock、atomic、channel ownership | 单进程内有效 |
| 同一服务多线程 / 多协程 | 锁、信号量、原子操作、取消传播 | 单实例内有效 |
| 多实例共享业务状态 | 数据库事务、条件更新、唯一约束、幂等键 | 跨进程 / 跨机器有效 |
| 全局计数或限流 | Redis 原子命令、令牌桶、集中式限流器 | 依赖外部共享组件 |
| 重复消息和重试 | 幂等消费、去重表、任务状态机 | 队列 / worker 系统有效 |

所以不要问“我是不是该加锁”，而要先问：**我要保护的状态在哪里？竞争会发生在什么边界？**

如果状态只在一个进程内，语言级锁可能够用。

如果状态在数据库里，就应该用数据库的事务和约束。

如果状态跨多个服务和外部副作用，就必须把幂等、重试、状态机和审计一起设计进去。

## 从零开始的学习路线

如果你是第一次系统学习后端并发，可以按下面这条路线走，不需要一开始就钻到操作系统细节里。

第一步，先把“等待”看懂。

你要能判断一个接口主要是在算，还是在等。如果它主要在等数据库、Redis、外部 API 或 LLM，那么并发模型、连接池、超时和取消就会比单纯优化算法更重要。

第二步，把执行单元看懂。

你不需要立刻掌握所有底层实现，但要能区分：

- 进程负责隔离。
- 线程负责传统并行和同步执行。
- 协程 / 轻量线程降低大量等待任务的调度成本。
- 异步事件循环用少量线程管理大量 I/O。

第三步，把共享状态看懂。

遇到并发问题时，先问：

- 哪份状态被共享了？
- 谁会读它？
- 谁会写它？
- 写入是不是读-改-写？
- 这份状态是在内存、数据库、Redis，还是外部系统里？

第四步，再选择工具。

不要背答案，而要按边界选：

- 没必要共享，就不要共享。
- 单进程共享变量，用锁、原子操作或 channel ownership。
- 控制并发数量，用信号量、连接池和限流。
- 数据库状态，用事务、条件更新、版本号和唯一约束。
- 重试和重复请求，用幂等键和去重记录。
- 多实例全局状态，用数据库、Redis、队列等共享基础设施。

第五步，最后再学更底层的细节。

比如 `epoll`、调度器、JVM、GIL、CPU cache、内存模型、无锁结构，这些都很重要，但它们不是第一天就必须掌握的内容。对后端初学者来说，先建立“等待、调度、共享状态、竞争边界”的直觉更重要。

## 下一篇：数据库为什么是并发正确性的中枢

到这里，我们已经知道后端并发有两面：

- 一面是吞吐：请求很多、等待很多时，怎么不浪费资源。
- 一面是正确性：请求交错、重试、重复、扩容时，怎么不把状态写乱。

进程、线程、协程和异步事件循环，主要在解决第一面。它们让系统能用更合理的资源处理更多等待型请求。

锁、原子操作、事务、条件更新和幂等，主要在解决第二面。它们让系统在多请求、多 worker、多实例下仍然保持业务状态正确。

下一篇我们会把重点放到数据库上：为什么 Agent 服务不能只靠内存保存状态，为什么任务状态、用户余额、计费记录、工具调用和幂等键都应该被持久化，以及数据库如何通过事务、索引、约束和并发控制，成为后端系统里最重要的正确性中枢。

也就是说，学并发不是为了炫耀底层知识，而是为了回答一个很现实的问题：

**当很多人同时使用你的服务时，它既要快，也不能错。**
