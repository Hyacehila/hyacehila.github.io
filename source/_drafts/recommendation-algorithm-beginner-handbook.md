# 推荐算法入门手册：从数据分析到召回、排序与实验

> 来源文章：王诗佳《[推荐算法新人简易入门指南](https://km.netease.com/v4/detail/blog/245777)》，发布于 2025 年 6 月 24 日，更新于 2025 年 6 月 25 日。
>
> 原文标注为网易内部分享材料，不允许以任何形式外泄。本草稿仅用于内部学习与二次整理；若要公开发布，需要重新核对引用、链接、版权与信息公开范围。

## 原文内容总结

原文的核心判断是：**推荐算法工程师不是只负责训练模型，而是要对数据分析、策略与工程开发、算法模型以及最终业务指标形成完整闭环。** 新人如果只读论文或只会调用深度学习框架，很难独立完成一次推荐策略迭代。

文章围绕三类核心能力组织学习材料：

1. **数据分析能力**：用 SQL 查询和清洗用户行为数据，理解常见互联网指标，并能设计、分析 A/B 实验。
2. **推荐策略与工程能力**：掌握基础开发知识，理解召回、排序、重排和混合策略，能够把离线想法接入线上推荐链路。
3. **推荐模型能力**：从逻辑回归、FM、GBDT 等传统模型，逐步学习 Wide&Deep、DeepFM、DIN、BST、双塔模型等工业界常见架构，再根据实际需要关注对比学习、生成式推荐、LLM 多模态推荐和强化学习等前沿方向。

原文还根据不同新人角色给出了阅读侧重点：短期实习生优先学习模型并尽快验证想法；无法参与线上开发的长期实习生应补齐数据分析和模型能力；正式员工则需要建立覆盖数据、工程、策略和模型的完整技能树。贯穿全文的学习建议是：**每学一个知识点，都配一个可验证的小实践；既读论文，也关注工业界如何解决数据、延迟、稳定性和指标权衡问题。**

下面的手册保留原文的三大能力框架，但不再按“资料清单”展开，而是按新人完成一次推荐迭代的真实路径重新组织。其中，最小项目、八周路线和常见误区是基于原文框架补充的实践建议。

## 一、先建立正确的系统视角

推荐系统的目标不是给每个用户算出一个分数，而是在有限算力、延迟和内容供给下，为一次请求生成一组相对合适的结果。一个常见的在线链路可以抽象为：

```text
用户请求
  → 候选召回
  → 规则过滤
  → 粗排/精排
  → 多目标融合
  → 重排
  → 结果展示
  → 行为日志
  → 指标分析与 A/B 实验
  → 下一轮迭代
```

其中：

- **召回**解决“从海量物品中快速找出几百到几千个候选”的问题，优先保证覆盖率和相关性。
- **排序**解决“在候选集中估计用户对每个物品的偏好”的问题，通常会预测点击、播放、转化、时长等一个或多个目标。
- **重排**解决“逐条高分不等于整页体验好”的问题，需要处理多样性、重复、频控、内容安全和业务规则。
- **实验与分析**决定一个方案能否上线。离线指标只能说明模型在历史数据上的表现，最终仍要结合线上指标、护栏指标和业务价值判断。

新人最重要的认知转变是：**模型只是推荐系统中的一个部件。** 一个更复杂的模型，如果数据口径错误、召回漏掉关键候选、线上延迟超标或实验设计有偏差，仍然不会带来可信的收益。

## 二、数据分析是定位问题的起点

### 2.1 先学会读懂行为数据

推荐系统常见数据可以分成三类：

| 数据类型 | 常见字段 | 主要用途 |
| --- | --- | --- |
| 用户数据 | 用户 ID、地区、设备、会员状态、长期兴趣 | 用户画像、分群、冷启动特征 |
| 物品数据 | 物品 ID、类目、作者、发布时间、文本或多模态特征 | 内容理解、过滤、相似度计算 |
| 行为数据 | 曝光、点击、播放、停留、收藏、购买、时间戳 | 样本构造、指标分析、序列建模 |

学习 SQL 时，不要停在语法记忆。建议按推荐问题组织练习：

1. 用 `SELECT`、`WHERE` 和聚合函数统计曝光、点击和用户数。
2. 用 `JOIN` 关联用户、物品与行为表，检查数据是否存在一对多放大。
3. 用 `GROUP BY` 分析不同人群、内容类型和流量入口的指标差异。
4. 用窗口函数构造用户行为序列、计算分位数、留存和排名。
5. 检查重复日志、延迟到达、空值、异常值和口径变化。

一次合格的数据分析不只输出“指标下降了”，还应回答：下降发生在哪类用户、哪类内容、哪个流量入口和哪个时间段；变化来自曝光结构、用户行为还是数据采集；结论是否受到样本量、节假日或版本发布影响。

### 2.2 用指标树代替单指标思维

原文列出了 PV、UV、DAU、MAU、CTR、CVR、留存率、LTV、CAC、GMV、完播率和互动率等常见指标。实际使用时，应把它们组织成指标树，而不是平铺成术语表。

以内容推荐为例，可以将指标分为四层：

- **规模指标**：DAU、请求量、曝光用户数，判断实验覆盖范围是否正常。
- **消费指标**：CTR、播放率、有效播放率、人均时长，描述用户是否愿意消费内容。
- **长期体验指标**：留存、负反馈率、内容多样性、作者覆盖，防止短期点击收益损伤长期体验。
- **商业指标**：CVR、GMV、广告收入、会员转化，评估推荐对业务结果的贡献。

不要孤立优化 CTR。标题党内容可能提高点击，却降低有效播放、满意度或长期留存。推荐优化通常是多目标问题，需要明确：哪个是主指标，哪些是护栏指标，哪些只用于解释结果。

### 2.3 A/B 实验要回答因果问题

A/B 实验的基本流程包括随机分流、实验设计、样本量与时长估计、指标监控、显著性分析和业务决策。新人至少应能检查以下问题：

- 实验组和对照组是否使用稳定且互斥的分流方式。
- 实验期间是否存在版本发布、活动、节假日或流量结构变化。
- 核心指标、护栏指标和反向指标是否在实验前定义。
- 是否区分统计显著性与业务价值，避免把极小但显著的变化直接等同于值得上线。
- 是否存在多重检验、样本污染、提前偷看结果或实验单元不独立等问题。

原文建议实验覆盖完整的工作日与周末周期。具体时长不应机械固定，还需要根据流量、指标波动、最小可检测效应和业务周期计算。

## 三、理解召回、排序和重排的分工

### 3.1 召回先解决覆盖问题

当物品库很大时，不可能让复杂排序模型逐一评估全部物品。召回层使用计算成本较低的方法生成候选集，常见路径包括：

| 召回方式 | 核心思路 | 适用场景 | 主要局限 |
| --- | --- | --- | --- |
| 热度召回 | 推荐全局或分群热门物品 | 新用户、系统兜底、热点场景 | 个性化弱，容易强化头部效应 |
| 规则或 LBS 召回 | 按地区、时间、类目或业务规则取候选 | 本地生活、时效内容、明确约束 | 依赖规则维护，泛化能力有限 |
| UserCF | 找相似用户喜欢的物品 | 用户行为较丰富、兴趣群体明显 | 大规模计算和用户冷启动较难 |
| ItemCF | 找与用户历史物品相似的物品 | 行为共现稳定、可解释性要求较高 | 对新物品不友好，可能越推越窄 |
| Embedding 召回 | 将用户或物品映射为向量后做近邻检索 | 需要语义相似或泛化能力 | 依赖样本构造和向量检索基础设施 |
| 双塔召回 | 分别编码用户和物品，在同一向量空间检索 | 大规模个性化召回 | 交互表达能力受双塔结构限制 |

工业系统通常不是选择唯一一种召回，而是并行使用多路召回，再做去重、配额和融合。一个实用的入门组合是：

```text
热门召回 + ItemCF 召回 + Embedding/双塔召回 + 业务兜底
```

分析召回效果时，除了最终业务指标，还要关注召回率、候选覆盖、各路召回贡献、重复率、新物品覆盖和后续排序能否识别优质候选。

### 3.2 排序学习用户对候选的偏好

排序模型通常接收用户特征、物品特征、上下文特征和交叉特征，输出点击、转化、时长或其他目标的预测值。

新人可以按以下顺序建立模型理解：

1. **逻辑回归（LR）**：理解样本、标签、特征工程、损失函数和概率输出，建立完整训练链路。
2. **因子分解机（FM）**：理解稀疏特征的二阶交互，以及 Embedding 如何表达类别特征关系。
3. **GBDT 与 GBDT+LR**：理解树模型如何捕捉非线性关系，以及传统模型组合的工程思路。
4. **Wide&Deep、DeepFM**：理解“记忆已有模式”和“泛化到未见组合”的平衡，以及低阶、高阶特征交互。
5. **DIN、BST 等序列模型**：理解用户兴趣不是固定向量，当前候选物品与历史行为之间的关系会影响预测。

模型学习不要只背结构图。对每个模型至少回答五个问题：输入特征是什么、标签如何构造、模型比基线多表达了什么、离线指标为何可能提升、上线成本和失败条件是什么。

### 3.3 多目标融合决定“优化什么”

真实推荐通常同时关心点击、播放、时长、转化、留存和负反馈。常见做法包括为不同目标分别建模，再通过加法、乘法、分段函数或学习式融合生成最终分数。

多目标融合没有脱离业务的通用最优公式。调整权重时，需要观察目标之间的量纲、分布和相关性，并用 A/B 实验验证。一个目标的预测值更准，也不代表最终融合后的用户体验一定更好。

### 3.4 重排优化整组结果

排序模型通常逐个物品打分，重排则从列表整体出发。常见任务包括：

- 使用最大边际相关性（MMR）、行列式点过程（DPP）或规则配额提高多样性。
- 限制同作者、同类目或相似内容连续出现。
- 处理重复曝光、频次控制、新内容探索和业务保量。
- 执行内容安全、版权、库存、地域和用户屏蔽等硬约束。

重排会牺牲一部分单点预测分数，以换取列表体验、生态健康或长期价值。评估时应同时观察准确性指标和列表级指标。

## 四、工程能力决定方案能否落地

原文把 Java 作为推荐策略开发的重点，原因是许多在线推荐服务使用 Java 生态。但具体语言取决于团队技术栈；新人真正需要掌握的是可迁移的工程能力：

- 使用集合、映射和基本数据结构处理用户、物品与行为数据。
- 理解接口、模块边界、配置管理、异常处理和单元测试。
- 理解并发、线程池、缓存、超时和降级的基本概念。
- 能读取服务日志，定位请求、特征、模型版本和策略配置。
- 了解离线训练、模型导出、在线加载、灰度发布和回滚的大致流程。

模型上线前至少要回答：特征在线是否可取、训练与服务口径是否一致、请求延迟是否满足要求、模型失败后如何降级、版本如何追踪、指标异常时如何回滚。

## 五、用一个最小项目串起全部知识

比起连续阅读多本书，更有效的入门方式是完成一个小而完整的推荐项目。可以使用公开的显式评分或隐式反馈数据集，目标不是做出最强模型，而是走通一次可验证的闭环。

### 5.1 项目目标

为每个用户生成 Top-K 推荐列表：先用热门推荐与 ItemCF 形成候选集，再用一个基础学习模型排序，比较召回、排序和重排各阶段的效果。

### 5.2 实施步骤

1. **数据检查**：统计用户数、物品数、行为数、稀疏度和时间分布，处理重复与异常记录。
2. **划分数据**：优先按时间切分训练集和测试集，避免把未来行为泄漏到训练数据中。
3. **建立基线**：先实现热门推荐，确保数据管道和评估代码可用。
4. **实现 ItemCF**：根据共现行为计算物品相似度，为用户召回相似物品。
5. **训练基础排序模型**：合并并去重多路召回候选，选择 LR、FM 或 DeepFM 完成特征处理、训练、验证和打分。
6. **离线评估**：根据任务选择 Recall@K、Precision@K、NDCG@K、AUC、覆盖率和多样性等指标，并解释每个指标的局限。
7. **模拟重排**：加入类目去重、作者频控或多样性规则，观察准确性与列表体验的变化。
8. **复盘实验**：记录假设、改动、结果、失败原因和下一步，而不是只保存最好的数字。

### 5.3 完成标准

完成项目后，你应能清楚说明：

- 数据如何变成训练样本，负样本如何产生。
- 每路召回能覆盖哪些用户和物品。
- 模型相对基线增加了什么表达能力。
- 离线指标为什么变化，是否可能存在数据泄漏或评估偏差。
- 如果上线，需要增加哪些特征服务、监控、实验和降级能力。

## 六、八周学习路线

这条路线适合有 Python 基础、能阅读简单机器学习代码、希望在两个月内形成完整推荐系统认知的新人。时间有限时，应优先完成每周产出，而不是追求材料全部读完。

| 周次 | 学习重点 | 可验证产出 |
| --- | --- | --- |
| 第 1 周 | SQL、行为数据和指标口径 | 完成一份用户行为分析，解释曝光、点击、转化的关系 |
| 第 2 周 | A/B 实验与统计基础 | 写出一份实验设计，包括主指标、护栏指标、分流和停止条件 |
| 第 3 周 | 推荐系统链路与热门召回 | 画出召回—排序—重排—实验闭环，实现热门基线 |
| 第 4 周 | UserCF、ItemCF、Embedding | 实现至少一种协同过滤，并分析覆盖率和冷启动问题 |
| 第 5 周 | LR、FM 与特征工程 | 训练一个排序基线，完成样本和特征检查 |
| 第 6 周 | Wide&Deep、DeepFM 或 DIN | 复现一个深度模型，并与简单基线公平比较 |
| 第 7 周 | 多目标融合、重排与工程约束 | 加入多样性或频控规则，记录收益与代价 |
| 第 8 周 | 实验复盘与方案表达 | 输出项目报告：问题、假设、方法、结果、限制和上线清单 |

如果你只能参与离线模型工作，可把第 7 周的工程实现替换为“线上链路阅读与接口设计”；如果即将承担正式岗位，应继续补充服务开发、特征平台、向量检索、监控告警和实验平台知识。

## 七、如何阅读论文和前沿工作

原文列出的前沿方向包括对比学习、生成式推荐、基于 LLM 的多模态推荐和强化学习。这些方向值得关注，但不适合作为零基础起点。

阅读一篇论文时，建议按下面的顺序做笔记：

1. 它解决的是召回、排序、重排还是系统问题。
2. 基线方法在什么条件下失败。
3. 新方法增加了什么信息、结构或训练目标。
4. 数据集、负采样、指标和对照实验是否支持论文结论。
5. 计算成本、在线延迟、特征可得性和系统复杂度如何变化。
6. 哪些结论只在特定数据集或业务场景中成立。

优先阅读工业界论文和技术文章，特别关注 SIGIR、WSDM、KDD、RecSys、WWW、CIKM 等会议的 industry track。不过，“来自大公司”不能代替对实验设计和适用边界的判断。

## 八、常见入门误区

### 误区一：模型越新，效果越好

复杂模型只有在数据量、特征、训练流程和线上基础设施匹配时才可能发挥价值。先建立简单且可信的基线，通常比直接复现前沿模型更重要。

### 误区二：离线指标提升就可以上线

离线数据来自历史策略，存在选择偏差；指标也可能与真实业务目标不完全一致。离线结果适合筛选方案，线上实验才用于判断实际影响。

### 误区三：只关注 CTR

CTR 容易计算，但不能完整代表消费质量、长期满意度和商业价值。任何主指标都需要护栏指标和反向指标约束。

### 误区四：只训练模型，不检查数据

标签穿越、重复样本、曝光缺失、特征口径不一致和错误负采样，都可能制造虚假的离线收益。推荐项目应把数据检查放在模型训练之前。

### 误区五：把论文结构直接搬进线上系统

论文通常聚焦某个研究问题，不会完整处理超时、降级、缓存、频控、内容安全和实验冲突。生产方案需要在模型收益与工程成本之间做取舍。

## 九、推荐学习资源

以下资源主要整理自原文，可按当前阶段选择，不必一次读完。

### 数据分析与实验

- 《SQL 必知必会》（第四版）：适合快速建立 SQL 基础。
- LeetCode SQL 题库：适合练习关联、聚合和窗口函数。
- 《Trustworthy Online Controlled Experiments》：系统学习线上受控实验。

### 推荐系统基础

## 结语

推荐算法入门的合理顺序不是“读完所有模型再开始实践”，而是先理解推荐系统闭环，用 SQL 和指标看懂问题，用简单召回与排序建立基线，再逐步增加模型表达能力、列表策略和工程约束。每一步都应留下可验证的产出：一段查询、一张指标表、一个基线模型、一份实验设计或一次失败复盘。

当你能从数据发现问题，提出可以证伪的假设，完成离线验证，说明上线条件，并通过 A/B 实验判断收益时，才真正开始具备推荐算法工程师的完整工作能力。
