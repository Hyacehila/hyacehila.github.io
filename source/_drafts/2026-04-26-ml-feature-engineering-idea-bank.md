这份文档先不追求“像一篇完整文章”，而是作为一个可反复补充的经验草稿库。它要存放的是一些很实战的机器学习判断：

- 聚类得到的类别身份，本质上可以作为新的监督学习特征；
- 对特征重要性施加人为先验，往往比盲目换模型更有效；
- DeepFM 不是一个孤立的深度模型，而是围绕“高维稀疏类别交叉”设计出来的表示学习方法；
- Tree 模型、FM、DeepFM、目标编码、GBDT leaf embedding 等方法，很多时候都可以放在同一个问题下理解：**如何把原始字段转化成更容易被模型学习的结构化信号**。

如果要把这些想法整理成正式博客，不建议写成“我学到了三个技巧”。更好的中心问题是：

> 在结构化机器学习任务里，模型为什么常常不是缺少算法，而是缺少合适的特征表示？我们如何把群体身份、业务先验和稀疏交互转化成可学习信号？

这个问题足够集中，也能自然容纳聚类特征、特征加权、DeepFM 与树模型之间的联系。

## 可整理成的正式选题

### 选题一：把先验变成特征

题目可以是：

> 从特征工程到归纳偏置：聚类特征、特征加权与 DeepFM 的共同逻辑

核心观点：

很多机器学习技巧并不是孤立技巧，而是在向模型注入某种先验：

- 聚类特征注入的是“样本属于哪个潜在群体”的空间先验；
- 特征加权注入的是“哪些字段更值得模型优先关注”的业务先验；
- FM/DeepFM 注入的是“稀疏类别之间可以通过低维向量共享交互参数”的结构先验；
- GBDT leaf、目标编码、Embedding、Stacking 都是在把某种中间模型的判断结果转化成新特征。

适合写成一篇完整的思考型技术博客。

### 选题二：为什么表格数据不是简单喂给模型就完了

题目可以是：

> 表格数据的真正难点：字段异构、稀疏交叉与业务先验

核心观点：

表格数据的问题不只是“用什么模型”，而是字段本身没有天然统一结构。图像有局部空间结构，文本有序列结构，但表格里的年龄、城市、设备、用户 ID、行为次数、历史转化率和时间窗口统计并不共享同一种几何关系。模型要学好，必须先把这些字段变成更有意义的表达。

这篇更偏方法论，可以衔接已有的 Tree 模型文章。

### 选题三：DeepFM 和树模型到底是不是一类东西

题目可以是：

> DeepFM、LightGBM 与 CatBoost：同是表格模型，为什么适用场景完全不同？

核心观点：

DeepFM 和 LightGBM/CatBoost 都经常处理结构化数据，但它们面对的数据形态不同：

- 树模型适合中小规模、字段异构、连续变量和低中基数类别变量较多的表格任务；
- DeepFM 适合广告、推荐、搜索等场景中的极高维稀疏类别特征；
- 树模型通过分裂路径表达条件交互，DeepFM 通过 embedding 内积和 DNN 表达稀疏交互；
- 两者可以组合，例如 GBDT leaf index 作为类别特征输入 LR/FM/DeepFM。

这篇更适合专门解释 DeepFM，避免和已有 Tree 模型文章重叠。

## 想法卡片一：聚类特征是无监督模型产出的身份标签

原始经验：

> 聚类得到的分类特征也是构造了新的学习特征，可以考虑用这种自然的分类信息作为新的特征加入 ML。反向标记身份可能就是这个思想。这是一种模型堆叠思想，但不是常规集成学习中的知识。这种新的特征往往会起到很好的效果。

### 这条经验的价值

这条经验很有价值，尤其适合表格数据、用户行为数据、风控数据、营销人群数据、地理空间数据和医疗分型数据。它背后的逻辑是：

原始特征描述的是样本的局部字段值，而聚类特征描述的是样本在整体数据空间中的位置。

一个用户的年龄、城市、消费频率、访问时段和品类偏好分别是一些零散字段；聚类后得到的“用户属于第 7 类人群”，则是这些字段共同定义出来的潜在身份。监督模型接收到这个身份标签后，不需要自己从零学习整个人群结构。

### 它为什么像 Stacking

常规 Stacking 通常是把模型 A 的预测结果作为模型 B 的输入。例如：

```text
原始特征 -> 模型 A -> 预测概率 -> 模型 B
```

聚类特征也是类似逻辑，只是模型 A 是无监督模型：

```text
原始特征 -> 聚类模型 -> cluster id / distance / density -> 监督模型
```

所以它可以叫“表示层面的 stacking”，或者“无监督特征堆叠”。它不一定属于教科书里的集成学习，但它确实是在使用一个模型的中间结果帮助另一个模型学习。

### 可以构造哪些聚类特征

最基础的是 `cluster_id`，也就是样本属于哪个簇。它通常应该被当成类别特征，而不是连续数值。对于树模型来说，CatBoost 可以原生处理类别特征；对 LightGBM/XGBoost 通常需要合适的类别编码或 one-hot；对线性模型和神经网络则需要 one-hot 或 embedding。

更丰富的做法是加入到每个聚类中心的距离：

$$
d_k(x) = \|x - c_k\|
$$

如果有 K 个簇，就能得到 K 个距离特征。相比单个 `cluster_id`，距离特征能表达样本处在簇中心还是边界。很多时候边界样本更难预测，距离本身就含有不确定性信息。

还可以加入所在簇的样本数量、簇内密度、最近两个簇的距离差、软聚类概率、不同特征视角下的多个 cluster id。例如：

- 行为特征聚类：用户行为模式；
- 画像特征聚类：用户静态属性；
- 时间序列统计聚类：用户生命周期阶段；
- 地理特征聚类：空间区域身份；
- 商品属性聚类：商品类型或价格带。

这样一个样本就不只拥有一个身份，而是拥有多个视角下的身份。

### 最需要注意的风险

第一是数据泄漏。

如果只是在全量训练集上 fit 聚类器，再把聚类结果用于训练模型，通常问题不大；但如果在交叉验证评估中提前用到了验证折的数据分布，就可能让验证分数偏乐观。更严谨的做法是把聚类作为 pipeline 的一部分：每一折只在训练折 fit 聚类器，再 transform 验证折。

如果在 cluster 上做 target encoding，风险更大。簇内目标均值必须使用 out-of-fold 方式计算，否则会把目标标签泄漏给样本自己。

第二是距离度量不可靠。

K-Means 默认使用欧氏距离。如果特征尺度没有处理，价格、次数、时长这类大尺度变量会支配聚类结果。对于类别变量、高维稀疏变量或混合类型变量，直接 K-Means 常常不是最佳选择，可以考虑 Gower distance、K-Prototypes、HDBSCAN、谱聚类、UMAP 后聚类，或者先做 embedding 再聚类。

第三是簇数和业务解释。

簇数太少会把不同人群混在一起，簇数太多会变成近似 ID，带来过拟合。好的 cluster feature 不只是让指标提高，还应该能被解释为某种稳定人群、商品群、风险群或行为状态。

### 可执行实验设计

可以把聚类特征作为一个单独实验层：

1. Baseline：只用原始特征训练 LightGBM/CatBoost；
2. 加入 `cluster_id`；
3. 加入 `distance_to_centroids`；
4. 加入多视角 cluster；
5. 加入 out-of-fold cluster target encoding；
6. 用交叉验证比较 AUC、Logloss、RMSE、稳定性和线上模拟表现。

如果第 2 步就明显增益，说明潜在人群身份很重要；如果第 3 步继续增益，说明样本在群体空间中的相对位置也有信息；如果只有 target encoding 有增益，就要警惕泄漏和过拟合。

## 想法卡片二：特征加权是把业务判断注入模型

原始经验：

> 特征加权也是一种对 ML 影响巨大的操作。我们基于领域知识，对那些更有价值的特征给予更高的训练权重。

### 这条经验的价值

这条经验也很重要，但需要比“聚类特征”更谨慎。原因是：不同模型理解“特征权重”的方式完全不同。

对线性模型、距离模型和神经网络来说，特征缩放、特征复制、正则化强度、输入噪声、embedding 维度等方式，确实会改变模型对某些特征的敏感度。

但对树模型来说，简单把一个数值特征乘以 10，通常不会让它更重要。因为树模型主要看排序和分裂增益。一个变量从 `x` 变成 `10x`，样本顺序没有变，阈值也只是同步缩放。它不等价于“提高特征权重”。

所以更准确的说法是：

> 特征加权不是一个统一操作，而是一组向模型表达领域先验的手段。

### 特征加权的几种落地方式

第一种是差异化正则化。

在线性模型或神经网络里，可以对不同特征或不同参数组设置不同正则强度。专家认为稳定可靠的特征，惩罚弱一些；容易受噪声影响的特征，惩罚强一些。这样不是强迫模型相信某个特征，而是改变模型在复杂度受限时的偏好。

第二种是列采样概率或特征采样权重。

一些 boosting 框架支持在列采样时对特征设置不同采样概率。这个做法的含义不是“这个特征的数值更大”，而是“这个特征在建树过程中更频繁获得被尝试的机会”。它适合在特征很多、强特征容易被随机采样错过，或者希望弱化大量噪声特征时使用。

第三种是分裂增益惩罚或特征成本。

如果某些特征昂贵、不稳定、线上不可用或业务可信度低，可以在模型选择分裂时对它们施加更高成本。这样模型只有在这些特征确实带来足够收益时才会使用它们。

第四种是特征构造优先级。

很多时候，所谓“加权”不是在模型参数里做，而是在特征工程里做：让核心特征参与更多交叉、更多时间窗口统计、更多分桶、更多聚合。比如金融风控里，逾期历史、收入稳定性、近期负债变化可能比设备型号更值得被构造成多种视角。

第五种是约束。

单调约束、交互约束、业务规则约束，也是一种更强的领域先验。例如“风险应该随逾期次数增加而不下降”，这比简单提高特征权重更明确。

### 何时有价值

特征加权最适合以下场景：

- 样本量不够大，模型容易学到伪相关；
- 特征很多，但业务知道其中一部分更稳定；
- 训练集和线上分布可能有偏移，需要保护更稳健的变量；
- 特征成本不同，需要控制线上推理或数据采集成本；
- 需要可解释性，不能让模型过度依赖不可解释字段。

它最不适合的场景是：领域知识本身很弱，却用强先验压制模型学习。这会把人的偏见硬编码进模型。

### 可执行实验设计

建议不要只看一次指标，而是做 ablation：

1. 原始模型；
2. 加入专家认为重要的特征构造；
3. 对低可信特征增加正则或减少采样；
4. 加入单调/交互约束；
5. 用不同时间切分验证稳定性；
6. 检查特征重要性、SHAP、错误样本和分群表现。

如果加权只在随机切分上有效，在时间外推上失效，很可能只是过拟合了训练集结构。

## 想法卡片三：DeepFM 解决的是高维稀疏交叉问题

原始问题：

> DeepFM 通过交叉项乘积解决数据高度稀疏的问题，这是什么？它和 LightGBM、CatBoost 之类的树模型学习框架有什么联系？它也是很好的机器学习方法吗？

### DeepFM 的问题背景

DeepFM 主要来自推荐系统、广告 CTR/CVR 预估等场景。这类任务的典型输入不是几十列连续变量，而是大量高基数类别特征：

- user_id；
- item_id；
- shop_id；
- city；
- device；
- query；
- category；
- ad_id；
- historical behavior ids。

one-hot 之后，特征维度可能达到百万、千万甚至更高，但单个样本只激活其中很少几个位置。这就是高维稀疏。

在这种情况下，真正重要的往往不是某个字段单独出现，而是字段组合：

- 某类用户 + 某类商品；
- 某个城市 + 某种广告；
- 某个时间段 + 某种兴趣；
- 某个 query + 某个 item。

问题是，显式为每个组合学习一个独立参数几乎不可行。组合数量太大，而且大多数组合在训练集中极少出现。

### FM 的核心：用 embedding 内积共享交叉参数

Factorization Machine 的二阶部分可以写成：

$$
\sum_{i=1}^{n}\sum_{j=i+1}^{n} \langle v_i, v_j \rangle x_i x_j
$$

其中 $v_i$ 和 $v_j$ 是两个特征的低维隐向量。两个特征的交叉权重不再是一个独立参数，而是由两个向量的内积决定。

这件事的关键在于参数共享。

即使 user A 和 item B 没有在训练集中频繁共同出现，只要 user A 和其他 item 有共现，item B 和其他 user 有共现，它们各自的 embedding 就能被更新。模型就可以通过 embedding 内积估计 user A 与 item B 的匹配程度。

这就是“通过交叉项乘积解决稀疏”的准确含义：不是稀疏性消失了，而是模型不再为每个稀疏组合单独学习一个参数，而是把组合权重分解为低维向量之间的关系。

### DeepFM 比 FM 多了什么

DeepFM 通常由两部分并行组成：

- FM 分支：负责一阶特征和二阶显式交叉；
- DNN 分支：使用共享 embedding，学习更高阶、更非线性的隐式交叉。

因此它的输出可以粗略理解为：

$$
\hat{y} = \sigma(y_{FM} + y_{DNN})
$$

它的价值在于：FM 分支保留了对二阶交叉的强归纳偏置，DNN 分支补充了更复杂的组合能力，两者共享 embedding，避免完全割裂。

### 它和树模型的关系

DeepFM 和 LightGBM/CatBoost 都可以处理表格数据，但它们的舒适区不同。

树模型擅长：

- 连续变量；
- 中小规模表格；
- 缺失值；
- 分桶和条件规则；
- 字段异构但维度不极端的任务；
- 特征数量在可管理范围内的工业表格任务。

DeepFM 擅长：

- 高维稀疏类别特征；
- 大规模推荐或广告数据；
- 用户、物品、上下文之间的组合关系；
- 需要 embedding 表示学习的任务；
- 训练样本足够多，能支撑大量 embedding 参数。

它们可以组合。典型做法是先用 GBDT/LightGBM 学出叶子节点索引：

```text
原始连续/统计特征 -> GBDT -> leaf index -> one-hot/category feature -> LR/FM/DeepFM
```

这相当于让树模型先把连续特征和规则交互离散成叶子身份，再交给线性模型或 embedding 模型处理。它和聚类特征有相似精神：都是把一个模型得到的结构身份变成新特征。

### DeepFM 是不是一个好方法

结论应该分场景：

- 在 CTR、CVR、推荐排序、广告预估等高维稀疏大样本场景里，DeepFM 是重要且经典的方法；
- 在普通中小规模表格任务里，它不一定比 LightGBM/CatBoost 好，甚至经常更难调、更容易过拟合；
- 在连续特征为主、样本量有限、业务需要快速强 baseline 的任务里，优先用树模型更现实；
- 在 ID、类别、行为序列和上下文交叉是核心信号的任务里，DeepFM 这类 embedding-based 方法更自然。

所以它不是“更高级的 LightGBM”，也不是“树模型替代品”。它解决的是另一类输入结构下的表示学习问题。

## 三条经验的共同结构

这三条经验可以统一成一个框架：

| 方法 | 注入的信息 | 产生的新表示 | 适合场景 | 主要风险 |
| :--- | :--- | :--- | :--- | :--- |
| 聚类特征 | 样本在全局空间中的群体身份 | cluster id、距离、密度、软分配概率 | 人群分层、商品分组、风险分群、行为模式 | 泄漏、距离失真、簇数过拟合 |
| 特征加权 | 专家对字段价值和稳定性的先验 | 正则差异、采样权重、约束、交叉优先级 | 小样本、噪声高、字段多、需要解释性 | 人为偏见、错误先验、验证不充分 |
| DeepFM/FM | 稀疏类别交叉可以低维分解 | embedding、二阶内积、高阶 DNN 表示 | 推荐、广告、搜索、CTR/CVR | 数据不足、调参复杂、线上成本 |

正式博客可以围绕这个表展开。不要把重点写成“某个模型很强”，而是写成：

> 好的机器学习经验，往往是在选择一种更合适的表示方式，让模型更容易学习真正的结构。

## 推荐的正式博客结构

可以这样组织：

1. 开头：表格 ML 的真正困难不是模型选择，而是表示选择；
2. 第一部分：聚类特征如何把“潜在群体身份”交给监督模型；
3. 第二部分：特征加权如何把业务先验转化为模型偏好；
4. 第三部分：DeepFM 如何用 embedding 内积解决高维稀疏交叉；
5. 第四部分：它们和树模型、Stacking、目标编码之间的共同逻辑；
6. 最后：给出实战 checklist，说明什么时候该用、怎么验证、如何避免泄漏。

## 实战 Checklist

准备使用聚类特征时：

- 是否先标准化或合理编码了参与聚类的特征？
- 聚类是否只在训练折 fit，再 transform 验证折？
- cluster id 是否作为类别变量处理？
- 是否比较了 cluster id、距离、密度、软概率的增益？
- 是否做了目标泄漏检查？

准备使用特征加权时：

- 这个“权重”对当前模型真的生效吗？
- 是通过正则、采样、约束、成本，还是通过特征构造表达？
- 专家先验是否能被 ablation 验证？
- 随机切分有效，时间切分是否也有效？
- 是否检查了分群表现和错误样本？

准备使用 DeepFM 时：

- 输入是否确实是高维稀疏类别特征为主？
- 样本量是否足够训练 embedding？
- 是否有大量 user-item-context 组合关系？
- 是否和 LightGBM/CatBoost baseline 做了公平比较？
- 是否考虑了线上推理成本、embedding 更新和冷启动问题？
