如果只看教程和公开数据集，数据处理似乎是一件很线性的事情：读入 CSV，处理缺失值，编码类别变量，划分训练集和测试集，然后训练模型。

真实项目很少这样。真实数据经常来自多个系统，字段语义不稳定，记录粒度不一致，非结构化信息混在日志、文本、代码、配置和第三方接口里。模型效果差的时候，也未必是模型不够强，而可能是输入表示从一开始就错了。

我在处理数据时遇到过的主要挑战，可以概括成三类：

- 非结构化数据不是“脏一点的表格”，而是需要重新定义抽取方式；
- 高维稀疏特征不是简单 one-hot 就能解决，而是需要表示学习和特征交叉意识；
- 技术优化如果脱离产品目标，很容易变成昂贵但无价值的工程消耗。

这三类问题看起来不同，但本质都指向同一件事：数据处理不是为了把数据变得整齐，而是为了让下游决策、模型或产品体验真正变好。

## 挑战一：非结构化数据的混乱，不适合只靠正则硬拆

最容易被低估的是非结构化数据。

很多有价值的信息并不在干净的关系型表里，而是藏在代码仓库、commit diff、issue、CVE 描述、安全报告、日志片段、用户对话、模型 trace、工具调用记录和错误堆栈中。

这类数据有两个特点。

第一，它们表面是文本，实际带有结构。

代码不是普通字符串，它有语法树、作用域、调用关系、数据流和控制流。漏洞报告不是普通段落，它包含影响版本、攻击前提、组件名称、漏洞类型、修复状态和证据来源。Agent 工具调用日志也不是普通 JSON，它包含用户意图、工具选择、参数构造、执行结果和恢复路径。

第二，它们的错误抽取成本很高。

如果只是做展示，少量误差可能还能接受。但如果这些数据要进入 SFT、RL、评测集、检索索引或安全规则库，抽取错误会被模型放大。错误样本不是“噪音”这么简单，它可能会教模型学到错误行为。

### 正则表达式的边界

正则表达式适合处理局部、稳定、格式明确的文本模式。它不适合承担复杂语义抽取。

比如从代码 diff 中提取“某个函数修改后是否引入了危险调用”，如果只靠字符串匹配，很容易遇到这些问题：

- 同名函数处于不同作用域；
- 调用被包装在 helper 中；
- 参数来源需要追踪；
- 注释、测试代码、生成代码混在一起；
- diff 上下文不足，需要回到完整文件；
- 语言语法差异导致匹配规则快速膨胀。

正则规则越写越多，系统并不会更可靠，只会更难维护。

### 用结构化工具处理结构化语义

我的做法是尽量把非结构化问题转回结构化问题。

对于代码类数据，优先使用 AST、parser、静态分析或 CodeQL 这类语义查询工具。目标不是“匹配一段文本”，而是查询代码结构：

- 某个函数是否调用了危险 API；
- 某个参数是否来自用户输入；
- 某个分支是否缺少权限检查；
- 某个对象是否跨越生命周期边界；
- 某个补丁是否改变了控制流或数据流。

对于日志和事件数据，优先定义事件 schema，而不是让每个下游任务自己解析字符串。比如把 Agent trace 解析成：

```text
task_id
turn_id
user_intent
tool_name
tool_args
tool_result_status
error_type
recovery_action
final_outcome
```

这样后续才能稳定地做统计、过滤、采样、训练和评估。

对于漏洞、安全报告或客服对话，可以使用规则、模型和人工复核结合的方式。LLM 可以帮助把文本转为结构化候选，但关键字段仍需要校验：实体 ID、版本范围、时间、证据来源、标签体系都不能只靠自然语言生成。

### 数据契约要放在流水线入口

非结构化数据抽取最需要的是数据契约。

我会在流水线入口定义清楚：什么样的数据允许进入下游，什么样的数据必须拒收或进入人工复核。比如：

- 必须包含稳定实体 ID；
- 时间字段必须能解析且落在合理范围；
- 标签必须来自固定枚举；
- 关键字段不能由低置信度抽取结果填充；
- 代码片段必须能定位到仓库、commit、文件和行号；
- 多来源证据冲突时必须保留冲突状态，而不是强行合并。

这听起来保守，但高质量数据集宁可少一点，也不能把模糊样本伪装成确定样本。

## 挑战二：高维稀疏特征会让传统表示失效

第二类挑战来自特征空间。

在很多业务数据中，字段并不都是连续数值。更多时候，我们会遇到大量离散变量：

- 用户 ID；
- 商品 ID；
- 广告 ID；
- 搜索 query；
- 城市；
- 设备型号；
- 内容类别；
- 历史行为中的 item 列表。

如果把这些字段直接 one-hot，特征维度会迅速膨胀。单个样本只激活极少数特征，大部分位置都是 0。这就是高维稀疏。

稀疏本身不是问题，问题在于关键信号常常来自交叉。

一个用户 ID 单独出现不一定有意义，一个商品 ID 单独出现也不一定有意义，但“某类用户在某个上下文下对某类商品感兴趣”可能就是核心信号。组合才是模式。

### 为什么树模型会遇到瓶颈

LightGBM、XGBoost、CatBoost 在很多表格任务上非常强，尤其适合连续变量、低中基数类别变量、缺失值和条件规则。但当输入变成极高维稀疏 one-hot 时，树模型不一定是最自然的选择。

树模型通过分裂来构造局部规则。面对海量低频类别，很多分裂候选缺少足够统计支持。模型可能记住高频组合，却很难泛化到从未充分出现过的组合。

这时问题不是“树不够高级”，而是表示方式不合适。

### Embedding 和 DeepFM 的思路

在推荐、广告、搜索等场景里，更常见的做法是把高维稀疏类别映射到低维 dense embedding。

Factorization Machine 的关键想法是：不要为每个特征组合单独学习一个参数，而是为每个特征学习一个向量，用向量内积表示二阶交叉：

$$
\sum_{i=1}^{n}\sum_{j=i+1}^{n} \langle v_i, v_j \rangle x_i x_j
$$

这样即使某个 user-item 组合很少出现，只要 user 和其他 item 有过共现，item 和其他 user 有过共现，它们的 embedding 就能被更新，模型就可以对稀疏组合做泛化。

DeepFM 在这个基础上加入 DNN 分支，让模型同时拥有二阶显式交叉和高阶隐式交叉能力。它不是 LightGBM 的“升级版”，而是面向另一类数据结构的表示学习方法。

这部分和我前面整理的 ML 特征工程草稿有关：聚类特征、特征加权、FM/DeepFM、GBDT leaf index，本质上都在做同一件事，即把原始字段变成更容易学习的表示。

### 显式交叉仍然重要

有了 embedding，不代表可以放弃业务特征工程。

我更倾向于把显式特征和隐式表示结合起来：

- 对稳定业务字段构造时间窗口统计；
- 对关键 ID 构造历史行为聚合；
- 对类别变量做频次、目标统计或分层编码；
- 对用户、商品、上下文做多视角交叉；
- 对冷启动对象保留可解释的规则特征；
- 再用 embedding 模型学习大规模稀疏关系。

模型可以自动学习，但它学习得是否有效，仍取决于输入是否表达了正确的信息结构。

## 挑战三：技术优化可能脱离产品价值

第三类挑战更像工程习惯问题。

技术人员很容易把精力投入到可量化但不一定有价值的目标上：吞吐量提高 30%、离线 AUC 提升 0.3%、预处理任务从 20 分钟压到 10 分钟、特征数量从 500 扩到 2000。

这些工作可能有价值，但前提是它们连接到真实目标。

如果业务方根本不使用这个模型，AUC 提升没有意义；如果数据每天只跑一次，把任务从 20 分钟优化到 10 分钟未必重要；如果新增 1500 个特征让系统更难解释、更难监控、更难上线，离线指标提升可能不值得。

我踩过的坑是：把“技术上更好”误认为“产品上更好”。

### 评估指标必须回到任务

在写数据处理代码前，我现在会先问：

- 这批数据最终服务哪个产品场景？
- 它改善的是用户体验、运营效率、风控收益，还是模型能力？
- 如果这个数据集质量提高，最终指标应该怎么变化？
- 有没有一个真实任务可以衡量它？

如果清洗数据是为了提升 AI Agent 的工具调用能力，那么评估标准就不应该只是“样本数量更多”或“字段更完整”，而应该看：

- 任务完成率是否提升；
- 工具选择是否更准确；
- 参数构造错误是否减少；
- 失败后的恢复能力是否变强；
- 用户是否更少需要二次纠正。

如果数据处理服务推荐系统，那么评估也不应该只看离线 loss，还要看线上点击、转化、留存、冷启动、公平性和延迟。

### 数据处理也要有 ROI 意识

不是所有数据都值得处理到极致。

有些字段质量差、覆盖低、采集成本高、线上不稳定，即使短期离线有收益，也不一定值得进入核心链路。相反，有些朴素字段虽然不炫技，但稳定、便宜、可解释，长期价值更高。

我会把数据处理任务分成三类：

- 必须做：影响口径正确性、数据安全、线上可用性；
- 值得做：能稳定提升核心指标，且维护成本可控；
- 可以不做：只改善局部离线指标，但无法进入行动闭环。

这个分类能避免把所有数据问题都当成技术挑战。

## 我的处理原则

面对真实数据，我会坚持几个原则。

第一，先定义数据的下游用途，再设计清洗标准。

用于展示的数据、用于离线分析的数据、用于模型训练的数据、用于安全决策的数据，对准确率、召回率、延迟、可解释性和可追溯性的要求完全不同。不要用同一套清洗策略处理所有任务。

第二，能结构化就不要停留在字符串层面。

代码、日志、trace、报告、对话都可以通过 parser、schema、DSL、抽取模型和校验规则转成结构化记录。结构化程度越高，下游复用越可靠。

第三，承认表示学习和特征工程是同一条链路。

Embedding、DeepFM、聚类特征、GBDT leaf、目标编码都不是孤立技巧，它们都在回答“如何让模型看到更好的表示”。不要把特征工程理解成深度学习之前的旧方法。

第四，所有数据处理都要有验证闭环。

清洗规则、抽取规则、特征构造、采样策略、模型输入，都应该通过真实任务指标验证。没有验证闭环的数据处理，只是把原始混乱变成了另一种形式的混乱。

## 实践检查清单

- 非结构化数据是否已经尽量转成带 schema 的结构化记录？
- 抽取规则是否能追溯到原始证据，而不是只保留处理后结果？
- 高维稀疏特征是否需要 embedding、FM/DeepFM 或其他表示学习方法？
- 新增特征是否通过 ablation 证明了稳定收益？
- 数据处理优化是否连接到真实产品指标，而不是只改善中间技术指标？

## 结语

真实数据处理不像教程。它更像是在不稳定系统之间建立一条可信通道：上游数据不断变化，下游模型和产品又要求稳定输出。

所以数据处理的核心能力，不只是会清洗、会建模、会调参，而是能判断哪些混乱必须被结构化，哪些稀疏关系需要新的表示，哪些技术工作真正值得投入。

只有当数据处理开始服务真实任务，而不是服务工程自我满足时，它才真正进入了数据价值链。
