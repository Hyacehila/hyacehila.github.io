在很多机器学习项目里，模型效果迟迟上不去时，第一反应往往是换模型：从 Logistic Regression 换到 Random Forest，从 XGBoost 换到 LightGBM，从 LightGBM 换到 CatBoost，再进一步尝试神经网络或 AutoML。

换模型当然有价值，但在表格数据和推荐系统里，很多关键收益并不来自“模型名字变了”，而来自模型终于看到了更合适的特征表示。

原始字段经常是零散的。年龄、城市、设备、访问次数、价格、用户 ID、商品 ID、历史点击率、最近 7 天活跃次数、最近 30 天购买金额，这些字段被放在同一张表里，但它们并不天然共享一个统一的几何结构。表格数据不像图像有局部空间，也不像文本有顺序上下文。它更像是业务系统在不同阶段留下的异构痕迹。

因此，很多实战技巧的共同目标都是：**把原始字段转化成模型更容易学习的结构化信号**。

聚类特征、特征加权和 DeepFM 看起来属于不同主题：一个来自无监督学习，一个来自领域知识，一个来自推荐系统深度模型。但它们背后有一条共同线索：

- 聚类特征把“样本属于哪个潜在群体”显式交给模型；
- 特征加权把“哪些字段更可信、更重要”作为先验注入训练过程；
- DeepFM 把“高维稀疏类别之间存在可共享的交叉关系”写进模型结构；
- 它们本质上都是在给模型提供归纳偏置，让模型少走弯路。

这篇文章想讨论的不是某个单点技巧，而是一个更集中的问题：**在结构化机器学习中，我们如何把群体身份、业务先验和稀疏交互转化成可学习的特征表示？**

## 聚类特征：让模型看到样本的潜在身份

聚类特征的基本做法很简单：先使用无监督算法对样本聚类，再把聚类结果作为新特征加入监督模型。

例如，我们有一批用户，每个用户都有年龄、城市等级、消费频率、近 7 天活跃次数、品类偏好、客单价和访问时段等特征。直接把这些字段喂给模型，模型当然也可以学习。但如果我们先做一次聚类，得到类似这样的潜在人群：

- 高活跃、低客单价、夜间访问用户；
- 低频、高客单价、品牌偏好用户；
- 新注册、行为短、设备集中用户；
- 老用户、复购强、价格敏感用户。

然后把 `cluster_id` 加入模型，监督模型就不再只看到零散字段，而是额外看到一个“这个样本属于哪类人”的身份标签。

这类身份信息常常很有用，因为它表达的是样本在整体数据空间中的位置。单个字段描述局部属性，聚类描述全局结构。

### 为什么它像一种非传统 Stacking

常规 Stacking 通常是把一个模型的预测结果作为另一个模型的输入：

```text
原始特征 -> 模型 A -> 预测概率 -> 模型 B
```

聚类特征的逻辑类似，只是第一个模型不是监督模型，而是无监督模型：

```text
原始特征 -> 聚类模型 -> cluster id / distance / density -> 监督模型
```

所以它可以被理解为“无监督特征堆叠”。它不一定属于教科书意义上的集成学习，但它确实是在用一个模型学到的中间结构帮助另一个模型。

这和很多工业实践是一致的。GBDT leaf index、AutoEncoder embedding、topic model 分布、图算法社群 ID、用户画像分群，都是把某种模型产出的结构身份变成新特征。

### 不要只使用 cluster id

最容易想到的是加入一个 `cluster_id`。但在很多任务里，cluster id 只是起点。

如果使用 K-Means，可以额外计算样本到每个聚类中心的距离：

$$
d_k(x) = \|x - c_k\|
$$

这样如果有 K 个簇，就能产生 K 个连续距离特征。距离特征比单一标签保留更多信息：一个样本是在某个簇的中心，还是位于两个簇之间的边界？边界样本往往更难预测，这种“不确定身份”本身就可能影响目标变量。

还可以构造：

- 所在簇的样本数量；
- 所在簇的局部密度；
- 最近簇和次近簇的距离差；
- 软聚类概率；
- 不同特征子集上的多个 cluster id；
- cluster 内的历史统计特征。

其中“不同特征子集上的多个 cluster id”非常值得重视。用户的行为特征、静态画像、地理位置、消费能力和时间序列状态，未必应该被混在同一次聚类里。多视角聚类可以让模型同时看到“行为身份”“画像身份”“生命周期身份”和“空间身份”。

### 最大风险是泄漏和伪结构

聚类特征不是免费午餐。它至少有两个风险。

第一是数据泄漏。

如果你在交叉验证前先对全量数据 fit 聚类器，再把聚类结果放进每一折训练，验证分数可能偏乐观。因为验证折的数据分布已经参与了聚类中心的形成。无监督泄漏通常比目标泄漏轻，但在严谨评估中仍应避免。

更稳妥的做法是：在每一折交叉验证中，只用训练折 fit 聚类器，再 transform 验证折。线上推理时，也只能使用训练阶段固定下来的聚类器去处理新样本。

如果进一步做 cluster target encoding，也就是计算每个簇的历史目标均值，那么必须使用 out-of-fold 方式。否则样本自己的标签会参与构造自己的特征，这就是明确的目标泄漏。

第二是伪结构。

K-Means 使用欧氏距离，如果数值尺度没有处理，大尺度变量会支配聚类结果。类别变量、高维稀疏变量、混合类型变量也未必适合直接 K-Means。聚类算法找到的是某种距离定义下的结构，不一定等于业务结构。

因此，聚类特征的正确使用方式不是“加了就一定好”，而是作为一个可验证的表示假设：

> 如果样本确实存在稳定的潜在人群结构，那么把这种结构显式提供给监督模型，应该能提升泛化表现。

这句话里最重要的是“稳定”和“泛化”。如果只在一次随机切分上提高，却在时间切分、区域切分或线上数据上失效，那么聚类可能只是在记忆训练集分布。

## 特征加权：把业务判断变成模型偏好

另一条非常实战的经验是：不同特征的价值并不相同。业务专家常常知道某些字段更可信、更稳定、更接近因果机制；也知道某些字段虽然在训练集上相关，但可能只是短期噪声、采集偏差或平台策略的副产物。

这时，一个自然想法是：能不能让模型更重视核心特征，少受噪声特征影响？

这个想法很有价值，但要小心一个误解：**特征加权不是所有模型都以同一种方式支持。**

对线性模型来说，如果输入特征没有标准化，特征尺度会直接影响参数学习和正则化效果。对距离模型来说，特征尺度会改变距离度量。对神经网络来说，输入尺度、embedding 维度、参数组正则、dropout 和噪声注入都会影响模型对特征的使用。

但对树模型来说，把一个连续变量简单乘以 10，通常不会让它“更重要”。因为树模型主要看样本排序和分裂增益。`x` 和 `10x` 的排序相同，能切出的样本划分也基本相同，阈值只是同步缩放。

所以更准确的表述应该是：

> 特征加权不是一个单一操作，而是一组表达领域先验的方法。

### 方法一：差异化正则化

在线性模型和神经网络中，可以对不同特征或参数组设置不同正则强度。对专家认为可靠的核心特征，使用更弱的惩罚；对高噪声特征，使用更强的惩罚。

这不是强迫模型相信某个特征，而是在模型容量受限或数据不足时，改变模型的偏好。它背后的思想是：当多个解释都能拟合训练集时，优先选择更符合业务常识、更可能稳定外推的解释。

### 方法二：列采样概率

在 boosting 框架中，列采样是一种常见正则手段。如果某些框架支持按特征设置采样权重，就可以让核心特征在建树过程中获得更多被尝试的机会，让低可信特征更少参与候选分裂。

这个方法适合特征很多、噪声很高、强特征容易在随机采样中被错过的场景。但它也需要小心：如果人为提高错误特征的出现概率，模型会更稳定地犯错。

### 方法三：分裂增益惩罚和特征成本

有些特征虽然有效，但线上获取成本高、延迟大、缺失率高，或者未来可能不稳定。这类特征不应该和低成本稳定特征拥有完全相同的使用门槛。

一种思路是在模型选择特征时加入成本意识：只有当昂贵特征带来的收益足够大时，才允许模型使用它。这样得到的模型未必离线指标最高，但可能更符合生产系统的真实目标。

### 方法四：特征构造优先级

最常见、也最容易落地的“特征加权”，其实不是改模型参数，而是改变特征工程资源分配。

如果你确信某些字段很重要，可以围绕它们构造更多视角：

- 时间窗口统计；
- 分桶和非线性变换；
- 与关键字段交叉；
- 组内排名和百分位；
- 历史均值、方差、趋势；
- 缺失模式和异常模式；
- 业务规则派生变量。

这相当于提高核心字段在表示空间中的分辨率。模型看到的不再是一个粗糙字段，而是围绕该字段展开的一组结构化描述。

### 方法五：单调约束和交互约束

有时业务先验不是“某个特征更重要”，而是“某个特征应该如何影响预测”。

例如在风控中，历史逾期次数增加，风险不应该下降；在价格模型中，其他条件不变时，严重磨损通常不应该让估值上升。这类知识用简单特征权重表达并不准确，更适合用单调约束表达。

交互约束也类似。如果某些特征组合在业务上没有意义，或者容易导致泄漏，可以限制模型不要学习这类交互。

这种约束比“提高权重”更强，也更清晰。它不是告诉模型“多看这个特征”，而是告诉模型“这个特征的作用方向或交互边界应该是什么”。

### 特征加权最适合解决什么问题

它最适合以下场景：

- 数据量不大，模型容易追逐噪声；
- 特征很多，且质量参差不齐；
- 训练集和线上环境可能存在分布偏移；
- 业务知道哪些字段稳定，哪些字段只是短期相关；
- 模型不仅要准，还要可解释、可控、可上线。

但它也有明显风险。领域知识如果是错的，特征加权会把错误先验放大。人的经验如果来自过时业务环境，也可能压制模型发现新模式。

所以特征加权一定要通过 ablation 验证。它不能只靠“听起来合理”。至少要比较原始模型、加权模型、约束模型和不同时间切分下的稳定性，并检查错误样本是否真的变得更符合业务预期。

## DeepFM：为高维稀疏交叉而生的模型

DeepFM 所处理的问题，和普通表格任务有明显差异。

在广告点击率预测、推荐排序、搜索排序等场景中，特征往往是极高维、极稀疏的类别特征。一个样本可能包含：

- 用户 ID；
- 商品 ID；
- 广告 ID；
- 店铺 ID；
- 城市；
- 设备；
- 时间段；
- 搜索词；
- 类目；
- 历史行为序列中的若干 item。

这些字段 one-hot 之后维度极高，但每个样本只激活极少数位置。更麻烦的是，预测目标往往强烈依赖特征交叉：

- 某个用户群体是否喜欢某类商品；
- 某个 query 下某个 item 是否匹配；
- 某个城市中的某类广告是否更容易被点击；
- 某个设备和某个时间段是否对应特殊行为模式。

如果我们为每个组合都学习一个独立权重，参数数量会爆炸，而且大量组合在训练集中出现次数很少。传统 one-hot 交叉很快会遇到稀疏性问题。

### FM 如何用内积表达交叉

Factorization Machine 的关键想法是：不要为每个特征组合学习一个独立参数，而是为每个特征学习一个低维向量，然后用两个向量的内积表示二阶交叉强度。

FM 的二阶交互项可以写成：

$$
\sum_{i=1}^{n}\sum_{j=i+1}^{n} \langle v_i, v_j \rangle x_i x_j
$$

这里 $v_i$ 是第 $i$ 个特征的 embedding，$\langle v_i, v_j \rangle$ 是两个 embedding 的内积。

这个设计解决稀疏问题的方式非常关键：组合权重不再独立存在，而是由两个特征各自的 embedding 决定。

即使某个用户和某个商品很少共同出现，只要这个用户和其他商品有过交互，这个商品和其他用户有过交互，它们的 embedding 就能从其他共现关系中被更新。模型可以通过 embedding 内积对未充分出现的组合做泛化。

这不是魔法，而是参数共享。FM 把大量稀疏组合压缩到低维向量空间里，让相似特征拥有相似表示，从而让未充分观察到的组合也能得到合理估计。

### DeepFM 在 FM 之外加入高阶交叉

DeepFM 可以粗略理解为 FM 和 DNN 的并联结构。

FM 分支负责一阶特征和二阶显式交叉，DNN 分支使用共享 embedding 学习更高阶、更复杂的非线性交叉。最终两部分共同输出预测：

$$
\hat{y} = \sigma(y_{FM} + y_{DNN})
$$

它的意义在于保留两种能力：

- FM 分支提供对二阶交叉的强归纳偏置；
- DNN 分支提供更灵活的高阶组合能力；
- 共享 embedding 避免把低阶和高阶表示完全割裂。

相比手工构造交叉特征，DeepFM 更自动；相比纯 DNN，它保留了 FM 对二阶交叉的明确建模。这也是它在 CTR/CVR 场景中成为经典方法的重要原因。

### DeepFM 和树模型不是谁替代谁

DeepFM 和 LightGBM、CatBoost 都能处理结构化输入，但它们不是同一类问题的简单替代。

树模型的强项是字段异构、中小样本、连续变量、缺失值、条件规则和可解释性。它通过分裂路径表达交互：

```text
if 最近 7 天点击次数 > 10
and 客单价 < 100
and 城市等级 in {2, 3}
then ...
```

DeepFM 的强项是高维稀疏类别交叉。它通过 embedding 和内积表达关系：

```text
user embedding · item embedding
context embedding · ad embedding
```

这两种表达非常不同。树模型像是在样本空间中不断切区域；DeepFM 像是在高维稀疏 ID 之间学习一个低维关系空间。

因此，在一个普通房价预测、设备故障预测、信贷评分或医学表格任务中，LightGBM/CatBoost 往往更现实。样本量有限、连续变量多、特征维度不极端时，DeepFM 很可能成本更高、收益更不稳定。

但在推荐和广告场景中，如果核心信号来自 user、item、ad、query、context 之间的大量稀疏组合，DeepFM 这类 embedding-based 方法就更自然。

### 两者可以组合

树模型和 FM/DeepFM 并不互斥。一个经典思路是先用 GBDT 处理连续特征和统计特征，再把每棵树的叶子节点索引作为类别特征输入 LR、FM 或 DeepFM：

```text
原始表格特征 -> GBDT -> leaf index -> one-hot/category -> LR/FM/DeepFM
```

这件事和聚类特征非常相似。聚类把样本映射为 cluster identity，GBDT 把样本映射为 leaf identity。两者都是把模型学到的中间结构变成新特征。

换句话说，聚类特征、GBDT leaf、FM embedding、DeepFM DNN 分支，本质上都是在做表示转换。区别只在于它们转换出的结构不同：

- 聚类输出群体身份；
- GBDT 输出规则路径身份；
- FM 输出低维交叉向量；
- DeepFM 输出低阶与高阶交叉表示。

## 三种经验的共同逻辑

把上面的内容放在一起，可以得到一个统一表述：

| 方法 | 它补充了什么 | 它把什么变成特征 | 最适合的场景 | 最大风险 |
| :--- | :--- | :--- | :--- | :--- |
| 聚类特征 | 全局空间结构 | 潜在人群/样本群体身份 | 用户分层、商品分组、风险分群、行为模式 | 泄漏、距离失真、伪聚类 |
| 特征加权 | 领域先验 | 模型对字段的偏好、约束或使用成本 | 小样本、高噪声、强业务约束、需要解释 | 错误先验、压制真实信号 |
| DeepFM | 稀疏交叉结构 | 类别特征 embedding 与交互关系 | 推荐、广告、搜索、CTR/CVR | 数据不足、调参复杂、冷启动 |

这三者并不是孤立技巧，而是三种不同的归纳偏置：

- 聚类特征相信“相似样本属于相似群体”；
- 特征加权相信“有些字段更稳定、更可信、更接近机制”；
- DeepFM 相信“稀疏交叉可以通过低维向量关系泛化”。

机器学习模型从来不是在真空里学习。模型结构、损失函数、正则化、特征表示、编码方式和验证方法，都在告诉模型应该优先寻找哪类规律。所谓经验，很多时候就是知道该给模型什么样的提示。

## 一个实战工作流

如果我面对一个新的结构化机器学习任务，会倾向于按下面的顺序处理。

第一步，建立强 baseline。

先用 LightGBM、CatBoost 或 AutoGluon 跑出一个可靠基线。不要一开始就陷入复杂特征工程。强 baseline 的意义是给后续所有技巧提供参照系：每个新特征、新约束、新模型都必须证明自己真的带来增益。

第二步，做错误样本和分群分析。

看模型在哪些人群、商品、时间段、地区、价格带或行为状态下表现差。聚类特征往往来自这里：如果错误集中在某些隐含群体中，就说明模型可能没有充分识别样本身份。

第三步，尝试无监督结构特征。

对合理的特征子集做聚类，加入 cluster id、距离、密度和软概率。严格使用交叉验证 pipeline，避免验证分布或目标标签泄漏。

第四步，注入业务先验。

把专家认为稳定的特征做更充分的派生；对高风险特征施加正则、采样限制、成本惩罚或约束；对具有明确方向的变量尝试单调约束。每个先验都要通过 ablation 验证，而不是只靠直觉。

第五步，判断是否需要 DeepFM 这类 embedding 方法。

如果核心信号主要来自高维稀疏类别交叉，例如 user-item-context，且样本量足够大，就考虑 FM、DeepFM、DCN、xDeepFM、AutoInt 等路线。如果只是普通中小表格任务，优先把树模型和特征工程做好。

第六步，做时间外推和线上一致性检查。

特征工程最容易在随机切分中显得有效，却在真实未来数据中失效。所有涉及目标统计、聚类、分群、ID、历史窗口的特征，都应该检查时间边界、线上可得性和冷启动表现。

## 结语：好的特征不是更多字段，而是更好的表达

很多机器学习经验之所以有效，不是因为它们神秘，而是因为它们改变了模型看到问题的方式。

聚类特征让模型看到潜在人群，特征加权让模型尊重领域先验，DeepFM 让模型在稀疏 ID 世界里共享交叉参数。它们都在做同一件事：把原始数据中不容易直接学习的结构，转化为模型可以使用的表示。

所以在表格 ML 和推荐系统里，一个很重要的判断是：当前瓶颈到底是模型能力不够，还是表示方式不对？

如果是前者，换模型可能有效；如果是后者，更强的模型也可能只是在错误表示上做更复杂的拟合。真正有价值的工程经验，是知道什么时候该换模型，什么时候该换特征，什么时候该把业务知识、无监督结构或交互假设明确地写进学习过程。
